# 집필 워크플로우 v3 — 사용 가이드

## 컨셉

**"기술의 대가가, 이 기술을 처음 접하는 사람에게 이야기로 알려주는 책"**

- 스토리텔링 기반. 교과서가 아니라 이야기.
- 어려운 용어는 비유로 풀어 설명.
- 완성 코드를 버전별로 쪼개서 점진적으로 성장하는 구조.
- 하나의 AI(Claude)가 처음부터 끝까지 관통.

## 시작하기

Claude Code에서 이 프로젝트 폴더를 열고 `새 책 만들기`를 입력하세요.

## 전체 워크플로우 (7 STEP)

```
Phase 1 ── 의도 확립
  STEP 1. 씨앗              "이 책은 뭐다"

Phase 2 ── 재료 파악
  STEP 2. 코드 해부          "재료가 뭐가 있지"

Phase 3 ── 이야기 설계
  STEP 3. 시나리오 + 버전     "어떤 순서로 이야기하지"
  STEP 4. 뼈대 세우기         "목차와 코드 실습 배치"

Phase 4 ── 집필
  STEP 5. 챕터 집필 (반복)    "쓴다"

Phase 5 ── 완성
  STEP 6. 프롤로그            "숲을 보여준다"
  STEP 7. 마무리              "머릿말, 맺음말"

Phase 6 ── 출판 (publisher 에이전트 = 인쇄소)
  출판정보 생성               "서점 등록용 정보 생성"
  인쇄소                     "MD→PDF 조판 + 레이아웃 최적화"
```

## 사용 가능한 명령어

| 명령어 | 설명 |
|--------|------|
| `새 책 만들기` | 새 프로젝트 시작 → STEP 1부터 진행 |
| `현재 상태` | 진행 상황 확인 |
| `씨앗 심기` | STEP 1 진행 |
| `코드 분석` | STEP 2 진행 |
| `시나리오 설계` | STEP 3 진행 |
| `뼈대 세우기` | STEP 4 진행 |
| `챕터 작성 [번호]` | STEP 5 해당 챕터 진행 |
| `검토 [챕터]` | 의도감시 + 감수 검토 재실행 |
| `프롤로그 생성` | STEP 6 진행 |
| `마무리` | STEP 7 진행 (머릿말 + 맺음말) |
| `출판정보 생성` | 서점 등록용 출판정보 markdown 생성 |
| `이미지 분석` | 챕터 이미지 플레이스홀더 생성 |

## 출판정보 생성 (pub-info)

`출판정보 생성` 명령은 서점(교보문고, YES24, 부크크 등) 등록에 필요한 도서 정보를 markdown으로 생성한다.

### 생성되는 항목

**1. 도서 정보** (자동 + 유저 입력)

| 항목 | 자동/입력 | 소스 |
|------|----------|------|
| 책제목/부제목 | 자동 | seed-v*.md |
| 페이지수 | 자동 | PDF 페이지 카운트 |
| 키워드 | 자동 | seed 핵심 메시지 + 챕터 기술명 |
| 저자명 | 입력 | 유저에게 질문 |
| 제본형태 | 입력 | 기본: 무선제본 |
| 가격/발행일 | 입력 | 유저에게 질문 |

**2. 추가정보** (자동 생성)

| 항목 | 설명 | 작성 규칙 |
|------|------|----------|
| **책소개** | "왜 이 책을 사야 하는가" 설득 | 4단 구성: 훅 → 구성 상세 → 배울 내용 불릿 → 대상 독자 불릿 |
| **목차** | 챕터 h1~h3 헤딩 추출 | 자동 |
| **저자소개** | 저자 약력 | 유저 입력 |
| **요약/본문일부** | "이 책의 글이 이런 느낌이다" 체험 | 프롤로그 오프닝 발췌 (200~500단어, 독립적으로 읽히는 구간) |
| **판권지** | 도서명, 지은이, 출판사 정보 | 텍스트 형식 (콜론 구분) |

### 호출 방식

```
출판정보 생성          → POD + 전자책 두 벌 생성
출판정보 생성 POD      → B5(188x257mm)만
출판정보 생성 전자책    → A4(210x297mm)만
```

### 산출물

```
projects/{책이름}/book/publish-info-pod.md
projects/{책이름}/book/publish-info-ebook.md
```

## 워크플로우 파일 구조

```
.claude/workflow/
├── README.md                    # 이 파일
├── step1-씨앗.md                # STEP 1: 의도 확립
├── step2-코드해부.md             # STEP 2: 코드 분석
├── step3-시나리오.md             # STEP 3: 시나리오 + 버전 설계
├── step4-뼈대.md                # STEP 4: 목차 + 코드 실습 배치
├── step5-챕터집필.md             # STEP 5: 이야기 파트 + 기술 파트
├── step6-프롤로그.md             # STEP 6: 프롤로그
├── step7-마무리.md              # STEP 7: 머릿말, 맺음말 (부록 없음)
├── review-guide.md              # 검토 모드 가이드 (인사이트/의도감시/감수)
└── legacy/                      # v3 이전 워크플로우 파일
```

```

├── answers.md                  ← 모든 답변 누적
├── planning/                   ← STEP 1~4 산출물

│   ├── code-analysis-v1.md
│   ├── scenario-v1.md
│   └── outline-v1.md

├── book/
│   ├── front/                  ← 머릿말, 깃헙소스, 프롤로그
│   ├── body/                   ← 본문 챕터
│   ├── back/                   ← 맺음말
│   ├── publish-info-pod.md     ← 출판정보 (POD)
│   └── publish-info-ebook.md   ← 출판정보 (전자책)

```

## 앞부속 구조

```
머릿말 → 깃헙소스/자료 받기 → 목차 → 프롤로그 → 챕터
```

- **머릿말**: 저자의 경험이 아니라 독자에게 "왜 배워야 하는가"를 알려주는 글
- **깃헙소스**: GitHub 레포 안내 + 환경 설정 (부록 대체)
- **프롤로그**: 코드 없이 전체 여정을 이야기로 보여주는 글 (A4 2페이지 이상)

## 핵심 규칙

- **상수는 질문하지 않는다**: 톤(존댓말), 스타일(스토리텔링), 챕터 구조(이야기→기술), 코드 분류(실습/설명/참고)
- **변수는 인사이트 질문으로 채운다**: 단순히 "뭐야?"가 아니라 "이건 생각해봤어?"
- **의도가 필터다**: seed.md의 의도가 이후 모든 결정의 기준
- **실패 먼저**: 정답부터 알려주지 않는다. 실수 → 에러 → 해결 순서
- **부록 없음**: 실습에 없는 심화는 챕터에도 넣지 않는다. 환경 설정은 깃헙소스에 포함
- **로드맵 없음**: 프롤로그가 전체 여정 미리보기 역할을 겸한다
- **독백은 이탤릭**: `*독백 내용*`
- **주인공은 오픈이**: 주변인물은 역할명만 (팀장, 선배, 동료)
