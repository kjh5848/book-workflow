# 검토 판정 규칙

> 출처: v1 verification/common-rules + v3 적응
> 관련: 검토 모드 3개 (인사이트/의도감시/감수)

---

## 판정 기준

| 판정 | 조건 | 다음 행동 |
|------|------|----------|
| **PASS** | 모든 필수 항목 통과 | 다음 STEP 진행 |
| **CONDITIONAL_PASS** | 필수 통과, 일부 권장 미통과 | 다음 STEP 진행 (경고 포함) |
| **FAIL** | 필수 항목 실패 | 수정 요청 → 재검토 |

---

## 재시도 프로토콜

```
FAIL → 저자에게 수정 요청 → 수정 → 재검토 → 최대 2회
  → 2회 초과 시 저자와 상의 (강제 통과 or 방향 전환)
```

## 검토 모드별 체크리스트

### 인사이트 검토 (STEP 1~5)

```
□ 저자 답변에서 모호하거나 구체화가 필요한 부분이 있는가?
□ 코드 분석 결과와 저자 답변 사이에 충돌이 있는가?
□ 이전 STEP 답변과 현재 답변 사이에 모순이 있는가?
□ 분량(100p 권장) 관점에서 현재 범위가 현실적인가?
□ 독자 관점에서 빠진 선행 개념이 있는가?
□ 저자가 아직 결정하지 않은 채 넘어간 부분이 있는가?
```

### 의도감시 검토 (STEP 5만)

```
□ 의도 일치     이 챕터의 내용이 "책의 의도"에 부합하는가?
□ 범위 이탈     "의도 밖 범위"에 해당하는 내용이 슬며시 들어오지 않았는가?
□ 코드 성격     예제 코드가 교육용인가? (실무 코드가 아닌가?)
□ 깊이 적절     이 책의 목적(이해 vs 구현)에 맞는 깊이인가?
□ 비유 독립     이야기 파트의 비유가 기술 파트 없이도 이해되는가?
□ 핵심 유지     챕터 핵심 메시지가 책 전체 핵심 메시지와 연결되는가?
□ 상수 준수     챕터 내부 구조(이야기→기술), 비유 길이, 코드 분류를 지켰는가?
```

### 감수 검토 (전 STEP)

| 편집장 | 관점 | 핵심 질문 스타일 |
|--------|------|----------------|
| **기술 감수자** | 정확성, 빠진 개념, 순서 | "독자가 OO를 모르는데 여기서 OO이 나와도 될까?" |
| **독자 대변인** | 난이도, 진입장벽, 완주 가능성 | "여기서 독자가 포기하지 않을까?" |
| **이야기 편집장** | 서사 흐름, 비유 적절성, 몰입도 | "이 도입이 설명으로 시작하는데, 경험담으로 열면 어떨까?" |

## 피드백 기록 형식

`review/feedback-log.md`에 누적:

```markdown
## [날짜] STEP [N] — [산출물명]
- **검토 유형**: [인사이트/의도감시/감수]
- **판정**: [PASS / CONDITIONAL_PASS / FAIL]
- **주요 피드백**: [요약]
- **수정 완료 여부**: [완료 / 미완료]
```
