## 핵심 톤: 대화체 스토리텔링

| 상황 | 예시 |
|------|------|
| 설명할 때 | "세션은 서버에 저장돼요." |
| 전환할 때 | "자, 이제 만들어볼까요?" |
| 강조할 때 | "이건 꼭 확인해보세요!" |
| 끌어들일 때 | "여러분은 어떻게 하시겠어요?" |

## 편집 규칙

### 볼드 텍스트
`**용어**` 양쪽에 띄어쓰기 필수.
```
❌ 이것은**중요한**부분입니다
✅ 이것은 **중요한** 부분입니다
```

### 능동태 우선
```
❌ "코드가 실행되어집니다"
✅ "코드를 실행하면"
```

### 영어 용어
처음 등장 시 한국어(English) 형태로:
```
✅ "의존성 주입(Dependency Injection)"
```
이후에는 한국어만 사용하거나 약어 사용.

### 이모지
본문에서 이모지 사용 금지. 박스 스타일에서만 허용.

## 피해야 할 표현

| 유형 | 예시 | 대안 |
|------|------|------|
| 진부한 표현 | "두말할 필요 없이" | 그냥 설명한다 |
| 주관적 감탄 | "정말 놀라운 기능!" | 구체적으로 뭐가 좋은지 |
| 시간 의존 | "최근에 나온" | 구체적 버전/날짜 |
| 불필요한 괄호 | "(물론 이것은...)" | 괄호 없이 자연스럽게 |

## 독자 수준별 적응 글쓰기

### 기본: 개념서 독자

- 배경지식은 있지만 이 주제만 모르는 독자
- **비유 먼저, 정의는 나중에**
- 한 번에 하나씩만 설명 (점진적 공개)

### 패턴: 비유 → 왜? → 정의

```markdown
세션은 놀이공원 팔찌 같은 거예요.               ← 비유
입장할 때 팔찌를 받으면, 그 다음부턴
팔찌만 보여주면 되잖아요?

그런데 왜 이런 게 필요할까요?                   ← 왜?
HTTP는 원래 기억력이 없거든요.
요청할 때마다 "저 누구예요"를 반복해야 해요.

(기술 파트에서)
세션(Session): 서버가 클라이언트별로              ← 정의
유지하는 상태 정보. 고유 ID로 식별.
```

## 존댓말 규칙

이야기 파트 내러티브와 설명문 모두 존댓말(-습니다/-입니다/-세요)로 쓴다.
- 내러티브: `"팀장이 왔다."` → `"팀장이 왔습니다."`
- 설명 문장: `~이다`, `~한다` → `~입니다`, `~합니다`
- **유지**: 캐릭터 대화 따옴표 안, 내면 독백 이탤릭 안 — 구어체 그대로

---

## 등장인물 역할 표기

스토리 속 직원은 실명 대신 역할로만 표기한다.
- 말하는 인물: `**팀장**`, `**나**`, `**동료**` 등 역할명
- 조회 대상: `A 사원`, `B 팀원`, `C 대리` (알파벳+직급)
- 예시 쿼리: `"진아 연차 몇 개?"` → `"A 사원 연차 몇 개?"`

## 이야기 파트 리듬 규칙

### 대화문
한 문장이면 한 줄로, 이어지는 내용이 많으면 문장 단위로 묶어 쓴다.
같은 화자의 연속 대사는 `<br>`로 이어도 되고, 하나의 따옴표로 합쳐도 된다.
```
✅ **팀장**: "AI로 사내 문서 검색 시스템 만들어봐. 직원들이 규정이나 정책 찾는 게 너무 번거롭다고 해서."
✅ **팀장**: "AI로 사내 문서 검색 시스템 만들어봐."<br>
           "직원들이 규정이나 정책 찾는 게 너무 번거롭다고 해서."
```
연속 대사에서 같은 화자 이름은 첫 줄에만 표시한다. 이후 줄은 따옴표만.

### 속마음은 이탤릭으로
등장인물의 내면 독백은 이탤릭(`*...*`)으로 표시한다. 괄호 `()`는 사용하지 않는다.
```
❌ (ChatGPT도 뭐든 대답하잖아. LLM에게 직접 물어보면 되는 거 아니야?)
✅ *ChatGPT도 뭐든 대답하잖아. LLM에게 직접 물어보면 되는 거 아니야?*
```

### 소설 문체 내러티브
이야기 파트 내러티브는 소설처럼 짧고 리드미컬하게 쓴다.
```
✅ 커넥트에 입사한 지 3일 차. 아직 사내 Wi-Fi 비밀번호를 포스트잇에 적어 모니터에 붙여놓던 시절.
   오전 10시, 아침부터 팀장이 저를 바라보며 다가오기 시작합니다.
```
- 명사구 단문: `"입사 3일 차."` (동사 없이 마침표)
- 동사 없는 절 허용: `"...던 시절."` (이었습니다 생략)
- 동적 진행형: `"팀장이 자리로 왔습니다."` → `"팀장이 저를 바라보며 다가오기 시작합니다."`
- `<br>` 태그는 내러티브에 쓰지 않는다

## 이미지 · 스크린샷 규칙

- **이미지는 플레이스홀더**: 집필 시 `[GEMINI PROMPT]`(개념도) / `[CAPTURE NEEDED]`(실습 캡처). 실제 이미지는 집필 완료 후.
- **스크린샷과 코드 블록 중복 금지**: 스크린샷 확보 시 동일 내용의 출력 코드 블록 제거.
- **기술은 저자가 결정**: AI가 제안, 저자가 최종 결정.

---

## 코드 블록 규칙

- 코드 블록에는 반드시 **언어 태그** 명시 (`java`, `python`, `bash` 등)
- 이야기 파트에 코드 블록 없음 (상수)
- 기술 파트에서 코드 보여줄 때: 목적 → 코드 → 설명 (참조: `code/references/code-explanation.md`)
- **[실습] 파일**: 전체 코드를 보여준다. "아래 코드를 `파일명`에 작성합니다"로 시작
- **[설명] 파일**: 핵심 흐름만 발췌. 전체 코드는 `code/exNN/` 참조
- **[참고] 파일**: 코드 블록 없음. 메서드 표로 요약
