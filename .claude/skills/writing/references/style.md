## 핵심 톤: 대화체 스토리텔링

| 상황 | 예시 |
|------|------|
| 설명할 때 | "세션은 서버에 저장돼요." |
| 전환할 때 | "자, 이제 만들어볼까요?" |
| 강조할 때 | "이건 꼭 확인해보세요!" |
| 끌어들일 때 | "여러분은 어떻게 하시겠어요?" |

## 편집 규칙

### 볼드 텍스트
`**용어**` 양쪽에 띄어쓰기 필수.
```
❌ 이것은**중요한**부분입니다
✅ 이것은 **중요한** 부분입니다
```

### 능동태 우선
```
❌ "코드가 실행되어집니다"
✅ "코드를 실행하면"
```

### 영어 용어
처음 등장 시 한국어(English) 형태로:
```
✅ "의존성 주입(Dependency Injection)"
```
이후에는 한국어만 사용하거나 약어 사용.

### 이모지
본문에서 이모지 사용 금지. 박스 스타일에서만 허용.

## 피해야 할 표현

| 유형 | 예시 | 대안 |
|------|------|------|
| 진부한 표현 | "두말할 필요 없이" | 그냥 설명한다 |
| 주관적 감탄 | "정말 놀라운 기능!" | 구체적으로 뭐가 좋은지 |
| 시간 의존 | "최근에 나온" | 구체적 버전/날짜 |
| 불필요한 괄호 | "(물론 이것은...)" | 괄호 없이 자연스럽게 |

## 독자 수준별 적응 글쓰기

### 기본: 개념서 독자

- 배경지식은 있지만 이 주제만 모르는 독자
- **비유 먼저, 정의는 나중에**
- 한 번에 하나씩만 설명 (점진적 공개)

### 패턴: 비유 → 왜? → 정의

```markdown
세션은 놀이공원 팔찌 같은 거예요.               ← 비유
입장할 때 팔찌를 받으면, 그 다음부턴
팔찌만 보여주면 되잖아요?

그런데 왜 이런 게 필요할까요?                   ← 왜?
HTTP는 원래 기억력이 없거든요.
요청할 때마다 "저 누구예요"를 반복해야 해요.

(기술 파트에서)
세션(Session): 서버가 클라이언트별로              ← 정의
유지하는 상태 정보. 고유 ID로 식별.
```

## 이야기 파트 리듬 규칙

### 대화문 줄바꿈
대화문은 한 문장씩 줄바꿈하여 티키타카 리듬을 만든다.
```
❌ "AI로 사내 문서 검색 시스템 만들어봐. 직원들이 규정이나 정책 찾는 게 너무 번거롭다고 해서. 채팅창에 물어보면 바로 답해주는 거."

✅ "AI로 사내 문서 검색 시스템 만들어봐."
   "직원들이 규정이나 정책 찾는 게 너무 번거롭다고 해서."
   "채팅창에 물어보면 바로 답해주는 거."
```

### 속마음은 괄호로
등장인물의 내면 독백은 `()` 괄호로 감싼다. 이탤릭(`*...*`)은 사용하지 않는다.
```
❌ *ChatGPT도 뭐든 대답하잖아. LLM에게 직접 물어보면 되는 거 아니야?*
✅ (ChatGPT도 뭐든 대답하잖아. LLM에게 직접 물어보면 되는 거 아니야?)
```

### 짧은 서술 줄바꿈
감정 전환이나 반전이 있는 짧은 문장은 줄바꿈으로 호흡을 끊는다.
```
❌ 그럴듯했다. 뭔가 공식적인 느낌도 났다.
✅ 그럴듯했다.
   뭔가 공식적인 느낌도 났다.
```

## 코드 블록 규칙

- 코드 블록에는 반드시 **언어 태그** 명시 (`java`, `python`, `bash` 등)
- 이야기 파트에 코드 블록 없음 (상수)
- 기술 파트에서 코드 보여줄 때: 목적 → 코드 → 설명 (참조: `code/references/code-explanation.md`)
- 전체 코드 대신 핵심만 보여주고, versions/ 폴더 참조
