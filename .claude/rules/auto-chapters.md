# 자동 트리거: chapters/*.md

챕터 파일을 작성하거나 수정할 때 자동으로 적용된다.

---

## writer (작가)

챕터는 **이야기 파트**와 **기술 파트**로 구성된다.

### 챕터 헤더 blockquote
- 챕터 제목 바로 아래에 `>` blockquote로 버전/요약/핵심개념 3줄 작성
- 각 줄 끝에 반드시 `<br>` 태그를 붙여야 PDF에서 줄바꿈이 정상 출력됨 → why-log.md#2026-03-15-19
- 형식: `> 이번 버전: exNN → exNN<br>`  `> 한 줄 요약: ...<br>`  `> 핵심 개념: ...`

### 이야기 파트
- 전체 존댓말. 캐릭터 대화/독백만 구어체
- 코드 블록 절대 없음. 비유와 서사로만 전달
- 비유는 일상적인 것만, 페이지당 3개 이하
- 내면 독백은 괄호 `()` 사용
- 등장인물은 역할명 (**팀장**/**나**/**동료**)

### 파트 전환
- 이야기 파트에서 기술 파트로 넘어갈 때 전환 문장 + `---`로 구분. 예: "이제 직접 만들어 보겠습니다."

### 레포 소개 (기술 파트 시작 직후)
- 기술 파트 시작 시 레포 소개 블록 필수. URL 테이블만으로 끝내지 마라 → why-log.md#2026-03-15-3
- 형식: git clone 명령어 → 파일 계층 트리([실습]/[설명]/[참고] 태그) → start 코드(실습용) vs end 코드(완성본) 안내
- 파일 트리는 outline-vN.md에 정의된 구조를 그대로 사용
- 닫기 문장: "챕터를 따라 하며 코드를 작성하고, 막히면 완성 코드를 참고하세요"

### 기술 파트
- 코드 블록 언어 태그 필수
- 핵심만 10~15줄. 전체는 `code/exNN/` 참조
- 구조: 목적 한 줄 → 코드 → 설명 → 실행 결과
- 원본 코드(code/)가 진실의 원천. 불일치 시 챕터를 코드에 맞춤

### 공통
- 금지: 설교, 반복 강조, 정의 전 용어, 이모지
- **볼드** 양쪽 띄어쓰기, 문맥 전환 시 빈 줄
- 챕터 구조: 문제 → 기술소개 → 해결 → 결과
- 닫기: "이것만은 기억하자" + 다음 예고
- 챕터를 이미지 플레이스홀더 없이 완성하지 마라. 실행 결과 → `[CAPTURE NEEDED]`, 아키텍처/흐름도 → `[GEMINI PROMPT]` 삽입 필수 → why-log.md#2026-03-15-1
- `<img>` 경로를 변경할 때 실제 파일도 반드시 해당 경로로 이동하라. 경로만 바꾸고 파일을 그대로 두면 깨진다 → why-log.md#2026-03-15-6

## editor (편집장)

챕터 수정 시 **의도감시(D5)** 관점으로 점검한다.

### 점검 항목
- 의도 일치: 챕터 내용이 seed.md의 의도에 부합하는가
- 범위 이탈: 의도 밖 내용이 슬며시 들어오지 않았는가
- 코드 성격: 예제 코드가 교육용인가
- 깊이 적절: 책의 목적에 맞는 깊이인가
- 파트 분리: 이야기 파트에 코드가 섞이지 않았는가

### 필수 스킬
- review (skills/review/SKILL.md) — D5.의도-대조기
