# 책 쓰기 워크플로우

기술서를 이야기처럼 쓰는 워크플로우.
저자(도메인 전문가)와 AI가 티키타카하며 책을 완성한다.

---

## 워크플로우 개요

이 워크플로우는 **완성된 코드를 가지고** 기술서를 **이야기처럼** 쓰는 전체 과정을 정의한다.
저자(도메인 전문가)와 AI가 **10단계**에 걸쳐 대화하며 책을 완성한다.

### 전체 흐름 한눈에 보기

```
완성 코드 분석 → 씨앗(주제+의도) → 시나리오 → 코드 실습 설계 → 여정 → 목차+기술매핑
    ↓
챕터 집필(반복) → 리뷰(반복) → 프롤로그+로드맵 → 마무리
```

### 왜 이 순서인가?

| STEP | 이름 | 왜 이 순서? |
| --- | --- | --- |
| **1. 코드 분석** | 완성 코드 분석 + 버전 분해 | **코드가 먼저 있다.** 코드를 분석하고 버전별로 쪼개야 나머지를 설계할 수 있다 |
| **2. 씨앗** | 주제, 동기, 의도, 핵심 메시지 | 코드 분석 결과를 보면서 "이 책은 뭐다"를 잡는다. 의도(뭘 이해시킬 것인가)가 이후 모든 결정의 필터 |
| **3. 시나리오** | 책 전체를 관통하는 이야기 | 주제와 의도를 알아야 이야기를 설계할 수 있다 |
| **4. 코드 실습 설계** | 챕터별 코드 분류 | 시나리오를 알아야 챕터별로 뭘 실습하고 뭘 넘길지 정할 수 있다 |
| **5. 여정** | 독자의 학습 경로 | 시나리오 + 버전 분해 + 실습 설계를 기반으로 독자가 밟을 길을 그린다 |
| **6. 뼈대+기술매핑** | 목차 + 챕터별 기술 | 여정을 목차로 구체화하고, 완성 코드의 기술을 매핑한다 |
| **7. 살붙이기** | 챕터별 집필 | 이야기 파트 + 기술 파트 초안 작성. 챕터(=버전)마다 반복 |
| **8. 다듬기** | 챕터별 리뷰 | 룰 위반 체크 + 이야기/기술 분리 검증. 챕터마다 반복 |
| **9. 프롤로그+로드맵** | 숲 보기 + 성장 안내 | 모든 챕터가 끝나야 전체를 조망하는 프롤로그를 쓸 수 있다 |
| **10. 마무리** | 서문, 에필로그, 부록 | 책의 처음과 끝을 닫는다 |

### 구성 요소

이 워크플로우는 다음 구성 요소로 이루어져 있다:

- **상수 (Constants)** — 책을 시작하기 전에 고정하는 것들 (문체, 포맷, 구조 등)
- **변수 (Variables)** — AI가 질문해서 채우는 것들 (주제, 독자, 시나리오 등)
- **룰 (Rules)** — 글 쓸 때 지키는 규칙 10가지
- **스킬 (Skills)** — 단일 작업 도구 15개 (비유 생성, 코드 분석, diff 생성 등)
- **에이전트 (Agents)** — 스텝을 이끄는 대화 진행자 7개 + 감수 에이전트 1개
- **버전 관리** — git으로 산출물 이력 추적

### 핵심 원칙 요약

1. **의도가 필터다** — 책의 의도(뭘 이해시킬 것인가)가 모든 결정의 기준. 의도 밖은 넣지 않는다
2. **완성 코드 먼저** — 코드가 있어야 책을 쓸 수 있다
3. **코드 → 버전 분해** — 완성 코드를 독자가 따라갈 수 있는 스냅샷으로 쪼갠다
4. **코드 3단계 분류** — 실습(직접 작성) / 설명(코드 설명) / 참고(계층+한줄)
5. **챕터 = 한 버전 성장** — 매 챕터에서 프로젝트가 한 단계 올라간다
6. **이야기 파트 / 기술 파트 분리** — 비유가 흐름을 끊지 않게
7. **기술은 저자가 결정** — AI가 제안, 저자가 순서/대체/이동

## 상수 (Constants) — 책을 시작하기 전에 고정하는 것들

### 문체 상수

| 상수 | 값 | 설명 |
| --- | --- | --- |
| 포맷 | 짧은 쿡북 | 두꺼운 교과서가 아니라 짧고 임팩트 있게 |
| 문체 | 스토리텔링 | 설명이 아니라 이야기로 쓴다 |
| 언어 수준 | 초등학생도 이해 가능 | 어려운 용어 최대한 자제 |
| 톤 | 대화체 | 독자에게 말 거는 느낌 |
| 비유 | 필수 (단, 짧게) | 이야기 흐름을 끊지 않는 선에서 |
| 용어 정리 | 필수 | 이야기에서 비유로 넘어간 용어는 기술 파트에서 반드시 정식 정의/설명 |

### 책 유형 상수

| 상수 | 값 | 설명 |
| --- | --- | --- |
| 목적 | 개념서 | 코드가 아니라 숲을 보여준다 |
| 난이도 | 주제만 새로움 | 배경지식은 갖춘 독자. 책의 주제(내용)만 모른다 |
| 독자 수준 | 배경지식 완비 | 해당 책을 읽기 위한 배경지식은 다 있는 상태. 책의 주제만 모른다. 질문으로 수집하지 않는다 (상수) |
| 코드 비중 | 낮음 | 코드보다 개념과 흐름 중심 |
| 성장 방식 | 버전별 성장 | 챕터마다 이전 결과물을 조금씩 고치며 키워간다 |
| 코드 기반 | 완성 코드 먼저 | 완성된 코드가 먼저 있고, 그걸 버전별로 쪼개서 책을 쓴다 |

### 책 구조 상수 (독자가 읽는 순서)

```
┌─ 프롤로그: 숲 보기 (코드 없이 전체 개념을 이야기로)
├─ 로드맵: 뭘 만들 건지, 버전별로 어떻게 성장하는지
├─ 본문 챕터들 (v0.1 → v0.2 → ... → v1.0 성장)
│   └─ 각 챕터: [이야기 파트] → [기술 파트]
├─ 서문 / 에필로그 / 부록
```

### 챕터 내부 구조 상수 (이야기와 기술을 분리)

```
┌─────────────────────────────────────────────┐
│  이야기 파트 (흐름을 끊지 않는다)                │
│  - 도입 이야기 (경험, 상황)                     │
│  - 개념 설명 (비유는 1~2문장으로 짧게)            │
│  - 시나리오 진행 (이번 버전에서 뭘 바꾸나)         │
│  - "이것만은 기억하자" + 다음 챕터 예고           │
│                                               │
│  ※ 이야기 속 비유로 넘어간 용어들은               │
│    기술 파트에서 반드시 정식으로 정의한다           │
├─────────────────────────────────────────────┤
│  기술 파트 (이야기 끝난 후)                      │
│  - 용어 정리 (비유로 설명한 것의 정식 정의)         │
│  - 파일 계층 구조 ([실습] / [설명] / [참고] 분류)  │
│  - 실습 코드 (독자가 직접 작성하는 핵심 코드)      │
│  - 설명 코드 (코드를 보여주며 설명만)              │
│  - 더 알아보기 (심화 링크/참고)                   │
└─────────────────────────────────────────────┘
```

> 왜 분리하나? 비유 설명이 길어지면 이야기 문맥을 잃어버린다.
이야기는 이야기대로 쭉 흘러가게 하고, 기술 궁금한 건 뒤에서 풀어준다.
단, 비유로 넘어간 용어는 반드시 기술 파트에서 정식 정의를 해준다.
> 

> 왜 코드를 분류하나? 모든 코드를 다 작성하면 핵심을 놓친다.
MSA 책에서 Saga를 설명하는 챕터라면, JPA 코드를 설명할 필요 없다.
핵심 주제 코드만 실습하고, 나머지는 설명하거나 계층 구조로 보여준다.
>

## 변수 (Variables) — AI가 질문해서 채우는 것들

책마다, 챕터마다 달라지는 내용. **AI가 질문을 던져서 수집한다.**

### 책 레벨 변수 (한 번만 수집)

| 변수 | 수집 시점 | 예시 |
| --- | --- | --- |
| 완성 코드 경로 | STEP 1 | `source/` 폴더에 저자가 올린 완성 프로젝트 |
| 버전 분해 결과 | STEP 1 | 완성 코드를 v0.1~v1.0으로 쪼갠 결과 |
| 버전 로드맵 | STEP 1 | v0.1 Hello World → v0.2 글쓰기 → ... → v1.0 배포 |
| 주제 | STEP 2 | "Kubernetes", "스프링부트 웹 개발" |
| 집필 동기 | STEP 2 | "이 기술 도입하고 삽질한 경험을 나누고 싶어" |
| 저자 경험 | STEP 2 | "5년간 3개 회사에서 인프라 구축" |
| 차별점 | STEP 2 | "기존 책들은 코드만 나열, 나는 왜를 설명" |
| 핵심 메시지 | STEP 2 | "이건 결국 OO 문제를 풀기 위한 거다" |
| 책의 의도 | STEP 2 | "JWT 자체를 이해시키는 것 (실무 적용이 아님)" |
| 의도 밖 범위 | STEP 2 | "Spring Security 연동, 프로덕션 배포 등은 다루지 않음" |
| 시나리오 | STEP 3 | "블로그 플랫폼을 처음부터 끝까지 만들어가는 이야기" |
| 코드 실습 맵 | STEP 4 | 버전별 코드 파일을 실습/설명/참고로 분류한 결과 |
| 여정 맵 | STEP 5 | 출발점 → 기초 → 전환점 → 심화 → 실전 |

### 챕터 레벨 변수 (챕터마다 수집)

| 변수 | 수집 시점 | 예시 |
| --- | --- | --- |
| 이번 버전 목표 | STEP 6 | "v0.3: 로그인 기능 추가" |
| 이전 버전에서 뭘 바꾸나 | STEP 6 | "v0.2에 HttpSession으로 인증 붙이기" |
| 챕터 핵심 메시지 | STEP 6 | "인증은 결국 '너 누구야?' 확인하는 것" |
| 도입 이야기 | STEP 6 | "비밀번호 없이 서비스 열었다가 생긴 일" |
| 핵심 개념 (3개 이하) | STEP 6 | 세션, 쿠키, 인증/인가 |
| 개념별 비유 (짧게) | STEP 6 | "세션은 놀이공원 팔찌" |
| 실전 경험/실패담 | STEP 6 | "세션 안 끊어서 메모리 터진 이야기" |
| 한줄 기억 | STEP 6 | "로그인은 문 열기, 인가는 방 골라 들어가기" |
| 기술 파트 상세 | STEP 6 | HttpSession API, 세션 저장소 종류 등 |

## 룰 (Rules) — 글 쓸 때 지키는 규칙

1. **이야기 먼저**: 모든 챕터는 이야기 파트로 시작한다
2. **비유는 짧게**: 비유는 1~2문장. 길어지면 이야기 흐름이 끊긴다
3. **기술은 뒤에서**: 이야기 파트가 끝난 후 기술 파트에서 상세 설명
4. **비유 → 정의 필수**: 이야기에서 비유로 넘어간 용어는 기술 파트 첫 머리에서 반드시 정식 정의
5. **한 챕터 한 버전**: 챕터가 끝나면 결과물이 한 단계 성장해 있다
6. **점진적 깊이**: 쉬운 것 → 어려운 것, 그림 → 글
7. **코드 최소화**: 이야기 파트에는 코드 없음. 기술 파트에서만, 3~5줄 이내
8. **챕터 닫기**: "이것만은 기억하자" + 다음 버전 예고
9. **기술은 저자가 결정**: AI가 제안하되, 저자가 순서/대체/이동을 최종 결정
10. **코드 3단계 분류**: 실습(독자가 직접 작성) / 설명(코드 설명만) / 참고(계층 구조 + 한 줄 설명). 실습 코드는 챕터 핵심 주제에 해당하는 것만

## 스킬 (Skills) — 단일 작업 도구

| 스킬 | 하는 일 | 사용 시점 |
| --- | --- | --- |
| `코드-분석기` | 완성 코드의 구조/의존성/기능을 분석 | STEP 1 (코드 분석) |
| `버전-분해기` | 완성 코드를 버전별 스냅샷으로 쪼개기 | STEP 1 (코드 분석) |
| `diff-생성기` | 버전 간 차이점 요약 (v0.2→v0.3에서 뭐가 바뀌었나) | STEP 1 (코드 분석) |
| `시나리오-요약기` | 전체 시나리오 + 버전 로드맵을 요약 | STEP 3 (시나리오 확정) |
| `코드-실습-분류기` | 버전별 코드를 실습/설명/참고로 분류 + 계층 구조 생성 | STEP 4 (코드 실습 설계) |
| `비유-생성기` | 기술 용어 → 1~2문장 짧은 비유로 변환 | STEP 7 (이야기 파트 비유) |
| `이야기-오프너` | 챕터 도입부를 저자 경험 기반 이야기로 작성 | STEP 7 (챕터 시작) |
| `기술-매퍼` | 챕터 개념에 맞는 기술 후보를 제안 | STEP 6 (기술 매핑) |
| `용어-검사기` | 이야기 파트에서 어려운 용어/긴 비유 찾아 경고 | STEP 8 (다듬기) |
| `챕터-요약기` | 챕터 내용을 "이것만은 기억하자" 3줄로 압축 | STEP 7 (챕터 마무리) |
| `브릿지-작성기` | 현재 버전 → 다음 버전 연결 문장 생성 | STEP 8 (연결 다듬기) |
| `페르소나-체크` | 독자가 이해할 수 있는지 검증 | STEP 8 (톤 검사) |
| `이야기-기술-분리기` | 초안에서 이야기 파트와 기술 파트를 분리 | STEP 8 (구조 정리) |
| `프롤로그-생성기` | 전체 챕터에서 코드 빼고 개념만 추출하여 숲 보기 작성 | STEP 9 (프롤로그) |
| `로드맵-생성기` | 버전별 성장 과정을 시각적 로드맵으로 작성 | STEP 9 (로드맵) |

## 에이전트 (Agents) — 스텝을 이끄는 대화 진행자

| 에이전트 | 담당 스텝 | 하는 일 |
| --- | --- | --- |
| `코드분해-에이전트` | STEP 1, 4 | 완성 코드 분석 → 버전 분해 → 코드 실습 분류 (저자 조정) |
| `씨앗-에이전트` | STEP 2 | 질문을 던져 책 변수 수집 (주제, 의도, 핵심 메시지) |
| `시나리오-에이전트` | STEP 3 | 시나리오 설계 |
| `설계-에이전트` | STEP 5~6 | 여정 맵 → 목차 → 기술 매핑 (저자 조정 포함) |
| `집필-에이전트` | STEP 7 | 챕터별 변수 수집 + 이야기 파트/기술 파트 초안 작성 |
| `리뷰-에이전트` | STEP 8 | 룰 위반 체크 + 이야기/기술 분리 검증 + 다듬기 |
| `마무리-에이전트` | STEP 9~10 | 프롤로그 + 로드맵 + 서문/에필로그/부록 작성 |
| `감수-에이전트` | 전 STEP | 3인 편집장(기술 감수자, 독자 대변인, 이야기 편집장) 관점으로 질문 감수 |

### 감수-에이전트 상세 — 3인 편집장이 질문으로 감수한다

매 STEP 산출물이 완성되면, 저자 확인 **전에** 3인 편집장이 각자 관점에서 질문을 던진다.
저자는 질문에 답하면서 산출물 수정 여부를 결정한다. 무시해도 OK.

### 3인 편집장 구성

| 편집장 | 관점 | 핵심 질문 |
| --- | --- | --- |
| **기술 감수자** | 기술 정확성, 빠진 개념, 순서 최적화 | "독자가 DI를 모르는데 Ch.5에서 @Autowired가 나와도 될까?" |
| **독자 대변인** | 독자 눈높이, 난이도 곡선, 진입장벽 | "Ch.3→Ch.4 난이도가 갑자기 뛰는데, 독자가 여기서 포기하지 않을까?" |
| **이야기 편집장** | 서사 흐름, 비유 적절성, 몰입도 | "Ch.2 도입이 설명으로 시작하는데, 경험담으로 열면 어떨까?" |

### 동작 방식

1. **발동 시점**: 각 STEP 산출물 완성 → 편집장 감수 → 저자 확인
2. **개입 방식**: 질문 중심 — 질문으로 문제를 짚고, 저자가 스스로 깨닫게 유도
3. **출력 형식**: 3인이 각자 2~3개 질문
4. **저자 반응**: 질문에 답하면서 수정 여부 결정. "넘어가자"도 가능

### STEP별 감수 포인트

| STEP | 기술 감수자 | 독자 대변인 | 이야기 편집장 |
| --- | --- | --- | --- |
| 1. 코드 분석 | 버전 분해 순서가 기술 의존성에 맞나? 빠진 핵심 개념은? | 첫 버전이 너무 어렵지 않나? 성과가 눈에 보이나? | — |
| 2. 씨앗 | 핵심 메시지가 기술적으로 정확한가? 의도 범위가 적절한가? | — | 차별점이 독자에게 와닿는 이야기가 되나? |
| 3. 시나리오 | 시나리오가 기술 학습 순서와 맞나? | 시나리오에 독자가 감정이입할 수 있나? | 전환점이 극적인가? 긴장감이 있나? |
| 4. 코드 실습 설계 | 실습 분류가 기술 의존성에 맞나? 빠진 핵심 실습은? | 실습량이 적절한가? 너무 많거나 적지 않나? | — |
| 5. 여정 | 난이도 곡선에 기술적 비약은 없나? | 아하 모먼트가 적절한 위치에 있나? | — |
| 6. 목차+기술매핑 | **핵심**: 빠진 기술/개념은? 순서 최적화 제안 | 챕터 수가 적절한가? 독자가 완주할 수 있나? | 챕터 제목이 호기심을 유발하나? |
| 7. 살붙이기 | 기술 설명이 정확한가? 더 좋은 예시는? | 이 챕터만 읽어도 이해되나? | 도입 이야기가 몰입감 있나? 비유가 적절한가? |
| 8. 다듬기 | (리뷰-에이전트와 협업) | 앞뒤 챕터 연결이 자연스러운가? | 이야기 흐름이 끊기는 곳은? |

## 버전 관리 — git으로 산출물 이력 추적

산출물은 처음부터 완벽할 수 없다. 수정은 당연히 일어나고,
**이전 버전과 비교할 수 있어야 한다.** → git을 쓴다.

### 커밋 규칙

| 언제 커밋하나 | 커밋 메시지 형식 | 예시 |
| --- | --- | --- |
| STEP 완료 시 | `step{N}: {파일명} 초안 완성` | `step1: 01-code-analysis.md 초안 완성` |
| 저자 확인 후 | `step{N}: {파일명} 저자 확인 완료` | `step3: 03-reader.md 저자 확인 완료` |
| 수정 시 | `revise: {파일명} - {바꾼 이유}` | `revise: 06-toc.md - Ch.3 Security→HttpSession 변경` |
| 챕터 초안 | `draft: ch{NN} 초안` | `draft: ch03 초안` |
| 챕터 리뷰 후 | `review: ch{NN} 리뷰 반영` | `review: ch03 리뷰 반영` |

### 비교하는 법

```bash
# 특정 파일의 변경 이력 보기
git log --oneline -- 06-toc.md

# 이전 버전과 비교 (마지막 수정 전 vs 현재)
git diff HEAD~1 -- 06-toc.md

# 특정 시점과 비교
git diff abc1234 -- 06-toc.md

# 특정 STEP 완료 시점으로 돌아가기 (확인만)
git show abc1234:06-toc.md
```

### 수정 워크플로우

```
저자: "목차에서 Ch.3 기술을 바꾸고 싶어"
        ↓
AI: 06-toc.md 수정
        ↓
AI: git diff로 변경 내용 보여줌
  "Ch.3 기술: Spring Security → HttpSession
   Ch.10 추가: Spring Security (기존 Ch.3에서 이동)"
        ↓
저자: "맞아, 이걸로 가자"
        ↓
AI: 커밋 → revise: 06-toc.md - Ch.3 Security→HttpSession 변경
        ↓
AI: 연쇄 영향 체크
  "03-scenario.md의 버전 로드맵도 수정 필요해.
   05-journey.md 난이도 스파이크 구간도 바뀔 수 있어."
        ↓
연쇄 수정 → 각각 커밋
```

> 핵심: **수정할 때마다 커밋 + 이유 기록**. 나중에 "왜 바꿨지?" 할 때 `git log`로 추적 가능.
산출물 안에 이력을 쌓지 않는다 — 파일은 항상 최신 상태만 유지하고, 이력은 git이 관리.
>

## 질문 흐름

### STEP 1. 코드 분석 — "완성 코드를 분석하고 버전별로 쪼개자"

> 담당: `코드분해-에이전트` + `코드-분석기` + `버전-분해기` + `diff-생성기` 스킬
**전제: 저자가 `source/` 폴더에 완성 코드를 이미 올려둔 상태**
완성 코드를 분석하고, 독자가 따라갈 수 있는 버전별 스냅샷으로 쪼갠다.
> 

### 1-A. 완성 코드 분석

1. **완성 코드 올려줬어? 어떤 구조야?** → `코드-분석기` 스킬이 source/ 분석
2. **이 코드에서 핵심 기능을 나열하면?** (AI가 분석 결과 보여줌 → 저자 확인)
3. **이 기능들 중에 독자에게 안 보여줘도 되는 건?** (책 범위 밖 기능 제외)

```
AI 분석 결과 예시:
  source/ 분석 완료.
  - 기능 목록: 회원가입, 로그인(HttpSession), 글 CRUD, 댓글, 파일 업로드, 배포 스크립트
  - 주요 기술: Spring Boot, JPA, H2, Thymeleaf, HttpSession
  - 의존성: spring-boot-starter-web, spring-boot-starter-data-jpa, ...
```

### 1-B. 버전 분해 (AI 제안 → 저자 조정)

1. **이 기능들을 어떤 순서로 하나씩 붙여나갈까?** (AI가 제안 → 저자 조정)
2. **각 버전이 끝나면 독자 눈에 뭐가 보여야 해?** (눈에 보이는 성과)
3. **순서를 바꾸고 싶은 기능이 있어?**

```
AI 제안:
  v0.1: Hello World만 (Controller 1개)
  v0.2: 글 목록 (JPA Entity + Repository + 목록 페이지)
  v0.3: 글쓰기 (Form + POST 처리)
  v0.4: 로그인 (HttpSession)
  ...
  v1.0: 배포 (jar 빌드 + 서버 올리기)

저자 조정:
  "v0.2에서 JPA 바로 가지 말고 먼저 ArrayList로 하고,
   v0.3에서 JPA로 바꾸는 걸로 하자.
   왜 DB가 필요한지 느끼게 해야 해."
```

### 1-C. 버전별 코드 생성

1. **각 버전 코드를 `versions/` 폴더에 만들어줄까?** → `버전-분해기` 스킬
2. **버전 간 차이 확인해보자** → `diff-생성기` 스킬

```
diff-생성기 결과 예시:
  v0.2 → v0.3 변경점:
  + PostRepository.java (신규)
  + application.yml에 H2 설정 추가
  ~ PostController.java: ArrayList → Repository 주입으로 변경
  ~ post-list.html: 변경 없음
```

→ **생성: `01-code-analysis.md`** + `versions/v0.1/`, `versions/v0.2/`, ...

### STEP 2. 씨앗 — "무슨 책을 쓰고 싶어?"

> 담당: `씨앗-에이전트`
> 
1. **뭐에 대한 책이야?** → 변수: 주제
2. **왜 이 책을 쓰고 싶어?** → 변수: 집필 동기
3. **이 분야에서 너의 경험은?** → 변수: 저자 경험
4. **기존에 나와있는 비슷한 책들 봤어? 뭐가 아쉬웠어?** → 변수: 차별점
5. **이 책에서 네가 제일 하고 싶은 말 하나만 꼽으면?** → 변수: 핵심 메시지
6. **이 책의 의도가 뭐야? 독자가 뭘 "이해"하게 하고 싶어?** → 변수: 책의 의도
7. **그러면 이건 안 다뤄도 되는 거지?** (완성 코드에는 있지만 의도 밖인 것들) → 변수: 의도 밖 범위

```
Q: "이 책의 의도가 뭐야?"

```

### STEP 3. 시나리오 — "이 책의 이야기를 들려줘"

> 담당: `시나리오-에이전트`
책 전체를 관통하는 시나리오를 잡는다. (버전 분해는 STEP 1에서 이미 했다)

4. **독자가 이 시나리오를 따라가면 마지막에 뭐가 완성돼?** (최종 결과물)

→ AI 정리: 시나리오 요약 → 저자 확인

### STEP 4. 코드 실습 설계 — "독자가 뭘 직접 쓰고, 뭘 읽기만 할까?"

> 담당: `코드분해-에이전트` + `코드-실습-분류기` 스킬
각 버전(=챕터)에서 독자가 **직접 작성할 코드**, **설명만 읽을 코드**, **계층 구조만 볼 코드**를 분류한다.
모든 코드를 다 작성하지 않는다. 챕터의 핵심 주제에 해당하는 코드만 실습한다.
**의도 필터 적용**: `02-seed.md`의 "의도 밖 범위"에 해당하는 코드는 분류 대상에서 제외한다.
> 

### 코드 분류 기준

| 분류 | 표시 | 의미 | 독자 액션 |
| --- | --- | --- | --- |
| 실습 | `[실습]` | 챕터 핵심 주제 코드 | 독자가 직접 작성 |
| 설명 | `[설명]` | 중요하지만 핵심은 아닌 코드 | 코드 보면서 읽고 이해 |
| 참고 | `[참고]` | 이 챕터 주제가 아닌 코드 | 파일명 + 한 줄 설명만 확인 |

### 질문 흐름

1. **각 버전의 파일 계층 구조를 그려봤어. 확인해봐** → `코드-실습-분류기`가 versions/ 기반으로 트리 생성
2. **이 버전에서 핵심 주제가 뭐야?** (예: v0.4는 "인증", Saga 챕터면 "Saga 패턴")
3. **그 주제와 직접 관련된 코드는 어떤 파일이야?** → [실습] 코드 (독자가 직접 작성)
4. **직접 작성하진 않지만 코드 설명은 해줘야 할 건?** → [설명] 코드
5. **나머지는 계층 구조에 한 줄 설명만 보여줄게. 빠진 거 있어?** → [참고] 코드

→ AI 분류 → 저자 확인/조정 (버전마다 반복)

### 9-B. 로드맵 (버전별 성장 안내)

1. **로드맵을 보고 독자가 "나도 이렇게 만들어갈 수 있겠다" 느껴져?**

→ `로드맵-생성기`: 버전별 성장 과정을 시각적으로
→ **생성: `book/00-roadmap.md`**

```markdown
# 이 책의 로드맵

> 이 책은 하나의 프로젝트를 버전별로 키워가며 배웁니다.
> 각 챕터가 끝날 때마다 여러분의 프로젝트가 한 단계 성장합니다.

v0.1 ──→ v0.2 ──→ v0.3 ──→ v0.4 ──→ ... ──→ v1.0
Hello 글 목록 글쓰기 로그인 배포!
World

| 버전 | 챕터 | 뭘 배우나 | 끝나면 뭐가 되나 |
| ---- | ---- | --------- | ---------------- |
| v0.1 | Ch.1 | ...       | ...              |
| v0.2 | Ch.2 | ...       | ...              |
| ...  | ...  | ...       | ...              |
```

### 독자가 읽는 순서

```
1. 프롤로그 (숲 보기 — 코드 없이 전체 개념을 이야기로)
2. 로드맵 (뭘 만들 건지, 버전별 성장 안내)
3. 본문 Ch.1 ~ Ch.N (버전별 성장: 이야기 파트 → 기술 파트)
4. 서문 / 에필로그 / 부록
```

### 핵심 원칙

- **의도가 필터다** — "이 책은 뭘 이해시키는 게 목적인가?"가 모든 결정의 기준. 완성 코드에 있어도 의도 밖이면 넣지 않는다
- **완성 코드 먼저** — 코드가 있어야 책을 쓸 수 있다. source/에 올려두고 시작
- **코드 분석이 첫 번째** — 코드를 먼저 분석하고 버전별로 쪼개야 나머지를 설계할 수 있다
- **코드 → 버전 분해** — 완성 코드를 독자가 따라갈 수 있는 버전별 스냅샷으로 쪼갠다
- **코드 3단계 분류** — 실습(직접 작성) / 설명(코드 설명) / 참고(계층+한줄). 핵심 주제만 실습
- **상수**는 `config.md`에 한 번 박아두고 모든 단계에서 참조
- **변수**는 질문으로 채운다
- **시나리오 + 버전 로드맵**이 책 전체를 관통하는 뼈대
- **챕터 = 한 버전 성장** — 매 챕터에서 versions/v0.X/ 코드를 참조하며 쓴다
- **이야기 파트 / 기술 파트 분리** — 비유가 흐름을 끊지 않게
- **프롤로그에서 숲 먼저** — 코드 없이 전체를 보여주고 책 시작
- **기술은 저자가 결정** — AI가 제안, 저자가 순서/대체/이동
