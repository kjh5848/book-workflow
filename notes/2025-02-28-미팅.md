# 2/28 북 에이전트 미팅

# 1. 협의사항

## 1. Orchestrator 진행 흐름

- 질문지 튜닝 진행
- 드래프트 파일(초안) 생성
- 챕터 수는 정의하지 않음
- 포함하거나 제외해야 할 주제 선정
- 질문 리스트 작성
- 상수 값 정의
- 질문 폴더 운영
    - 즉시 필요한 내용 작성
    - 결과 피드백 작성
    - 상태 관리: `pending → done`
- 초안 ↔ 최신본 지속 검증
    - 초반 구조가 고정되면 이후 흐름이 깨질 수 있음
- 초안에 프롤로그 시나리오 포함 (스토리 필수)

---

## 2. 진행 순서

1. 시나리오 먼저 수집
2. 이후 초안 생성

> 기술 내용은 초기에 다루지 않음
> 
> 
> 상세 기술은 아키텍처 단계에서 질문
> 

---

## 3. 시나리오 구성 요소

시나리오에는 반드시 포함되어야 함:

- 받아야 할 값
- 상수 값

---

## 4. 질문 방식

- 스무고개 형식으로 진행
- 상호 질문 기반

---

## 5. Question 관리 구조

```
questions/
 ├─ pending/
 └─ done/
```

각 질문에는 포함:

- 즉시 필요한 내용
- 결과 피드백

---

## 6. 페르소나 설정

- 최대 2개 사용

---

## 7. 스토리텔링 규칙

- 어려운 용어 등장 시 흐름을 끊지 않음
- 먼저 쉬운 비유 제시
- 이야기 맥락 종료 후 개념 정의 진행 (페르소나 활용)

---

## 8. 책의 성격

- 개념서임을 명시해야 함
- 누구나 읽기 쉽고 이해 가능해야 함

# 2. 워크플로우 협의 사항

## 0. 코드 분석 (출발점)

> 코드의 맥락을 모르면 책을 쓸 수 없다.
> 

책 제작의 시작은 기술이 아니라

👉 코드 이해이다.

### 0-1. 프로젝트 분석

- 전체 구조 파악
- 흐름 이해
- 어떤 문제를 해결하는 코드인지 확인

### 0-2. 핵심 로직 분석

- 주요 도메인 흐름 정리
- 핵심 처리 로직 정리

---

## 1. 책 작성 의도 시작

```
"스프링 책을 쓰고 싶어"
```

오케스트레이터의 첫 질문:

👉 어떤 시나리오의 책인가?

---

## 2. 상세 시나리오 주입

- 해결하려는 문제
- 대상 독자
- 다루고 싶은 상황
- 전달하고 싶은 메시지

---

## 3. 시나리오 정리

역할:

- 스프링 전문가 관점에서 시나리오 정리
- 풍부한 시나리오를 구조화
- 흐름 중심으로 재정리

---

## 4. 기술 기술서 수집

시나리오가 정리되면

→ 관련 기술 흐름 수집

→ 기술 순서도는 사람이 튜닝

---

## 5. 시나리오 기반 목차 생성

시나리오를 기준으로

- 흐름 중심 목차 생성

---

## 6. 질문지 생성

시나리오 기반으로

- 추가 명확화 질문 생성

---

## 7. 목차 보완

코드 & 시나리오 주입 단계에서

- 필요한 이론 제안
- 선택적으로 반영

---

## 8. Workflow 검증 Agent 필요

각 단계 종료 시

👉 자동 검사 수행

역할:

- 부족한 요소 탐지
- 추가 제안 생성

---

## 9. 글의 의도 / 배경

필수 요소:

- 글의 의도
- 배경

👉 의도 관련 질문은 반드시 수행

👉 하지만 의도는 **상수값**

---

## 10. Task 처리 방식

- Task는 동적으로 입력
- 페이지 수는 변수로 처리
