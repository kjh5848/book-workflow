# 9장. RabbitMQ 쿡북: GitHub 파일 변경 감지 → 발행/구독 자동 동기화

## 9.1 시나리오와 전체 구조

- “관찰 서버(Producer)” ↔ RabbitMQ ↔ “수신 서버(Consumer)”
- 변경 감지(Polling) → 메시지 발행 → 파일 반영 흐름

---

## 9.2 실습 준비

- Java/Gradle 버전 확인(JDK 17, Gradle 8.x 기준)
- Docker 설치 및 RabbitMQ 로컬 실행
- GitHub 토큰 발급 및 레포 권한 확인(contents:read)
- GitHub 레포지토리 준비(README.md 1개, 경로 명시)
- Spring 서버 2개 프로젝트 생성(Producer/Consumer)

---

## 9.3 GitHub 파일 Polling 설계

- 어떤 값을 비교할 것인가(sha / etag / lastModified / content hash)
- Polling 주기 전략과 주의사항(레이트리밋/중복)

---

## 9.4 관찰 서버 구현(Producer)

- GitHub 파일 조회 API 호출(주소 기반)
- 변경 감지 로직(이전 값 저장)
- 변경 발생 시 RabbitMQ로 메시지 발행

---

## 9.5 RabbitMQ 구성

- Exchange / Queue / Routing Key 개념(최소)
- 실습용 토폴로지 설계(Direct 권장)
- 메시지 포맷(JSON: repo, path, sha, content, updatedAt 등)
- application.properties 접속 키 정리(host/port/virtual-host/username/password)
- 바인딩 확인 항목(Exchange/Queue/Binding 일치 여부)

---

## 9.6 수신 서버 구현(Consumer)

- Queue 구독(Listener)
- 메시지 역직렬화
- 로컬 `README.md` 파일 수정/반영 로직

---

## 9.7 통합 테스트 시나리오

- GitHub README 수정
- Producer가 변경 감지(로그/상태 확인)
- RabbitMQ 발행 확인(Exchange/Queue 메시지 도착)
- Consumer가 README 자동 업데이트 확인
- 메시지 페이로드 확인(repo/path/sha/content/updatedAt)

---

## 9.8 실무 확장 포인트

- Polling 대신 Webhook으로 전환 시나리오(변경 이벤트 수신 → 즉시 발행)
- Webhook 수신을 위한 공인 IP/도메인 및 방화벽 오픈 필요성
