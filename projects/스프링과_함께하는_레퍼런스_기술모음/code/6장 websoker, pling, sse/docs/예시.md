## **6.1 준비**

### **Polling 개념과 특징**

Polling은 클라이언트가 일정 주기마다 서버에 “변경됐어?”를 물어보는 방식입니다. 구현이 단순하고 HTTP 기반이라 어디서든 동작하지만, 변화가 없어도 계속 요청이 발생해 트래픽이 늘고 “최대 지연 = 폴링 주기”라는 한계가 생깁니다.

### **SSE(Server-Sent Events) 개념과 특징**

SSE는 클라이언트가 서버에 한 번 연결을 열어두면, 서버가 이벤트를 계속 밀어주는 “단방향 스트림”입니다. 브라우저 기본 API(EventSource)가 있어 프론트가 가볍고, 텍스트 기반 이벤트 전송에 강합니다. 단, 클라이언트→서버 방향은 별도 HTTP 요청이 필요하며(단방향), 서버/프록시 환경에 따라 버퍼링 설정이 필요할 수 있습니다.

### **WebSocket 개념과 특징**

WebSocket은 한 번 연결을 맺으면 양방향으로 메시지를 주고받는 방식입니다. 채팅, 협업 편집, 실시간 제어처럼 상호작용이 많을수록 유리합니다. 대신 연결 상태(세션), 재연결, 인증/권한, 메시지 라우팅 등을 앱 차원에서 관리해야 합니다. Spring에서는 STOMP를 얹어 pub/sub 구조를 쉽게 만듭니다.

### **한눈에 보는 차이 표**

| **구분** | **Polling** | **SSE** | **WebSocket(STOMP)** |
| --- | --- | --- | --- |
| 통신 방향 | 요청/응답 반복 | 서버 → 클라(단방향) | 양방향 |
| 연결 | 매 요청마다 | 1회 연결 유지 | 1회 연결 유지 |
| 지연 | 폴링 주기 영향 | 거의 실시간 | 실시간 |
| 구현 난이도 | 낮음 | 중간 | 높음 |
| 적합한 케이스 | 완료 여부 확인, 간단 상태 | 알림/진행률/스트림 이벤트 | 채팅/협업/양방향 제어 |

### **6.2.1 Polling 서버 구현**

### **주기적 조회 API**

가장 단순한 형태는 “현재 상태”를 주는 API입니다.

```java
@RestController
@RequiredArgsConstructor
public class UploadStatusController {

    private final UploadStatusService uploadStatusService;

    @GetMapping("/api/status/{uuid}")
    public UploadStatusResponse getStatus(@PathVariable String uuid) {
        return uploadStatusService.get(uuid);
    }
}
```

### **변경 데이터 조회 로직**

“변경된 것만” 받고 싶다면 since(마지막 체크 시각/버전)를 받고, 변경이 없으면 same을 주는 방식이 실전에서 자주 씁니다.

```java
@Service
public class UploadStatusService {

    private final Map<String, UploadStatusResponse> store = new ConcurrentHashMap<>();

    public UploadStatusResponse get(String uuid) {
        return store.getOrDefault(uuid, UploadStatusResponse.notFound(uuid));
    }

    public void markResizedDone(String uuid, String resizedUrl) {
        store.put(uuid, UploadStatusResponse.done(uuid, resizedUrl, System.currentTimeMillis()));
    }
}
```

포인트는 “리사이즈 완료 여부” 같은 상태를 서버가 들고 있고, 클라이언트는 /api/status/{uuid}를 주기적으로 조회한다는 큰그림입니다.

<script>
  const uuid = "YOUR_UUID";
  const timer = setInterval(async () => {

    const data = await res.json();

    // 1) 화면 반영
    document.querySelector("#status").textContent = data.status;

## **6.3 SSE 구현 (Server-Sent Events)**

### **6.3.1 SSE 서버 구현**

### **SseEmitter**

### **기반 연결**

```
@RestController
@RequiredArgsConstructor
public class SseController {

    private final SseService sseService;

    @GetMapping("/api/sse/subscribe/{uuid}")
    public SseEmitter subscribe(@PathVariable String uuid) {
        return sseService.subscribe(uuid);
    }
}
```

```
@Service
public class SseService {

    private final Map<String, SseEmitter> emitters = new ConcurrentHashMap<>();

    public SseEmitter subscribe(String uuid) {
        SseEmitter emitter = new SseEmitter(60_000L); // 60초(예시)
        emitters.put(uuid, emitter);

        emitter.onCompletion(() -> emitters.remove(uuid));
        emitter.onTimeout(() -> emitters.remove(uuid));
        emitter.onError(e -> emitters.remove(uuid));

        // 연결 직후 현재 상태 한 번 보내기(옵션)
        send(uuid, "CONNECTED", "ok");
        return emitter;
    }

    public void send(String uuid, String eventName, Object data) {
        SseEmitter emitter = emitters.get(uuid);
        if (emitter == null) return;
        try {
            emitter.send(SseEmitter.event().name(eventName).data(data));
        } catch (Exception e) {
            emitters.remove(uuid);
        }
    }
}
```

### **이벤트 전송**

리사이즈 완료 시점(예: complete 처리 직후, 또는 별도 상태 업데이트 시점)에 sseService.send(uuid, "RESIZED_DONE", payload)를 호출하면 됩니다.

### **6.3.2 Front 구현 (Vanilla JS)**

### **EventSource**

### **연결**

```
<script>
  const uuid = "YOUR_UUID";
  const es = new EventSource(`/api/sse/subscribe/${uuid}`);

  es.addEventListener("RESIZED_DONE", (e) => {
    const data = JSON.parse(e.data);
    document.querySelector("#img").src = data.resizedUrl;
    document.querySelector("#status").textContent = "DONE";
    es.close();
  });

  es.addEventListener("CONNECTED", () => {
    document.querySelector("#status").textContent = "CONNECTED";
  });

  es.onerror = () => {
    document.querySelector("#status").textContent = "RECONNECTING";
    // EventSource는 기본적으로 재연결을 시도합니다.
  };
</script>
```

### **서버 이벤트 수신**

프론트는 폴링처럼 “주기”를 관리하지 않고, 이벤트가 오면 화면을 갱신하는 구조가 됩니다.

### **6.3.3 SSE 특징 정리**

- 단방향 스트림: 서버→클라이언트로만 이벤트가 흐릅니다(클라→서버는 별도 HTTP).
- 재연결 특성: 브라우저 EventSource는 기본 재연결을 지원합니다(서버는 타임아웃/정리 처리 필요).

## **6.4 WebSocket 구현 (STOMP)**

### **6.4.1 WebSocket 서버 구성**

### **WebSocket 설정**

```
@Configuration
@EnableWebSocketMessageBroker
public class WsConfig implements WebSocketMessageBrokerConfigurer {

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws")   // 연결 엔드포인트
                .setAllowedOriginPatterns("*");
        // 실전에서는 Origin 제한 권장
    }

    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        // 1) Simple Broker: 구독 대상(서버가 중계)
        registry.enableSimpleBroker("/topic", "/queue");

        // 2) Application Prefix: 클라이언트 SEND가 컨트롤러로 들어오는 prefix
        registry.setApplicationDestinationPrefixes("/app");
    }
}
```

### **“Config에서 바로 브로커로 가는 경우” vs “컨트롤러를 타는 경우”**

STOMP 라우팅은 목적지(prefix)로 결정됩니다.

- **바로 Simple Broker로 라우팅되는 케이스**: destination이 /topic/** 또는 /queue/**
    
    클라이언트가 해당 destination으로 SEND하면(허용 시) 컨트롤러를 거치지 않고 브로커가 구독자에게 바로 전달합니다.
    
- **컨트롤러(@MessageMapping)로 들어오는 케이스**: destination이 /app/**
    
    클라이언트가 /app/**로 SEND하면 스프링 메시지 컨트롤러가 먼저 처리하고, 이후 서버가 /topic/** 등으로 publish 합니다.
    

아래는 이 차이를 “한 장”으로 설명할 때 쓰기 좋은 흐름입니다.

```
Client SEND /topic/chat       Broker route directly to subscribers
Client SEND /app/chat.send    Controller handles then server publish to /topic/chat
```

### **STOMP 컨트롤러 예시(특정 작업 후 브로커로 응답)**

```
@Controller
@RequiredArgsConstructor
public class ChatController {

    private final SimpMessagingTemplate messagingTemplate;

    @MessageMapping("/chat.send") // 클라 SEND /app/chat.send
    public void send(ChatMessage msg) {
        // 1) 특정 작업(검증, 저장, 가공 등)
        String processed = msg.message().trim();

        // 2) 브로커로 publish (구독자에게 전달)
        messagingTemplate.convertAndSend("/topic/chat", new ChatMessage(msg.roomId(), processed));
    }
}
```

### **6.4.2 Front 구현 (Vanilla JS)**

브라우저에서 STOMP를 쓰려면 보통 stompjs를 CDN으로 붙입니다(실습용).

```
<script src="https://cdn.jsdelivr.net/npm/@stomp/stompjs@7.0.0/bundles/stomp.umd.min.js"></script>

<script>
  const client = new StompJs.Client({
    brokerURL: "ws://localhost:8080/ws",
    reconnectDelay: 1000
  });

  client.onConnect = () => {
    // 1) 구독
    client.subscribe("/topic/chat", (frame) => {
      const msg = JSON.parse(frame.body);
      console.log("RECV:", msg);
    });

    // 2) 컨트롤러를 타는 send (특정 작업 후 브로커로 전송)
    client.publish({
      destination: "/app/chat.send",
      body: JSON.stringify({ roomId: "r1", message: "hello" })
    });

    // 3) 브로커 직행 send (설정/정책에 따라 허용 여부 결정)
    // client.publish({ destination: "/topic/chat", body: JSON.stringify(...) });
  };

  client.activate();
</script>
```

### **6.4.3 WebSocket 특징 정리**

- 양방향 통신: 클라이언트/서버가 자유롭게 메시지를 주고받습니다.
- 상태 관리 이슈: 연결 끊김, 재연결, 중복 구독, 인증/권한, 메시지 순서/중복 처리 같은 운영 이슈를 고려해야 합니다.

---

### **선택 가이드(실전 결론)**

- “리사이즈 완료됐나?” 같은 단순 완료 확인은 Polling이 가장 빠릅니다.
- “서버가 알려주면 바로 바뀌는 UI”는 SSE가 구현 대비 효율이 좋습니다.
- “사용자가 보내고 서버가 처리 결과를 되돌려주는 상호작용”이 많으면 WebSocket(STOMP)로 갑니다.
