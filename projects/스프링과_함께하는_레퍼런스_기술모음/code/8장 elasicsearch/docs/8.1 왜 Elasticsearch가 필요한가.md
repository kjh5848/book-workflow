## **1) RDB LIKE 검색의 한계**

---

### **0. 예시 데이터(가정)**

```text
id=1, title=갤럭시 S23, content=삼성 최신 모델
id=2, title=아이폰 15, content=애플 최신 모델
id=3, title=갤럭시 워치, content=웨어러블 제품
```

초보자 관점에서 보면 RDB는 “한 줄 문자열”을 통째로 비교하는 방식이고, Elasticsearch는 “단어 단위로 미리 색인해두는 방식”이라고 생각하면 이해가 쉽습니다.

---

### **1. 선행 와일드카드로 인한 인덱스 무력화**

```sql
SELECT *
FROM device_entity
WHERE content LIKE '%갤럭시%';
```

`LIKE %keyword%`는 문자열 앞에 와일드카드가 붙기 때문에 B-Tree 인덱스를 활용하기 어렵고, 데이터가 커질수록 전체 스캔 비용이 급격히 증가합니다.

같은 LIKE라도 아래처럼 “앞부분이 고정”되면 인덱스를 탈 수 있습니다.

```sql
-- 앞이 고정되면 인덱스를 사용할 수 있는 경우가 많음
SELECT *
FROM device_entity
WHERE content LIKE '갤럭시%';
```

하지만 실제 검색은 “문장 중간에 있는 단어”를 찾는 경우가 많아 `%갤럭시%`를 쓰게 되고, 결국 전체 스캔으로 이어집니다.

---

### **2. 단어 단위 검색/오타 허용의 부재**

```sql
-- "갤럭시"를 "갈럭시"로 입력하면 매칭 실패
SELECT *
FROM device_entity
WHERE content LIKE '%갈럭시%';
```

RDB의 LIKE는 토큰화, 동의어, 오타 허용 같은 검색 품질 기능이 없어서 사용자 입력 실수를 반영하기 어렵습니다.

예를 들어 사용자가 “갤럭시 워치”를 찾고 싶은데 `갤럭시`만 입력했을 때,
LIKE는 결국 전체 문자열 비교라 “부분 단어 검색 품질”이 떨어집니다.

## **2) Full-text Search와 분석기 기반 검색**

---

**(확인) 경로: src/main/java/com/metacoding/spring_elasticsearch/ElasticSearch/ElasticSearchService.java**

### **1. 분석기 기반 match/multi_match 검색**

```java
public List<DeviceEntity> searchAll(String keyword) {
    NativeQuery query = NativeQuery.builder()
            .withQuery(q -> q.bool(b -> b
                    .should(s -> s.multiMatch(m -> m
                            .fields("title^3", "content")
                            .query(keyword)
                            .fuzziness("AUTO")))
                    .minimumShouldMatch("1")))
            .build();
    ...
}
```

Elasticsearch는 입력 키워드를 분석기(Analyzer)로 토큰화한 뒤 역색인(Inverted Index)에서 빠르게 검색합니다. `multi_match`는 여러 필드를 동시에 검색하고, `title^3`로 제목에 가중치를 줘 품질을 높입니다.

예를 들어 `title = "갤럭시 S23 울트라"`라면 분석 결과가 대략 아래처럼 나뉩니다.

```text
토큰: ["갤럭시", "S23", "울트라"]
```

이 상태에서 사용자가 `갤럭시`만 입력해도 해당 문서를 빠르게 찾을 수 있습니다.

---

### **2. 오타 허용(Fuzzy)까지 포함한 검색 품질**

```java
public List<DeviceEntity> searchAll(String keyword) {
    ...
    .fuzziness("AUTO");
    ...
}
```

`AUTO`는 단어 길이에 따라 허용 가능한 편집 거리(Edit Distance)를 자동으로 조절합니다. 예: "갤럭시" ↔ "갈럭시" 같은 오타도 검색 결과에 포함됩니다.

간단 예시:

```text
사용자 입력: 갈럭시
실제 데이터: 갤럭시
```

LIKE로는 0건이지만, Fuzzy 검색은 “비슷한 단어”로 판단해 결과를 돌려줍니다.
