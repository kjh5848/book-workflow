# 시나리오 — 스프링과 함께하는 레퍼런스 기술 모음

## 시간축 설정

주니어 개발자 **나** 가 스타트업에 백엔드 개발자로 합류한 뒤, 실무에서 하나씩 새로운 기술을 마주치며 성장하는 **3개월간의 기록** 입니다.

```
1주차    Docker           — 입사 첫 주, 개발 환경 세팅
2주차    OAuth2.0/OIDC    — 소셜 로그인 기능 배정
3주차    Redis 세션 공유   — 서버 2대로 늘리면서 세션 문제
4주차    Base64 업로드     — 프로필 이미지 기능 추가
5주차    Presigned URL    — 이미지가 늘어나며 클라우드 전환
7주차    Polling/SSE/WS   — 실시간 알림 요구사항 등장
8주차    스트리밍          — 영상 콘텐츠 서비스 확장
10주차   Elasticsearch    — 검색 기능 고도화 요청
12주차   RabbitMQ         — 외부 시스템 연동 자동화
```

시간 간격은 균일하지 않습니다. 초반에는 매주 새 과제가 쏟아지고, 후반으로 갈수록 한 기술에 머무는 시간이 길어집니다. 주니어 개발자의 실제 성장 곡선과 같습니다.

---

## 독립 읽기 가이드

이 책은 **독립형** 입니다. 관심 있는 장부터 골라 읽을 수 있습니다. 다만 다음 의존성을 참고하세요.

- **1장 Docker는 선행 권장**. 3장(Redis), 8장(Elasticsearch), 9장(RabbitMQ)에서 docker-compose를 사용합니다. Docker를 이미 아는 분은 건너뛰어도 됩니다.
- **4장 -> 5장은 순서대로**. 로컬 업로드(Base64) -> 클라우드 업로드(S3)로 확장되는 흐름입니다.
- **6장 -> 7장은 순서대로**. 실시간 통신 비교 -> 대규모 스트리밍으로 깊이가 더해집니다.
- **2장, 8장, 9장은 완전 독립**. 어느 시점에 읽어도 됩니다.

처음부터 순서대로 읽으면, 한 회사에서 성장하는 주니어 개발자의 이야기를 따라가게 됩니다.

---

## 학습 순서

현재 1~9장 순서를 **유지** 합니다. 근거는 다음과 같습니다.

1. **1장 Docker가 선행 전제**. 3장(Redis), 8장(Elasticsearch), 9장(RabbitMQ)이 docker-compose를 사용하므로 반드시 첫 번째입니다.
2. **2장 OAuth는 독립이지만 초반 배치가 적절**. 입사 직후 로그인 기능은 가장 먼저 접하는 업무입니다. 난이도가 중상이지만, 초반에 "어려운 것도 해냈다"는 자신감을 줍니다.
3. **3장 Redis는 Docker 직후**. docker-compose를 막 배운 시점에 바로 활용하면 기억이 살아 있습니다.
4. **4장 -> 5장 의존성 유지**. 로컬 업로드(Base64) -> 클라우드 업로드(S3)로 자연스럽게 확장합니다.
5. **6장 -> 7장 의존성 유지**. 실시간 통신 비교 -> 대규모 스트리밍으로 깊이를 더합니다.
6. **8장, 9장은 후반부**. Docker 경험이 충분히 쌓인 시점에서 인프라 의존도가 높은 기술을 다룹니다.

**난이도 곡선**.

```
장   1    2    3    4    5    6    7    8    9
     하   중상  중   하   중상  중   중상  중   중
      |    |    |    |    |    |    |    |    |
      *         *         *         *    *
           *         *         *              *
                          *         *
```

패턴은 **파도형** 입니다. 어려운 장 다음에 쉬운 장이 오면서 독자에게 숨을 돌릴 틈을 줍니다.

## 장별 시나리오

### 1장 Docker로 시작하는 개발 환경 — 1주차, 입사 첫 날

- **상황**: 첫 출근. **팀장** 이 "로컬에 직접 설치하지 말고, Docker로 환경 맞추세요"라고 말합니다. Docker라는 단어는 들어봤지만, 실제로 써본 적은 없습니다.
- **문제**: Java와 MySQL을 로컬에 설치하려다가 버전 충돌이 납니다. 옆자리 **동료** 는 "그래서 Docker 쓰라니까"라며 웃습니다.
- **해결**: Dockerfile을 작성하고, 여러 컨테이너를 한 번에 관리하는 docker-compose로 Spring Boot + MySQL을 띄웁니다. `docker-compose up` 한 줄로 개발 환경이 완성됩니다.
- **얻는 것**: "내 컴퓨터에서는 되는데"가 사라집니다. 환경을 코드로 관리한다는 감각을 얻습니다.

### 2장 OAuth 2.0 & OIDC (Kakao) — 2주차, 첫 번째 기능 배정

- **상황**: **팀장** 이 "카카오 소셜 로그인 붙여주세요. 급하지 않으니 천천히 해도 괜찮아요"라고 합니다. 로그인이야 세션/쿠키로 해봤지만, "OAuth"라는 단어 앞에서 멈칫합니다.
- **문제**: 인가 코드, 액세스 토큰, ID 토큰, JWKS... 용어가 쏟아집니다. 흐름도를 봐도 화살표가 네 곳을 오갑니다. (이게 대체 몇 단계야?)
- **해결**: 인가 코드 발급 -> 액세스 토큰 교환 -> 사용자 정보 조회를 한 단계씩 따라갑니다. OIDC로 확장하면서 ID Token을 직접 검증합니다. JWT 인증 필터까지 완성합니다.
- **얻는 것**: OAuth 흐름도의 화살표가 읽히기 시작합니다. "소셜 로그인은 결국 토큰 교환"이라는 핵심을 잡습니다.

### 3장 Redis 세션 공유 — 3주차, 서버가 2대가 되던 날

- **상황**: 배포 후 모니터링을 보던 **나** 는 응답 시간이 점점 느려지는 걸 발견합니다. **팀장** 에게 보고하자 "서버 한 대 더 띄우죠"라고 합니다. (서버 늘리면 좋은 거 아닌가?)
- **문제**: 서버를 2대로 늘렸더니 로그인이 풀립니다. A 서버에서 로그인했는데 B 서버로 요청이 가면 "누구세요?" 상태입니다. 세션이 서버 메모리에 갇혀 있기 때문입니다.
- **해결**: Redis에 세션을 저장하고, Nginx로 로드밸런싱합니다. 어떤 서버로 요청이 가든 같은 세션을 공유합니다. docker-compose로 Redis + Nginx + Spring Boot 2대를 한 번에 구성합니다.
- **얻는 것**: "서버를 늘리면 상태 관리가 문제다"를 체감합니다. 외부 저장소로 상태를 분리하는 패턴을 익힙니다.

### 4장 Base64 이미지 업로드 — 4주차, 프로필 사진을 넣어달라는 요청

- **상황**: 기획팀에서 "사용자 프로필에 사진 넣어주세요"라고 합니다. 파일 업로드는 해본 적 없습니다. (이미지를 서버에 어떻게 보내지?)
- **문제**: 이미지를 서버로 보내는 방법이 여러 가지입니다. MultipartFile? Base64? 프론트엔드에서는 어떻게 보내야 하는지도 모릅니다.
- **해결**: Base64로 인코딩해서 JSON에 실어 보냅니다. 서버에서 디코딩 후 UUID 파일명으로 저장합니다. React에서 이미지 미리보기까지 연동합니다.
- **얻는 것**: 파일 업로드의 가장 단순한 형태를 익힙니다. "이미지도 결국 문자열"이라는 감각을 잡습니다.

### 5장 Presigned URL + S3 + Lambda — 5주차, 이미지가 서버 디스크를 채우다

- **상황**: 어느 날 서버 모니터링 알림이 울립니다. 디스크 사용량 90%. 프로필 사진이 쌓인 탓입니다. (4장에서 로컬에 저장한 게 문제였구나.)
- **문제**: 이미지를 클라우드에 올리려면 S3 버킷, IAM 권한, 업로드 방식을 결정해야 합니다. 서버를 거치면 트래픽이 두 배입니다.
- **해결**: Presigned URL로 클라이언트가 S3에 직접 업로드합니다. Lambda가 업로드된 이미지를 자동으로 리사이즈합니다. 서버는 URL만 발급하면 됩니다.
- **얻는 것**: "서버가 모든 걸 처리할 필요는 없다"를 배웁니다. 클라우드 서비스에 역할을 위임하는 감각을 얻습니다.
- **참고**: 이 장은 AWS 계정이 필요합니다. 다른 장과 달리 로컬만으로는 완결되지 않지만, 실무에서 반드시 마주치는 패턴이므로 포함했습니다.

### 6장 Polling / SSE / WebSocket 실전 비교 — 7주차, 실시간 알림이 필요해지다

- **상황**: 기획팀이 "새 댓글이 달리면 바로 알림이 떠야 해요"라고 합니다. (실시간? 새로고침 말고 진짜 실시간?)
- **문제**: 실시간 통신 방법이 세 가지나 있습니다. Polling, SSE, WebSocket. 각각 언제 써야 하는지 모릅니다. 셋 다 만들어보지 않으면 차이를 알 수 없습니다.
- **해결**: 같은 알림 기능을 Polling(2초 간격 Ajax), SSE(SseEmitter), WebSocket(STOMP)로 각각 구현합니다. 직접 비교하면서 장단점을 체감합니다.
- **얻는 것**: "실시간"이라는 단어 앞에서 세 가지 선택지를 떠올릴 수 있습니다. 요구사항에 따라 적절한 방식을 고를 수 있습니다.

### 7장 대규모 스트리밍 실전과 기술 선택 — 8주차, 영상 서비스로의 확장

- **상황**: 회사가 영상 콘텐츠 서비스를 준비합니다. **동료** 가 "6장에서 WebSocket 했으니까 영상도 비슷하지 않아?"라고 묻습니다. 아닙니다. 영상 스트리밍은 차원이 다릅니다.
- **문제**: 1GB짜리 영상을 그대로 전송하면 로딩에 몇 분이 걸립니다. 화질별 선택, 구간 이동, 대역폭 적응이 필요합니다. (유튜브는 어떻게 하는 거지?)
- **해결**: FFmpeg로 영상을 HLS 포맷(m3u8 + ts 조각)으로 변환합니다. 720p와 1080p를 모두 준비합니다. Spring Boot에서 정적 리소스로 매핑하여 스트리밍합니다.
- **얻는 것**: 영상 스트리밍의 핵심 원리("잘게 쪼개서 순서대로 보낸다")를 이해합니다. 기술 선택 가이드를 통해 실무에서의 판단 기준을 얻습니다.

### 8장 Elasticsearch 키워드 검색 — 10주차, "검색이 너무 느려요"

- **상황**: QA 팀에서 버그 리포트가 올라옵니다. "검색 결과가 5초 넘게 걸립니다." 게시글이 수만 건을 넘기면서 `LIKE '%키워드%'` 쿼리가 한계에 도달했습니다.
- **문제**: Elasticsearch는 RDB와 완전히 다른 세계입니다. 인덱스, 매핑, 분석기... 용어부터 낯섭니다. 한글 검색은 별도 분석기(nori)가 필요합니다.
- **해결**: Docker로 Elasticsearch + Kibana를 띄웁니다. RDB와 ES에 이중 저장하고, multi_match 가중치 검색과 Fuzzy 검색을 구현합니다. nori 분석기로 한글 형태소 분석까지 적용합니다.
- **얻는 것**: "검색 엔진은 왜 따로 있는가"를 체감합니다. 역인덱스의 원리와 한글 검색의 특수성을 이해합니다.

### 9장 RabbitMQ 메시지 큐 — 12주차, 시스템 간 연동 자동화

- **상황**: 회사의 문서 관리 시스템이 GitHub과 연동되어야 합니다. **나** 는 처음에 REST API로 직접 호출하면 되겠다고 생각합니다. (API를 직접 호출하면 되지 않나?)
- **문제**: 직접 호출은 결합도가 높습니다. GitHub이 느려지면 우리 시스템도 느려집니다. 실패하면 데이터가 유실됩니다. 시스템 간에 완충 장치가 필요합니다.
- **해결**: RabbitMQ를 중간에 두고, Producer가 GitHub 변경을 감지하여 메시지를 발행합니다. Consumer가 메시지를 구독하여 파일을 반영합니다. Docker로 RabbitMQ를 구성합니다.
- **얻는 것**: "직접 호출"과 "메시지 큐를 통한 간접 호출"의 차이를 체감합니다. 발행/구독 패턴으로 시스템을 느슨하게 연결하는 방법을 익힙니다.

## 프롤로그 서사 소재

입사 첫 날, 터미널 앞에서 멈췄습니다. **팀장** 이 건넨 온보딩 문서의 첫 줄은 `docker-compose up -d`였습니다. Docker가 뭔지도 모르는데, 명령어부터 치라니. 손가락이 키보드 위에서 얼어붙었습니다.

3개월 뒤, 저는 같은 터미널 앞에 앉아 있습니다. 다만 화면에는 RabbitMQ 관리 콘솔이 열려 있고, Producer가 보낸 메시지가 Consumer로 흘러가는 걸 지켜보고 있습니다. 손이 얼어붙지 않습니다.

이 책은 그 사이에 있었던 일들의 기록입니다. Docker부터 메시지 큐까지, "들어는 봤지만 한 번도 직접 해본 적 없는" 기술들을 하나씩 만져본 3개월의 이야기입니다.
