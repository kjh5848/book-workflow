# 시나리오 + 버전 설계

## 시나리오

### 한 줄 요약
신입 개발자가 팀장에게 "사내 AI 비서 만들어봐"라는 미션을 받고, 환각 체험부터 시작해 RAG 기반 AI 비서를 완성해가는 여정.

### 전체 흐름
1. **시작**: 팀장이 "우리 회사 규정 물어보면 답해주는 AI 만들어봐"라고 미션을 던진다.
2. **첫 시도 (실패)**: ChatGPT처럼 LLM에게 바로 물어본다 → 환각. 엉터리 답변이 나온다.
3. **임시 해결**: "그러면 정보를 직접 넣어주면 되지 않나?" → Context Injection. 되긴 하는데...
4. **한계 직면**: "문서가 1000개면 다 넣을 수 없잖아?" → 확장 불가능한 방법이었다.
5. **RAG 첫 도전 (부분 실패)**: 벡터 DB에 문서를 넣고 검색하는 방식을 시도 → 근데 엉뚱한 규정까지 같이 나온다.
6. **RAG 성공**: 문서를 잘게 쪼개고(청킹) 검색하니 관련 규정만 깔끔하게 나온다!
7. **심화**: "검색은 되는데 계산은?" → 추론 능력을 결합해서 규정 기반 계산까지 가능하게.

### 전환점
**step1 → step2**: LLM에게 "우리 회사 연차 규정 알려줘"라고 물었는데 완전히 틀린 답변이 나온 순간.
"아... 이 녀석은 우리 회사를 모르는구나. 그러면 알려주면 되지 않나?"라는 깨달음.
이 순간이 전체 여정의 출발점이자, 독자가 "왜 RAG가 필요한지"를 몸으로 느끼는 핵심 지점.

### 최종 결과물
사내 규정(인사/보안/복지)을 검색해서 정확하게 답변하고, 계산이 필요한 질문에도 추론으로 답하는 AI 비서.

## 버전 설계

| 버전 | 뭘 만드나 | 시나리오 속 위치 | 끝나면 뭐가 보이나 | 핵심 변경점 |
|------|----------|----------------|------------------|------------|
| v0.1 | LLM 단독 질의 | 첫 시도 (실패) | 터미널에서 질문 → 환각 답변 출력 | ChatOllama 연결, invoke() |
| v0.2 | Context Injection | 임시 해결 | 같은 질문 → 정확한 답변 (프롬프트에 정보 삽입) | f-string 프롬프트 구성 |
| v0.3 | RAG 청킹 미적용 | RAG 첫 도전 (부분 실패) | RAG인데 불필요한 규정까지 다 나옴 | ChromaDB, OllamaEmbeddings, 통째 Document |
| v0.4 | RAG 청킹 적용 | RAG 성공 | 관련 규정만 검색되어 깔끔한 답변 | 개별 Document, RetrievalQA |
| v0.5 | RAG + 추론 | 심화 | "리프레시 데이 몇 번 남았어?" → 계산까지 답변 | 복잡한 질문, 추론 결합 |

### 난이도 곡선

```
난이도
  ▲
  │            ┌── v0.3 (벡터DB/임베딩 등장)
  │        ┌───┘
  │    ┌───┘       v0.4 (v0.3과 구조 동일, 청킹만 변경)
  │────┘                  v0.5 (v0.4 + 복잡한 질문)
  │ v0.1  v0.2
  └──────────────────────────▶ 진행

⚠️ v0.2→v0.3 구간에서 난이도 점프 있음
   → 이야기 파트에서 벡터DB/임베딩을 비유로 충분히 풀어서 해결
```

## 전환점 상세

| 전환 | 독자가 느끼는 것 | 이야기 속 장치 |
|------|----------------|--------------|
| v0.1→v0.2 | "아 LLM은 우리 회사를 모르는구나" | 환각 답변을 보고 충격 |
| v0.2→v0.3 | "이걸 일일이 다 넣을 순 없잖아" | 문서 10개만 돼도 프롬프트가 넘침 |
| v0.3→v0.4 | "왜 보안규정까지 같이 나오지?" | 청킹 안 한 RAG의 비효율 체감 |
| v0.4→v0.5 | "검색은 되는데 계산 질문은?" | 단순 검색의 한계 |
