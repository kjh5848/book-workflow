# 시나리오 + 버전 설계

## 시나리오

### 한 줄 요약
신입 개발자가 사수 없이 혼자 사내 AI 비서를 만들어가는 여정

### 전체 흐름

1. **시작**: 입사 첫 주, 팀장이 "AI로 사내 문서 검색 시스템 만들어봐"라고 한다. ChatGPT에 물어봤더니 엉뚱한 답이 나온다. "사내 문서를 직접 넣어줘야 하는구나" 깨닫는다.
2. **기반 구축**: 사내 시스템(직원/연차/매출 DB)을 API로 만들고, 사내 문서를 정리하고, 벡터DB를 구축한다.
3. **핵심 기능**: RAG 파이프라인을 만들고, 정형(DB)+비정형(문서) 데이터를 통합하는 에이전트를 설계한다.
4. **운영+최적화**: 에이전트를 안정화하고, 검색 품질을 튜닝하고, 문서 이미지까지 처리하고, 평가 프레임워크로 품질을 측정한다.
5. **완성**: 동료가 "김대리 연차 몇 개고, 사용 규정은?"이라고 물으면 DB에서 연차를 조회하고 문서에서 규정을 찾아 한 번에 답해주는 AI 비서가 완성된다.

### 전환점 (3개)

| # | 전환점 | 위치 | 독자가 느끼는 것 |
|---|-------|------|------------|
| 1 | **LLM 환각 → "문서를 넣어야 하는구나"** | CH03 | LLM이 그럴듯하게 거짓말하는 걸 보고, RAG의 필요성을 체감 |
| 2 | **RAG만으로는 부족 → "정형+비정형 통합이 필요하구나"** | CH08 | "연차 몇 개?"는 DB 조회, "휴가 규정?"은 문서 검색 — 둘을 합쳐야 진짜 비서 |
| 3 | **기본 RAG → "튜닝하니까 이렇게 달라지는구나"** | CH10~12 | 엉뚱한 문서를 가져오던 게 리랭킹/하이브리드로 정확해지는 극적 변화 |

### 최종 결과물
사내 문서 기반 AI 업무 비서 — 정형(DB) + 비정형(문서) 통합 질의응답, RAG 튜닝 완료, 평가 프레임워크 포함

## 전환점 상세

### 전환점 1: LLM 환각 → RAG (CH03)
- **상황**: 팀장이 "AI한테 우리 회사 휴가 규정 물어봐" → LLM이 그럴듯하게 거짓말
- **깨달음**: "LLM은 우리 회사 문서를 모른다. 직접 넣어줘야 한다."
- **해결**: 문서를 직접 프롬프트에 넣어보고(Context Injection) → 길어지니까 벡터 검색(RAG)으로 자동화
- **독자 감정**: "아, 그래서 RAG가 필요한 거구나"

### 전환점 2: RAG만으로는 부족 → 통합 에이전트 (CH08)
- **상황**: "김대리 연차 몇 개고, 휴가 규정은?" → RAG는 문서만 검색하고, DB 조회는 못 함
- **깨달음**: "정형 데이터(DB)와 비정형 데이터(문서)를 합쳐야 진짜 비서"
- **해결**: QueryRouter로 질문 분류 → MCP 도구로 DB 조회 → RAG로 문서 검색 → 통합 응답
- **독자 감정**: "아, 에이전트가 이런 역할을 하는 거구나"

### 전환점 3: 기본 RAG → 튜닝 (CH10~12)
- **상황**: "사내 보안 정책 알려줘" → 엉뚱한 문서를 가져옴, 정확도가 떨어짐
- **깨달음**: "기본 RAG로는 한계가 있다. 튜닝이 필요하다."
- **해결**: 리랭킹, 하이브리드 검색, Query Rewrite 등 단계적 개선
- **독자 감정**: "같은 질문인데 튜닝 전후가 이렇게 다르다니"
