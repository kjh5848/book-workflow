# 들어가며

**팀장**: "AI로 사내 문서 검색 시스템 만들어봐."

입사 첫 주, 사수도 없었습니다.

 **1일차.** - 일단 ChatGPT에 물어봤습니다. "우리 회사 연차 규정이 어떻게 되나요?" AI는 자신 있게 답했습니다. 연차는 15일이고 3일 전까지 신청하면 된다고요. 우리 회사 규정을 열어보니 연차는 입사일 기준으로 달랐고 신청 기한도 일주일 전이었습니다. AI가 그럴듯하게 거짓말을 한 겁니다. 그날 하나를 깨달았습니다. 

**AI는 우리 회사 문서를 읽은 적이 없습니다.** 그래서 직접 넣어주기로 했습니다. 문서를 통째로 프롬프트에 복사해봤습니다. 작은 문서 하나는 됐습니다. 규정집 200페이지를 복사하니 토큰 한도를 넘겼습니다. 그때 RAG라는 걸 알게 됐습니다. "오픈북 시험"이라고 생각하면 됩니다. AI가 사내 문서 전체를 외울 필요 없이 질문할 때마다 관련 문서만 찾아서 보여주면 됩니다.


**2일차.** 문서를 넣기 전에 AI 비서가 조회할 사내 시스템부터 만들어야 했습니다. "팀원 연차 몇 개 남았어?"는 문서가 아니라 데이터베이스에서 답이 나오는 질문이니까요. 직원 정보와 연차, 매출 테이블을 만들고 API를 열었습니다. 식당에 비유하면 웨이터를 고용한 셈입니다. 손님이 주문하면 주방에서 요리를 가져다주는 것처럼요.

**3일차.** 사내 시스템은 됐습니다. 이제 문서를 넣을 차례입니다. 공유 드라이브를 열었는데 한숨이 나왔습니다. PDF와 DOCX, XLSX, 심지어 HWP까지. 최신 문서와 3년 전 폐기된 문서가 한 폴더에 섞여 있었습니다. 

(이걸 전부 넣으면?) 쓰레기를 넣으면 쓰레기가 나옵니다. AI도 마찬가지입니다. 도서관을 떠올렸습니다. 사서가 새 책을 받으면 분류하고 라벨을 붙이고 서가에 꽂습니다. 같은 원리입니다. 넣을 문서를 골라내고 형식을 통일하고 메타데이터를 붙이는 것. "신입사원에게 줘도 되는 문서인가?"를 필터링 기준으로 삼았습니다.

**4일차.** 문서를 골랐으면 검색 가능한 지식으로 바꿔야 합니다. 요리에 비유하면 네 단계입니다. 재료를 손질하고(파싱) 먹기 좋게 다지고(청킹) 양념을 입히고(임베딩) 냉장고에 넣습니다(벡터DB). "연차"와 "유급 휴가"가 같은 뜻이라는 걸 숫자 배열로 바꿔야 기계가 알아듣습니다.

**5일차.** 서가가 완성됐습니다. 동료는 서가를 뒤지고 싶어 하지 않았습니다. "검색 결과 다섯 개를 던져주지 말고, 그냥 답을 알려줘." 사서가 책을 찾아와서 읽어서 정리해서 답해주는 게 필요했습니다. 검색은 재료이고 답변은 요리입니다. 레시피를 만들었습니다. 검색, 질문 조합, AI가 읽고 답변, 정리. 이 네 단계를 하나의 흐름으로 연결했습니다. 출처 없는 답변은 근거 없는 주장이니까요. "어떤 문서에서 이 답을 찾았는지" 반드시 보여주게 만들었습니다. 한 번 물어보고 끝이 아니라 대화를 이어갈 수 있는 기억력도 붙여줬습니다. 드디어 질문하면 답해주는 AI 비서가 된 겁니다.

**6일차.** 동료가 이런 질문을 했습니다. "A 사원 연차 몇 개? 그리고 연차 신청 절차는?" 한 문장에 두 가지가 섞여 있었습니다. "연차 몇 개"는 데이터베이스를 조회해야 하고 "신청 절차"는 문서를 검색해야 합니다. RAG만으로는 절반밖에 답할 수 없었습니다. AI 비서를 "안내데스크"로 바꿔야 했습니다. 질문이 들어오면 유형을 분류하고 맞는 담당자를 호출하는 구조입니다. DB를 조회하는 담당자, 문서를 검색하는 담당자, 둘 다 호출해서 합치는 담당자. 이렇게 해서 **ConnectHR** 이 태어났습니다. 정형 데이터와 비정형 문서를 한번에 다루는 사내 AI 비서입니다.

**7일차.** ConnectHR을 동료에게 풀어줬습니다. 처음에는 신기해했습니다. 같은 질문을 할 때마다 20초씩 기다려야 했습니다. "아까도 물어봤는데 또 기다려?" 사서에게 기억력을 줬습니다. 한 번 답한 내용은 메모해두고 같은 질문이 오면 메모를 보여줍니다. 업무 일지도 쥐어줬습니다. 하루에 토큰을 얼마나 쓰는지, 비용은 얼마인지, 느린 질문은 어떤 건지.

**8일차.** 운영은 안정됐습니다. 진짜 문제가 터졌습니다. "병가 규정을 물어봤는데 출장 규정이 나왔다." 검색이 엉뚱한 문서를 가져오고 있었습니다. LLM 탓이 아니었습니다. 사서가 서가에서 잘못된 책을 꺼내온 겁니다. 사서의 검색 실력을 키워야 했습니다. 가위로 일정하게 자르던 걸 의미 단위로 바꾸고 검색 결과를 심사위원에게 다시 줄 세우게 하고 키워드 검색과 의미 검색을 합쳤습니다. 같은 질문인데 답이 달라졌습니다.

**9일차.** 검색이 좋아지니 이번엔 질문이 문제였습니다. 동료가 "WFH 정책 알려줘"라고 물었는데 ConnectHR이 멈칫했습니다. 문서에는 "재택근무"라고 적혀 있었으니까요. "휴가 규정 알려줘"도 마찬가지였습니다. 문서에는 "연차유급휴가"라고 돼 있었습니다. 사서에게 언어 능력을 줬습니다. 약어를 풀어쓰고 동의어를 확장하고 한 질문을 여러 표현으로 바꿔서 각각 검색하게 했습니다. 답변에 근거도 붙였습니다. "이 문서의 3페이지에서 찾았습니다"라고 원본 이미지와 함께.

**10일차.** 마지막 관문이 남았습니다. 조직도 PDF를 넣었는데 텍스트만 나왔습니다. 박스와 화살표가 사라지고 "대표이사경영지원본부기술개발본부"가 한 줄로 붙었습니다. 사서에게 눈을 달아줘야 했습니다. 이미지를 보고 의미를 이해하는 능력을 준 겁니다. 성적표도 만들었습니다. "찾아온 문서 중 몇 개가 정답이었나", "정답 문서를 몇 개나 찾았나", "AI가 지어낸 답변은 없었나". 이 지표로 ConnectHR의 실력을 숫자로 측정했습니다. 튜닝 전과 후의 차이가 눈에 보였습니다. 어디를 더 개선해야 하는지도 보이기 시작했습니다.

## 한눈에 보기
| 챕터 | 핵심 질문 | 끝나면 뭐가 보이나 |
|------|----------|-----------------|
| CH01 | AI는 왜 거짓말을 할까? | 환각 vs RAG 응답 비교 |
| CH02 | AI가 조회할 DB는 어떻게 만들까? | CRUD API 동작 |
| CH03 | 어떤 문서를, 어떤 기준으로 넣을까? | 수집 규칙 확립 (개념) |
| CH04 | 문서를 어떻게 검색 가능한 형태로 바꿀까? | 벡터 검색 + 유사도 |
| CH05 | 검색 결과를 어떻게 답변으로 만들까? | 질문 → 답변 + 출처 |
| CH06 | DB 조회와 문서 검색을 한 번에 하려면? | **ConnectHR 탄생** |
| CH07 | 캐시, 비용 추적, 에러 처리는? | 운영 안정화 |
| CH08 | 왜 엉뚱한 문서를 가져올까? | 튜닝 전후 비교 |
| CH09 | 같은 뜻인데 왜 못 찾을까? | 쿼리 변환 + 근거 |
| CH10 | PDF 이미지는? 품질은 어떻게 측정할까? | **성적표 + v1.0 완성** |

세 가지 전환점이 이야기를 이끕니다.

- 1. **CH01** — LLM 환각을 보고 RAG가 필요하다는 걸 깨닫습니다.
- 2. **CH06** — RAG만으로는 부족하고 DB와 문서를 합쳐야 한다는 걸 깨닫습니다.
- 3. **CH08** — 기본 RAG로는 한계가 있고 튜닝이 필요하다는 걸 깨닫습니다.

모든 챕터는 **실패에서 시작** 합니다. 

에러를 만나고 원인을 찾고 해결하는 과정이 곧 학습입니다.

각 챕터는 "이야기 파트"와 "기술 파트"로 나뉩니다. 이야기 파트에서 왜 이 기술이 필요한지를 상황과 비유로 먼저 보여주고 기술 파트에서 코드로 구현합니다. 이야기 파트만 읽어도 전체 흐름을 이해할 수 있습니다.

## 이 책이 맞는 분

- Python 기초 문법은 아는 분 (함수, 클래스, 패키지 설치)
- LLM이나 RAG는 처음이거나 이론만 읽어본 분
- "조각난 예제 말고 하나의 프로젝트를 처음부터 끝까지 만들어보고 싶은" 분

## 권장 사양

이 책의 예제를 따라하려면 아래 환경이 필요합니다.

| 항목 | 권장 사양 | 비고 |
|------|----------|------|
| **Python** | 3.10 ~ 3.12 | 3.13 이상은 일부 호환성 문제 가능 |
| **RAM** | 16GB 이상 | 임베딩 모델 로딩 시 8GB 이상 사용 |
| **디스크** | 10GB 여유 | 모델 파일 + Docker 이미지 |
| **Docker** | Docker Desktop 설치 | PostgreSQL 실행에 필요 |
| **LLM** | Ollama + llama3.1:8b | 로컬에서 무료로 실행 |
| **Vision LLM** | Ollama + qwen2.5vl:7b | CH10 이미지 파싱에 필요 |

CH06 이후 에이전트 기능(Tool Calling)을 사용하려면 Ollama 기준 `llama3.1:8b` 이상 모델이 필요합니다. CH10에서 PDF 이미지를 분석하려면 비전 모델 `qwen2.5vl:7b` 이 필요합니다.

**로컬 환경이 어려운 경우** OpenAI API(gpt-4o-mini)를 대안으로 쓸 수 있습니다. 각 예제의 `.env` 파일에서 `LLM_PROVIDER=openai` 로 변경하고 API 키만 설정하면 동일하게 동작합니다. Ollama 설치나 GPU 없이도 모든 챕터를 진행할 수 있습니다.

자세한 환경 설정 방법은 **부록 A. 환경 설정** 을 참고하세요.
