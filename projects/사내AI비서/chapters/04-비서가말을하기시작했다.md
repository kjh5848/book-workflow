# Ch.4: 비서가 말을 하기 시작했다 — v0.4

> 이번 버전: v0.3 → v0.4
> 한 줄 요약: 검색 결과에 LLM을 연결했더니, 텍스트 조각이 아니라 자연어 답변이 나오기 시작했다.
> 핵심 개념: RAG 체인, 프롬프트 엔지니어링, LCEL, 멀티턴 대화

---

## 이야기 파트

### 자료는 찾았는데, 누가 읽어주지?

v0.3이 완성됐습니다. CLI에서 "연차 사용 규정"을 검색하면 관련 문단이 떠오릅니다. 유사도 85%짜리 색인 카드가 나오고, 출처도 표시됩니다.

팀장에게 보여줬습니다.

> "오, 검색은 되네. 근데 이게 뭐야? 이걸 내가 읽어야 해?"

화면에는 이렇게 떠 있었습니다.

> "연차는 입사 1년 후 15일이 부여되며, 3년 이상 근속 시 매 2년마다 1일이 가산된다. 연차 사용 시 최소 1일 전 소속 부서장의 승인을 받아야 하며..."

규정 원문 그대로입니다. 틀린 말은 아닙니다. 그런데 팀장이 원한 건 이게 아니었어요.

"내 연차 며칠 남았어?"라고 물으면 "7일 남았습니다"라고 답해주는 걸 원한 거죠. 규정 원문을 복사해서 던져주는 게 아니라.

이게 v0.3의 한계였습니다. 도서관에서 관련 책을 찾아오는 데는 성공했는데, 그 책을 읽고 요약해서 답해줄 사람이 없는 상태. 자료는 올라왔는데, 비서가 아직 입을 열지 않은 거예요.

---

### 비서에게 대본을 쥐여주다

여기서 한 가지를 떠올렸습니다.

진짜 비서를 생각해봅시다. 비서에게 "연차 며칠 남았어?"라고 물으면, 비서는 어떻게 할까요?

1. 관련 서류함에서 자료를 찾습니다. (검색)
2. 자료를 읽고 이해합니다. (이해)
3. 질문에 맞게 정리해서 말해줍니다. (답변)

v0.3까지는 1번만 했습니다. 서류함에서 자료를 찾아서 책상 위에 올려놓은 것까지는 성공이에요. 하지만 2번과 3번 — 읽고 이해해서 말로 답해주는 건 아직이었습니다.

이 2번과 3번을 해줄 수 있는 게 바로 LLM입니다. Ch.2에서 만났던 그 "똑똑하지만 우리 회사를 모르는 컨설턴트"요. 그때는 아무 자료 없이 질문을 던졌다가 환각을 맞았습니다. 하지만 이번에는 다릅니다. 서류함에서 관련 자료를 먼저 찾아놨으니까요.

컨설턴트에게 브리핑 자료를 건네고 "이 자료를 보고 답해주세요"라고 하면 됩니다. 이때 컨설턴트가 일하는 방식을 정해주는 게 **대본**입니다.

"당신은 사내 문서 Q&A 비서입니다. 제공된 문서에서만 근거를 찾아 답하세요. 답을 찾을 수 없으면 '확인되지 않습니다'라고 하세요. 답변 마지막에 출처를 표시하세요."

이 대본이 **프롬프트(Prompt)** 입니다. LLM에게 "어떤 역할로, 어떤 규칙을 지키며 답하라"고 지시하는 것이에요. 대본이 좋으면 비서가 정확하게 답하고, 대본이 허술하면 비서가 엉뚱한 행동을 합니다. 이 대본을 잘 쓰는 기술을 **프롬프트 엔지니어링**이라고 부릅니다.

---

### 조립 라인을 만들다

비유를 좀 더 구체적으로 해볼게요.

지금 만들려는 건 일종의 **조립 라인**입니다. 자동차 공장의 컨베이어 벨트처럼, 각 단계가 순서대로 이어져서 최종 제품(답변)이 나오는 구조예요.

1단계: 질문이 들어옵니다.
2단계: 서류함(벡터DB)에서 관련 자료를 찾습니다.
3단계: 찾은 자료 + 이전 대화 내용 + 질문을 합쳐서 대본(프롬프트)에 끼워 넣습니다.
4단계: LLM이 대본을 읽고 답변을 만들어냅니다.
5단계: 답변에서 출처 정보를 정리합니다.

이 조립 라인을 코드로 만들 때 쓰는 게 **LangChain의 LCEL**입니다. LCEL은 LangChain Expression Language의 약자인데, 이름이 어려우니 그냥 "조립 라인 문법"이라고 생각하면 됩니다. 파이프(`|`) 기호로 단계를 이어붙입니다. 마치 공장에서 컨베이어 벨트 위의 부품이 다음 공정으로 넘어가듯이요.

검색기 → 프롬프트 → LLM → 파서

이 네 단계가 하나의 파이프라인으로 연결되면, 질문 하나를 넣으면 끝에서 답변이 나옵니다. 이게 **RAG 체인**입니다.

---

### 처음으로 비서가 말을 하다

조립 라인을 만들고, 같은 질문을 다시 던져봤습니다.

> "연차 사용 규정이 어떻게 돼?"

v0.3에서는 이랬습니다.

> "연차는 입사 1년 후 15일이 부여되며, 3년 이상 근속 시 매 2년마다 1일이 가산된다. 연차 사용 시 최소 1일 전..."

v0.4에서는 이렇게 바뀌었습니다.

> "연차는 입사 1년 후 15일이 부여됩니다. 3년 이상 근속하면 2년마다 하루씩 추가됩니다. 연차를 쓰려면 최소 하루 전에 부서장 승인을 받아야 합니다. [출처: HR_취업규칙_v1.0.pdf, 3p]"

차이가 느껴지시나요?

원문 복붙이 아닙니다. LLM이 자료를 읽고, 핵심만 골라서, 사람이 읽기 편한 문체로 다시 말해줬습니다. 출처도 달려 있어서 원본을 확인하고 싶으면 바로 찾아갈 수 있습니다.

"비밀번호 변경은 어떻게 하나요?"

> "보안규정에 따르면 비밀번호는 90일마다 변경해야 합니다. 영문 대소문자, 숫자, 특수문자를 포함하여 8자리 이상이어야 합니다. [출처: SEC_보안규정_v1.0.docx, 2p]"

"상반기 매출이 어떻게 돼?"

> "2025년 상반기 매출 현황을 보면, 영업팀이 가장 높은 매출을 기록했으며... [출처: FIN_2025_상반기_매출현황.xlsx]"

아까 팀장이 원했던 바로 그 모습입니다. 물어보면 알려주는 것.

---

### 하지만 기억을 못 한다

기뻐서 질문을 이어봤습니다.

> "연차 사용 규정 알려줘"
> → (제대로 답변)

> "그러면 경조사 휴가는?"

두 번째 질문에서 문제가 터졌습니다. 비서가 "경조사 휴가"를 단독으로 검색합니다. 방금 연차 규정에 대해 이야기하고 있었다는 걸 모릅니다. 앞의 대화를 기억하지 못하는 거예요.

사람한테 이러면 좀 당황스럽겠죠. "아까 연차 이야기 하고 있었잖아. 경조사 휴가도 같은 취업규칙에 있을 텐데." 이런 맥락을 비서가 전혀 모릅니다.

이유는 간단합니다. LLM은 매번 새로 시작하거든요. 이전 대화를 기억하는 기능이 기본적으로 없습니다. 매 요청이 독립된 건이에요.

해결 방법도 간단합니다. **대화 기록을 적어두면 됩니다.**

비서 옆에 메모장을 하나 놓는 거예요. 질문이 올 때마다 "사용자가 뭘 물었고, 비서가 뭘 답했는지"를 메모장에 적어둡니다. 다음 질문이 오면, 메모장을 보고 "아, 아까 연차 이야기 하고 있었구나"라고 맥락을 파악한 뒤 답합니다.

그런데 메모장이 무한히 길어지면 문제가 생깁니다. 대화 100건의 기록을 매번 LLM에게 보내면 토큰이 폭발합니다. Ch.2에서 겪었던 그 문제요.

그래서 **최근 5건만 기억합니다.** 메모장에 6번째 대화가 적히면, 가장 오래된 첫 번째 대화가 지워지는 방식이에요. 창문을 밀면 새 풍경이 보이고 이전 풍경은 사라지는 것처럼. 이걸 **슬라이딩 윈도우 방식의 대화 기억**이라고 부릅니다.

5건이면 충분할까요? 대부분의 경우 충분합니다. 사람도 아까 뭘 물었는지 5개 전까지는 기억하잖아요. 20분 전에 물었던 질문의 맥락까지 기억할 필요는 없습니다.

---

### 채팅 창이 생겼다

지금까지는 터미널에서 작업했습니다. 검은 화면에 명령을 치고, 텍스트가 쭉 올라오는 방식이었어요. 개발자한테는 익숙하지만, 팀장이나 다른 직원에게 "여기에 `python src/cli_search.py --query '연차 규정'`을 치세요"라고 할 수는 없겠죠.

그래서 채팅 UI를 만들었습니다. Ch.1에서 Admin 대시보드를 만들 때 쓴 FastAPI를 다시 씁니다. 이번에는 대시보드가 아니라 채팅 화면이에요.

브라우저에서 `http://localhost:8000`을 열면 채팅 창이 뜹니다. 카카오톡 같은 대화 인터페이스입니다. 입력창에 질문을 타이핑하고 전송 버튼을 누르면, 비서의 답변이 말풍선으로 올라옵니다.

"연차 사용 규정 알려줘"라고 치면, 잠시 후 비서가 대답합니다.

> "연차는 입사 1년 후 15일이 부여됩니다. 3년 이상 근속하면 2년마다 하루씩 추가됩니다. 연차를 쓰려면 최소 하루 전에 부서장 승인을 받아야 합니다."
> [출처: HR_취업규칙_v1.0.pdf, 3페이지]

이어서 "경조사 휴가는?"이라고 치면, 이번에는 앞의 맥락을 기억합니다.

> "경조사 휴가는 별도 규정에 따릅니다. 본인 결혼 시 5일, 배우자 출산 시 10일, 부모 사망 시 5일 등의 휴가가 부여됩니다."
> [출처: HR_취업규칙_v1.0.pdf, 4페이지]

"그럼"이라고만 물었는데도 맥락을 잡았습니다. 비서가 메모장을 보고 "아, 아까 취업규칙 이야기 하고 있었지"라고 파악한 거예요.

---

### "이거 좋은데?"

팀장에게 다시 보여줬습니다. 이번에는 브라우저 채팅 화면을 열고, 직접 질문을 해보라고 했어요.

"보안 교육은 언제 해야 돼?"

> "전 직원은 연 1회 이상 정보보안 교육을 이수해야 합니다. [출처: HR_정보보안서약서.pdf]"

"우리 팀 야근 수당은?"

> "죄송합니다. 제공된 문서에서 야근 수당 관련 내용을 찾을 수 없습니다. HR팀에 직접 문의하시거나, 관련 문서를 시스템에 추가해주세요."

팀장이 고개를 끄덕였습니다.

> "오, 모르는 건 모른다고 하네. 이거 좋은데?"

Ch.2에서 ChatGPT가 환각을 내뱉던 것과는 완전히 다릅니다. 문서에 없으면 지어내지 않고 "없다"고 합니다. 대본에 "확실하지 않으면 없다고 답하라"고 써놨기 때문이에요.

하지만 팀장이 한 가지 더 물었습니다.

> "그런데 이거, 내 연차가 며칠 남았는지도 대답할 수 있어?"

당연히 안 됩니다. "연차 며칠 남았어"는 DB에 있는 정형 데이터입니다. 이 비서는 사내 문서(비정형 데이터)만 읽을 수 있어요. Ch.1에서 만든 장부에는 접근을 못 합니다.

"비서인데 이것도 모르나?"

이 문제는 다음 챕터에서 해결합니다. 문서 질문이든 DB 질문이든 하나의 창구에서 답하게 만드는 거예요.

---

### 이것만은 기억하자

- **검색 결과에 LLM을 연결하면, 텍스트 조각이 자연어 답변이 된다.** v0.3의 검색(R)에 v0.4의 생성(G)을 더해서 RAG가 완성됐다.
- **프롬프트는 비서의 대본이다.** "문서에서만 답하라", "모르면 모른다고 하라", "출처를 달아라" — 이 지시 하나하나가 비서의 행동을 결정한다.
- **LLM은 대화를 기억하지 못한다.** 매번 새로 시작한다. 메모장(대화 기록)을 만들어서 최근 대화를 함께 보내줘야 맥락을 이해한다.
- **모르면 모른다고 하는 게 좋은 비서다.** 환각보다 "해당 문서에 없습니다"가 훨씬 낫다.
- 다음 챕터에서는 이 비서에게 연장통을 쥐여줍니다. 문서 질문만 답하던 비서가, DB 조회도 할 수 있게 됩니다. "연차 며칠 남았어?"도 "보안 규정 알려줘"도 한 곳에서 답하는 비서가 됩니다.

## 기술 파트

### 용어 정리

| 이야기 속 비유 | 진짜 용어 | 정식 정의 |
|--------------|----------|----------|
| 비서의 대본 | 프롬프트 (Prompt) | LLM에게 역할, 규칙, 맥락을 지시하는 입력 텍스트. 시스템 프롬프트(역할/규칙)와 사용자 프롬프트(질문+참고자료)로 나뉜다. |
| 대본을 잘 쓰는 기술 | 프롬프트 엔지니어링 (Prompt Engineering) | LLM이 원하는 방식으로 응답하도록 프롬프트를 설계하고 최적화하는 기법. "문서에서만 답하라", "모르면 모른다고 하라" 같은 지시를 포함한다. |
| 조립 라인 (컨베이어 벨트) | LCEL (LangChain Expression Language) | LangChain에서 검색기, 프롬프트, LLM, 파서 등의 컴포넌트를 파이프(`\|`) 연산자로 연결하는 선언적 체인 구성 문법. |
| 조립 라인 전체 | RAG 체인 (RAG Chain) | Retriever(검색) → Prompt(조립) → LLM(생성) → Parser(정리)를 하나로 연결한 파이프라인. 질문을 넣으면 근거 기반 답변이 나온다. |
| 비서 옆 메모장 | 대화 히스토리 (Conversation History) | 이전 대화 내용(질문+답변)을 기록하여 LLM에게 맥락으로 전달하는 것. 멀티턴 대화를 가능하게 한다. |
| 메모장에서 최근 5건만 유지 | 윈도우 메모리 (Window Memory) | 고정 크기의 슬라이딩 윈도우로 최근 N턴의 대화만 유지하는 메모리 전략. 오래된 대화는 자동으로 제거되어 토큰 사용량을 제한한다. |
| 검색된 자료 + 질문을 합쳐서 대본에 끼워넣기 | 컨텍스트 주입 | 검색된 문서 조각과 대화 히스토리를 프롬프트 템플릿의 지정된 위치에 삽입하는 과정. LLM이 참고할 수 있는 배경 정보를 제공한다. |

### 설명 ① rag_chain.py — LCEL 파이프라인

이 파일이 v0.4의 심장입니다. 검색기, 프롬프트, LLM, 파서를 파이프라인으로 연결합니다.

**프롬프트 (대본)**:

```python
RAG_SYSTEM_PROMPT = """
당신은 메타코딩 사내 문서 Q&A 비서입니다.

규칙:
1. 반드시 제공된 문서에서만 근거를 찾아 답변
2. 답을 찾을 수 없으면 "해당 내용은 제공된 문서에서 확인되지 않습니다."
3. 답변 마지막에 근거 문서명 명시 [출처: 문서명]
4. 추측이나 외부 지식 사용 금지
"""
```

이 네 줄이 비서의 행동을 결정합니다. 1번이 없으면 LLM이 자기 학습 지식으로 답합니다. 2번이 없으면 모를 때 지어냅니다. 3번이 없으면 출처 없이 답합니다. 4번은 환각을 추가로 억제하는 장치입니다.

**LCEL 파이프라인 (조립 라인)**:

```python
def build_rag_chain():
    retriever = _build_retriever()   # ChromaDB 검색기
    llm = _build_llm()               # Ollama 또는 OpenAI

    chain = (
        {
            "context": itemgetter("question") | retriever | _format_docs,
            "history": itemgetter("history"),
            "question": itemgetter("question"),
        }
        | ChatPromptTemplate.from_messages([
            ("system", RAG_SYSTEM_PROMPT),
            ("human", "이전 대화:\n{history}\n\n참고 문서:\n{context}\n\n질문: {question}"),
        ])
        | llm
        | StrOutputParser()
    )
    return chain, retriever
```

한 줄씩 읽어봅시다.

첫 번째 블록(`{ ... }`)은 입력을 세 갈래로 나눕니다.
- `"context"`: 질문을 받아서 → 검색기로 관련 문서를 찾고 → 텍스트로 포맷팅
- `"history"`: 이전 대화 기록을 그대로 통과
- `"question"`: 질문을 그대로 통과

세 결과가 프롬프트 템플릿의 `{context}`, `{history}`, `{question}` 자리에 각각 들어갑니다. 그 다음 LLM이 답변을 생성하고, `StrOutputParser()`가 문자열로 정리합니다.

`|` 기호가 컨베이어 벨트의 이음새입니다. 앞 단계의 출력이 뒷 단계의 입력이 됩니다.

**문서 포맷팅 함수**:

```python
def _format_docs(docs):
    formatted = []
    for i, doc in enumerate(docs, 1):
        source = doc.metadata.get("file_name", "알 수 없음")
        page = doc.metadata.get("page", "?")
        formatted.append(f"[문서 {i}] 출처: {source}, p.{page}\n{doc.page_content}")
    return "\n\n".join(formatted)
```

검색된 문서 조각에 번호와 출처를 붙여서 프롬프트에 넣기 좋은 텍스트로 변환합니다. LLM이 "[문서 1]"을 참조해서 "[출처: HR_취업규칙_v1.0.pdf]"라고 답할 수 있는 거예요.

> **`itemgetter`란?**: Python 내장 함수입니다. `itemgetter("question")`은 딕셔너리에서 `"question"` 키의 값을 꺼내는 함수를 만들어줍니다. LCEL에서 입력 데이터의 특정 키를 선택할 때 씁니다.

> **검색기(Retriever)**: `_build_retriever()`는 Ch.3에서 만든 ChromaDB를 LangChain의 `Chroma.as_retriever()` 형태로 감싸줍니다. `search_kwargs={"k": 5}`로 상위 5개 문서를 가져옵니다.

### 설명 ② response_parser.py — 답변 정제 + 출처 추출

LLM의 원문 답변을 깔끔하게 정리하는 모듈입니다.

```python
def parse_answer_text(raw_answer):
    """<think>...</think> 태그 제거 (DeepSeek R1용)"""
    cleaned = re.sub(r"<think>.*?</think>", "", raw_answer, flags=re.DOTALL)
    return cleaned.strip()

def parse_sources_from_docs(docs):
    """검색된 문서에서 출처 정보 추출 (중복 제거)"""
    sources = []
    seen = set()
    for doc in docs:
        source = doc.metadata.get("file_name", "알 수 없음")
        page = doc.metadata.get("page", "?")
        key = f"{source}::{page}"
        if key not in seen:
            seen.add(key)
            sources.append({
                "doc": source,
                "page": page,
                "snippet": doc.page_content[:100]
            })
    return sources
```

`parse_answer_text`는 특정 LLM(DeepSeek R1)이 답변에 `<think>` 태그로 추론 과정을 남기는 걸 제거합니다. 사용자에게는 최종 답변만 보여주면 되니까요.

`parse_sources_from_docs`는 검색된 문서 5개에서 출처 정보를 뽑되, 같은 문서의 같은 페이지가 중복되면 한 번만 표시합니다. `"HR_취업규칙_v1.0.pdf::3"` 같은 키로 중복을 체크합니다.

**최종 응답 구조:**

```python
def build_response(raw_answer, docs):
    return {
        "answer": parse_answer_text(raw_answer),
        "sources": parse_sources_from_docs(docs)
    }
```

API가 돌려주는 JSON은 항상 `answer` + `sources` 구조입니다. 채팅 UI가 이 구조를 읽어서 말풍선(답변)과 출처 뱃지(sources)를 각각 렌더링합니다.

### 설명 ③ conversation.py — 멀티턴 대화 관리

이야기 파트에서 비유한 "비서 옆 메모장"을 구현한 모듈입니다.

**WindowMemory (메모장)**:

```python
class WindowMemory:
    def __init__(self, k=5):
        self._history = deque(maxlen=k)   # 최근 k턴만 유지

    def save_turn(self, question, answer):
        self._history.append({"question": question, "answer": answer})

    def get_history(self):
        lines = []
        for turn in self._history:
            lines.append(f"사용자: {turn['question']}")
            lines.append(f"AI 비서: {turn['answer']}")
        return "\n".join(lines) if lines else "없음"
```

`deque(maxlen=5)`가 핵심입니다. Python의 `deque`는 최대 크기를 정할 수 있는 큐입니다. 5턴이 차 있는 상태에서 새 대화가 추가되면, 가장 오래된 대화가 자동으로 빠집니다. 이야기 파트에서 말한 "창문을 밀면 새 풍경이 보이고 이전 풍경은 사라지는" 그 방식이에요.

`get_history()`는 저장된 대화를 텍스트로 변환합니다. 이 텍스트가 프롬프트의 `{history}` 자리에 들어갑니다.

**ConversationManager (세션별 메모장 관리)**:

```python
class ConversationManager:
    def __init__(self):
        self._sessions = {}           # {session_id: (WindowMemory, last_access)}
        self._window_size = int(os.getenv("CONVERSATION_WINDOW_SIZE", "5"))
        self._ttl = int(os.getenv("SESSION_TTL_SECONDS", "3600"))

    def get_history_text(self, session_id):
        memory = self._get_or_create_memory(session_id)
        return memory.get_history()

    def save_turn(self, session_id, question, answer):
        memory = self._get_or_create_memory(session_id)
        memory.save_turn(question, answer)
```

채팅 UI에 동시에 여러 사람이 접속할 수 있으니, 사람마다(세션마다) 별도의 메모장을 관리합니다. A 직원의 대화 기록과 B 직원의 대화 기록이 섞이면 안 되니까요.

세션은 1시간(3600초) 동안 접근이 없으면 자동으로 삭제됩니다. 퇴근 후 방치된 세션이 메모리를 계속 잡고 있는 걸 방지합니다.

> **왜 5턴인가요?** `.env` 파일의 `CONVERSATION_WINDOW_SIZE`로 조절할 수 있습니다. 5턴이면 사용자 질문 5개 + AI 답변 5개 = 10개 메시지입니다. 한국어 기준으로 약 2000~3000 토큰 정도예요. LLM 컨텍스트에서 검색 결과(~1500 토큰) + 프롬프트(~200 토큰)와 합치면 4000~5000 토큰으로, 대부분의 모델이 충분히 처리할 수 있는 양입니다.

### 설명 ④ chat_api.py — POST /api/chat 엔드포인트

채팅 UI에서 질문을 보내면 이 엔드포인트가 처리합니다. 전체 흐름을 순서대로 보겠습니다.

```python
@router.post("/api/chat")
async def chat(body: ChatRequest, request: Request):
    # ① 세션 확인
    session_id = body.session_id or get_session_id(request)

    # ② 대화 기록 조회
    history = conv_manager.get_history_text(session_id)

    # ③ 관련 문서 검색
    docs = retriever.invoke(body.question)

    # ④ RAG 체인 실행
    raw_answer = chain.invoke({
        "question": body.question,
        "history": history
    })

    # ⑤ 답변 정제 + 출처 추출
    result = build_response(raw_answer, docs)

    # ⑥ 대화 기록 저장
    conv_manager.save_turn(session_id, body.question, result["answer"])

    # ⑦ 응답 반환
    response = JSONResponse(content={
        "answer": result["answer"],
        "sources": result["sources"],
        "session_id": session_id
    })
    set_session_cookie(response, session_id)
    return response
```

7단계가 순서대로 실행됩니다.

1. 세션 ID를 확인합니다. 쿠키에 있으면 기존 세션, 없으면 새 UUID를 발급합니다.
2. 해당 세션의 대화 기록(최근 5턴)을 가져옵니다.
3. 질문으로 ChromaDB에서 관련 문서 5개를 검색합니다.
4. 질문 + 대화 기록을 RAG 체인에 넣어서 답변을 생성합니다.
5. 답변을 정제하고 출처를 추출합니다.
6. 이번 대화를 세션의 메모장에 저장합니다.
7. JSON 응답을 돌려주고, 세션 쿠키를 설정합니다.

**요청/응답 스키마:**

```python
class ChatRequest(BaseModel):
    question: str           # 사용자 질문 (1~2000자)
    session_id: str | None  # 선택 (없으면 자동 생성)

class ChatResponse(BaseModel):
    answer: str             # AI 답변
    sources: list[dict]     # [{"doc": "...", "page": 1, "snippet": "..."}]
    session_id: str         # 세션 ID
```

> **대화 초기화**: `DELETE /api/chat/session`을 호출하면 해당 세션의 대화 기록만 초기화됩니다. 새로운 주제로 질문하고 싶을 때 씁니다. 채팅 UI에 "새 대화" 버튼이 이 API를 호출합니다.

### 더 알아보기

- **프롬프트 엔지니어링 팁**: "문서에서만 답하라"만으로는 부족할 수 있습니다. "추측 금지", "모르면 모른다고 답하라", "답변은 3문장 이내로" 같은 구체적 규칙을 추가할수록 답변 품질이 올라갑니다. 반대로 규칙이 너무 많으면 LLM이 혼란스러워할 수 있어요. 4~6개 규칙이 적당합니다.
- **LCEL vs 전통적 체인**: LangChain 이전 버전에서는 `LLMChain`, `SequentialChain` 같은 클래스로 체인을 만들었습니다. LCEL은 파이프(`|`) 연산자로 더 직관적이고 간결하게 표현합니다. 새 프로젝트라면 LCEL을 쓰는 게 좋습니다.
- **Ollama vs OpenAI**: Ollama는 로컬에서 LLM을 실행합니다. 사내 문서가 외부로 나가지 않아서 보안에 유리합니다. 다만 GPU가 없으면 응답이 느릴 수 있어요. OpenAI는 빠르고 정확하지만 외부 API를 사용합니다. `.env`의 `LLM_PROVIDER`로 전환할 수 있습니다.
- **세션 관리와 보안**: 세션 쿠키는 `HTTPOnly`(JavaScript 접근 차단)와 `SameSite=lax`(CSRF 방지)로 설정되어 있습니다. 프로덕션에서는 HTTPS + `Secure` 옵션도 켜야 합니다.
- **WindowMemory 말고 다른 방식은?**: 전체 대화를 요약해서 저장하는 Summary Memory, 중요한 대화만 선별하는 Entity Memory 등이 있습니다. WindowMemory는 가장 단순하고 예측 가능해서 학습 단계에서 쓰기 좋습니다.
