# Ch.5: 비서에게 연장통을 쥐여주자 — v0.5

> 이번 버전: v0.4 → v0.5
> 한 줄 요약: 문서 질문만 답하던 비서가, DB 조회까지 해서 하나의 창구에서 모든 질문에 답하게 되었다.
> 핵심 개념: 질문 라우팅, MCP 도구, ReAct 에이전트

---

## 이야기 파트

### 비서인데 이것도 모르나?

Ch.4에서 팀장이 채팅 UI를 직접 써본 뒤, 꽤 만족스러워했습니다. "보안 교육은 언제 해야 돼?" — 답을 잘합니다. 모르면 모른다고 합니다. 좋았어요.

그런데 팀장이 한 가지를 더 물었죠.

> "내 연차 며칠 남았어?"

비서가 대답을 못 합니다. 아니, 대답은 합니다. 사내 문서에서 "연차 사용 규정"을 찾아서 규정 내용을 읊어줍니다. 하지만 팀장이 원한 건 그게 아니에요. "15일 중에 7일 썼으니까 8일 남았습니다"라는 답을 원한 겁니다.

왜 못 했을까요?

"연차 며칠 남았어"는 사내 문서에 적혀 있는 게 아닙니다. 이건 Ch.1에서 만든 **장부(DB)** 에 있는 숫자예요. 직원 테이블에 누가 언제 입사했는지, 연차 테이블에 총 15일 중 7일을 썼다는 게 기록되어 있습니다. 그런데 v0.4 비서는 서류함(벡터DB)만 뒤질 줄 알지, 장부(PostgreSQL)를 열어볼 줄 모릅니다.

"영업팀 매출 얼마야?"도 마찬가지입니다. 매출 숫자는 장부에 있습니다. 비서가 사내 문서에서 "매출"이라는 단어를 찾아서 보고서 형식의 문단을 가져오긴 하는데, 실시간 숫자가 아닙니다.

비유하자면 이렇습니다. 비서가 서류함은 열 줄 아는데, 컴퓨터 앞에 앉아서 엑셀을 열어볼 줄은 모르는 상태. 질문의 종류에 따라 다른 능력이 필요한데, 비서에게 서류 뒤지기라는 한 가지 기술만 가르쳐 준 거예요.

---

### 안내데스크가 필요하다

잠깐, 실제 회사를 생각해봅시다.

큰 회사에 가면 1층에 **안내데스크**가 있습니다. 방문객이 오면 이렇게 물어보죠.

> "저 인사팀 찾아왔는데요." → "3층 왼쪽입니다."
> "택배 보내려는데요." → "지하 1층 우편실입니다."
> "김 부장님 만나려는데요." → "먼저 3층 인사팀에서 방문 등록 하시고, 5층으로 가세요."

안내데스크가 하는 일은 간단합니다. **질문을 듣고, 어디로 보내야 할지 판단하는 것.** 택배 질문을 인사팀으로 보내면 안 되고, 인사 질문을 우편실로 보내면 안 됩니다.

우리 비서에게도 이 안내데스크가 필요합니다.

"연차 며칠 남았어?" → 이건 장부(DB)를 열어봐야 해. 서류함이 아니라.
"보안 규정 알려줘" → 이건 서류함(벡터DB)을 뒤져야 해. 장부가 아니라.
"이서연의 연차 잔여일이랑 연차 사용 규정 둘 다 알려줘" → 이건 둘 다 봐야 해.

이렇게 질문을 듣고 어디로 보낼지 판단하는 게 **라우터(Router)** 입니다. 안내데스크의 직원처럼, 질문의 종류를 파악해서 적절한 곳으로 연결해주는 역할이에요.

---

### 연장통을 만들다

안내데스크가 "3층 인사팀으로 가세요"라고 안내한다 해도, 인사팀에 도착했을 때 실제로 업무를 처리할 수 있어야 의미가 있겠죠. 우리 비서에게도 실제로 일을 할 수 있는 **도구**가 필요합니다.

연장통을 생각해봅시다. 연장통에는 드라이버, 망치, 펜치 같은 도구가 들어 있고, 상황에 따라 필요한 도구를 꺼내 씁니다. 못을 박을 때는 망치를, 나사를 풀 때는 드라이버를 꺼내죠.

비서의 연장통에는 4가지 도구가 들어갑니다.

1. **직원 목록 조회** — "영업팀 누가 있어?" → 장부에서 직원 명단을 꺼내옴
2. **연차 잔여 조회** — "이서연 연차 며칠?" → 장부에서 연차 숫자를 꺼내옴
3. **매출 합계 조회** — "이번 달 매출 얼마야?" → 장부에서 매출 합계를 계산
4. **문서 검색** — "보안 규정 알려줘" → 서류함에서 관련 문서를 찾아옴

4번은 Ch.3~4에서 이미 만든 겁니다. 1~3번이 새로 추가됩니다. Ch.1에서 만든 장부(PostgreSQL)에 접근하는 도구들이에요.

이 도구들은 하나씩 독립적입니다. 연차 조회 도구는 연차만 찾습니다. 매출 도구는 매출만 계산합니다. 각자 자기 일만 하고 결과를 돌려줍니다. 드라이버가 못을 박지 않듯이요.

---

### 비서가 스스로 판단하기 시작했다

안내데스크(라우터)도 만들었고, 연장통(도구 4개)도 채웠습니다. 그런데 한 가지 문제가 남아 있어요.

누가 연장통에서 도구를 꺼내서 쓸 건가요?

지금까지의 비서는 시키는 대로만 했습니다. "서류함에서 찾아" 하면 서류함을 뒤지고, "장부 열어" 하면 장부를 엽니다. 하지만 우리가 원하는 건, 질문을 듣고 **스스로 어떤 도구를 꺼내야 할지 판단하는 비서**입니다.

"이서연의 연차 잔여일이랑 연차 규정 같이 알려줘."

이 질문을 받으면, 비서가 이렇게 생각해야 합니다.

> "연차 잔여일 → 숫자 데이터니까 '연차 조회' 도구를 쓰자."
> (도구 실행 → "이서연: 15일 중 3일 사용, 12일 남음")
> "연차 규정 → 문서에 있을 테니 '문서 검색' 도구를 쓰자."
> (도구 실행 → "연차는 입사 1년 후 15일 부여...")
> "두 결과를 합쳐서 답변하자."

이렇게 **생각하고(Reasoning) → 행동하고(Acting) → 결과를 보고 → 또 생각하는** 패턴을 **ReAct**라고 부릅니다. 사람이 문제를 풀 때의 사고 과정과 비슷합니다. "이건 뭘로 해결하지?" → 시도 → "됐나?" → 안 됐으면 다른 방법 시도.

진짜 비서도 이렇게 일하잖아요. "부장님, 이서연 씨 연차요?" 하면 비서가 인사 시스템을 열어보고, 필요하면 취업규칙도 찾아보고, 두 정보를 합쳐서 "12일 남았습니다. 참고로 3년 이상 근속하면 매 2년마다 1일이 추가됩니다"라고 답해주는 거예요.

이 비서 — 생각하고, 도구를 꺼내고, 결과를 판단하는 비서 — 를 **에이전트(Agent)** 라고 부릅니다.

---

### 드디어 하나의 창구에서

에이전트를 완성하고, 같은 질문을 다시 해봤습니다.

"이서연의 연차 며칠 남았어?"

v0.4에서는 이랬습니다.

> "연차는 입사 1년 후 15일이 부여됩니다. 3년 이상 근속하면..."
> (규정만 읊는다. 숫자를 모른다.)

v0.5에서는 이렇게 바뀌었습니다.

> "이서연 님의 연차 현황입니다. 총 15일 중 3일을 사용하여 12일이 남아 있습니다."

달라졌습니다. 비서가 직접 장부를 열어서 숫자를 확인한 겁니다.

이번에는 복합 질문을 던져봤습니다.

"이서연의 연차 잔여일이랑 연차 사용 규정 같이 알려줘."

> "이서연 님은 총 15일 중 3일을 사용하여 현재 12일의 연차가 남아 있습니다.
> 연차 사용 규정에 따르면, 연차는 입사 1년 후 15일이 부여되며, 3년 이상 근속 시 매 2년마다 1일이 가산됩니다. 사용 시 최소 1일 전 부서장 승인이 필요합니다."

하나의 질문에 장부(DB) 조회와 서류함(문서) 검색을 동시에 한 겁니다. 비서가 연장통에서 두 가지 도구를 차례로 꺼내 쓴 거예요.

팀장에게 보여줬습니다.

> "영업팀 매출 얼마야?"

> "영업팀의 2024년 11~12월 매출 합계는 약 3,680만 원입니다. 김민준 과장이 약 2,120만 원으로 가장 높은 실적을 기록했습니다."

> "오, 이제 진짜 비서 같은데?"

드디어 하나의 창구에서 모든 질문에 답합니다. "보안 규정 알려줘"도, "연차 며칠 남았어?"도, "매출 얼마야?"도. 질문의 종류에 따라 비서가 알아서 장부를 열거나 서류함을 뒤집니다.

---

### 실패도 있었다

물론 처음부터 매끄럽진 않았습니다.

안내데스크(라우터)를 처음 만들었을 때, "김민준 과장 이번 달 실적 어때?"라는 질문을 서류함(문서 검색)으로 보내버린 적이 있습니다. "실적"이라는 단어가 보고서 문서에도 자주 나오거든요. 하지만 이건 장부에서 숫자를 찾아야 하는 질문이에요.

그래서 안내데스크를 3단계로 만들었습니다.

1단계: **키워드로 빠르게 판단.** "연차", "매출", "목록" 같은 단어가 있으면 장부로 보냄. "규정", "절차", "정책" 같은 단어가 있으면 서류함으로 보냄.
2단계: 1단계에서 결론이 안 나면, **DB 테이블 이름과 비교.** "remaining_days" 같은 기술적 표현이 있으면 장부로 보냄.
3단계: 그래도 모르겠으면, **LLM에게 물어봄.** "이 질문은 숫자를 찾는 건가, 절차를 찾는 건가?"

3단계까지 가는 경우는 드뭅니다. 대부분 1단계에서 결정됩니다. "연차 며칠?" → "연차" 키워드 발견 → 장부로. "보안 규정 알려줘" → "규정" 키워드 발견 → 서류함으로. 빠르고 정확합니다.

그래도 양쪽 키워드가 동시에 나타나면? "연차 규정 알려주고 내 잔여일도 확인해줘." 이런 경우에는 **둘 다 보내는** 복합(hybrid) 경로로 분류합니다.

---

### 이것만은 기억하자

- **안내데스크(라우터)가 질문의 종류를 판단한다.** 장부(DB)를 볼 건지, 서류함(문서)을 뒤질 건지, 둘 다 볼 건지.
- **연장통(도구)은 비서가 실제로 일할 수 있게 해주는 장비다.** 직원 조회, 연차 조회, 매출 조회, 문서 검색 — 4개 도구가 각각 하나의 일만 한다.
- **에이전트는 스스로 판단하는 비서다.** 질문을 듣고, 어떤 도구를 꺼내야 할지 생각하고, 실행하고, 결과를 보고 또 판단한다(ReAct).
- **하나의 창구에서 모든 질문에 답할 수 있게 되었다.** 문서 질문이든, DB 질문이든, 복합 질문이든.
- 다음 챕터에서는 이 비서에게 매뉴얼을 줍니다. 지금은 되기는 하는데, 구조가 정리되지 않은 상태예요. 캐시로 같은 질문에 빠르게 답하고, 로그로 비서의 행동을 기록하는 — 제대로 된 업무 체계를 잡아주는 거예요.

## 기술 파트

### 용어 정리

| 이야기 속 비유 | 진짜 용어 | 정식 정의 |
|--------------|----------|----------|
| 안내데스크 | 라우터 (Query Router) | 사용자 질문을 분석하여 정형(DB 조회), 비정형(문서 검색), 복합(둘 다) 중 적절한 처리 경로로 분기하는 모듈. |
| 연장통 | 도구 (Tool / MCP Tool) | 에이전트가 호출할 수 있는 개별 기능 단위. `@tool` 데코레이터로 정의하며, 입력을 받아 결과를 돌려준다. LangChain에서는 이를 MCP(Model Context Protocol) 도구라고도 부른다. |
| 스스로 판단하는 비서 | 에이전트 (Agent) | LLM이 도구를 선택·실행·판단하는 자율적 실행 주체. 질문을 받으면 어떤 도구를 쓸지 스스로 결정하고, 결과를 보고 추가 행동 여부를 판단한다. |
| 생각 → 행동 → 관찰 | ReAct (Reasoning + Acting) | LLM이 (1) 현재 상황을 분석하고(Reasoning), (2) 도구를 실행하고(Acting), (3) 결과를 관찰한 뒤(Observation) 다음 행동을 결정하는 패턴. 복잡한 질문을 단계적으로 해결할 수 있다. |
| 안내데스크 3단계 판단 | 3단계 폴백 전략 | Step 1(규칙 기반 키워드 매칭) → Step 2(DB 스키마명 매칭) → Step 3(LLM 판단) 순서로 시도하는 질문 분류 방식. 빠르고 확실한 방법을 먼저 쓰고, 안 되면 점점 정교한 방법으로 넘어간다. |
| 장부에서 숫자 꺼내기 | SQL 쿼리 (Structured Query) | PostgreSQL 같은 관계형 데이터베이스에서 데이터를 조회하는 구조화된 질의. `SELECT`, `JOIN`, `WHERE` 등의 문법을 사용한다. |

### 실습 ① schema.sql — DB 초기화 + 샘플 데이터

v0.5부터 장부(PostgreSQL)를 사용합니다. Ch.1에서 Docker로 PostgreSQL을 띄웠던 것과 같은 방식으로 DB를 준비합니다.

`CH08_통합_에이전트_설계/` 폴더에서 아래 명령을 실행하세요.

```bash
docker-compose up -d
```

Docker가 PostgreSQL 컨테이너를 띄우고, `data/schema.sql`이 자동으로 실행됩니다.

**어떤 데이터가 들어가나요?**

3개의 테이블이 생성됩니다.

```
employees      — 직원 10명 (5개 부서: 영업부, 인사부, 개발부, 마케팅부)
leave_balance  — 연차 정보 10건 (총 일수, 사용 일수, 잔여 일수)

```

예를 들어, 이서연(E002)의 데이터는 이렇습니다.


### 설명 ① router.py — 3단계 QueryRouter

이야기 파트에서 비유한 "안내데스크"를 구현한 모듈입니다. 질문이 들어오면 3단계를 거쳐 어디로 보낼지 결정합니다.

**키워드 정의:**

```python
# 정형 데이터(DB) 관련 키워드 — 숫자/통계/명단 조회
STRUCTURED_KEYWORDS = [
    "잔여", "잔량", "연차", "휴가", "남은", "몇 일", "며칠",
    "매출", "합계", "총액", "금액", "얼마", "실적",
    "목록", "명단", "직원", "사원", "리스트", "조회",
    "통계", "평균", "부서별", "합산",
]

# 비정형 데이터(문서) 관련 키워드 — 절차/정책/안내
UNSTRUCTURED_KEYWORDS = [
    "절차", "방법", "어떻게", "규정", "정책", "기준",
    "온보딩", "입사", "안내", "가이드", "매뉴얼",
    "복지", "혜택", "보안", "출장", "비용",
    "설명해", "무엇인가", "어떤가",
]
```

이 키워드 목록이 안내데스크의 "판단 기준표"입니다. "연차"가 들어간 질문은 장부(DB)로, "규정"이 들어간 질문은 서류함(문서)으로 보냅니다.

**3단계 분류 로직:**

```python
def classify_query(self, query: str) -> str:
    # ① Step 1: 규칙 기반 키워드 매칭
    step1_result = self._step1_rule_based(query)
    if step1_result is not None:
        return step1_result

    # ② Step 2: DB 스키마 컬럼명 매칭
    step2_result = self._step2_schema_based(query)
    if step2_result is not None:
        return step2_result

    # ③ Step 3: LLM 판단 (폴백)
    if self._llm is not None:
        step3_result = self._step3_llm_based(query)
        if step3_result is not None:
            return step3_result

    # ④ 기본값: 비정형으로 처리
    return "unstructured"
```

위에서 아래로 순서대로 시도합니다. Step 1에서 결정되면 끝, 안 되면 Step 2, 그래도 안 되면 Step 3. 어디서도 결론이 안 나면 기본적으로 서류함(문서 검색)으로 보냅니다. 모르는 질문이 왔을 때 DB에 엉뚱한 SQL을 날리는 것보다, 문서에서 답을 찾아보는 게 더 안전하기 때문입니다.

**반환값은 3가지입니다:**

| 반환값 | 의미 | 예시 질문 |
|--------|------|----------|
| `"structured"` | 장부(DB)만 조회 | "이서연 연차 며칠?" |
| `"unstructured"` | 서류함(문서)만 검색 | "보안 규정 알려줘" |
| `"hybrid"` | 둘 다 실행 | "연차 잔여일이랑 규정 같이 알려줘" |

**Step 1 — 키워드 매칭 상세:**

```python
def _step1_rule_based(self, query: str) -> Optional[str]:
    query_lower = query.lower()
    structured_hits = sum(1 for kw in STRUCTURED_KEYWORDS if kw in query_lower)
    unstructured_hits = sum(1 for kw in UNSTRUCTURED_KEYWORDS if kw in query_lower)

    # 두 쪽 모두 히트 → hybrid (한 쪽이 2배 이상 우세하면 그 쪽)
    if structured_hits > 0 and unstructured_hits > 0:
        if structured_hits >= unstructured_hits * 2:
            return "structured"
        if unstructured_hits >= structured_hits * 2:
            return "unstructured"
        return "hybrid"

    if structured_hits > 0:
        return "structured"
    if unstructured_hits > 0:
        return "unstructured"
    return None
```

양쪽 키워드가 동시에 나타나면 hybrid로 분류합니다. 단, 한 쪽이 2배 이상 압도적이면 그 쪽으로 보냅니다. "연차 잔여일 조회 매출 합산" 같은 질문은 정형 키워드가 4개인데 비정형은 0개이니 `structured`로 갑니다.

> **Step 3의 LLM 판단은 언제 쓰이나요?** "김민준 과장 이번 달 실적 어때?" 같은 질문은 Step 1에서 "실적" 키워드로 `structured`를 잡기도 하지만, "요즘 회사 분위기가 어때?" 같은 모호한 질문은 Step 1, 2 모두 결론을 못 내립니다. 이때 LLM에게 JSON 형식으로 분류를 요청합니다. 빈도가 낮으므로 성능 부담은 크지 않습니다.

### 설명 ② mcp_tools.py — 4개 도구 정의

이야기 파트에서 비유한 "연장통"을 구현한 모듈입니다. 4개의 도구가 `@tool` 데코레이터로 정의되어 있습니다.

**도구 1: leave_balance — 연차 잔여 조회**

```python
@tool
def leave_balance(emp_no: str) -> dict:
    """직원의 연차 잔여 일수를 조회한다."""
    if emp_no.startswith("E") and emp_no[1:].isdigit():
        rows = _run_query("""
            SELECT e.emp_no, e.name, e.department,
                   l.total_days, l.used_days,
                   (l.total_days - l.used_days) AS remaining_days
            FROM employees e
            JOIN leave_balance l ON e.emp_no = l.emp_no
            WHERE e.emp_no = %s
        """, (emp_no,))
    else:
        rows = _run_query("""
            SELECT e.emp_no, e.name, e.department,
                   l.total_days, l.used_days,
                   (l.total_days - l.used_days) AS remaining_days
            FROM employees e
            JOIN leave_balance l ON e.emp_no = l.emp_no
            WHERE e.name LIKE %s
        """, (f"%{emp_no}%",))

    if rows:
        return rows[0]
    return {"error": f"직원 '{emp_no}'을(를) 찾을 수 없습니다."}
```

`@tool` 데코레이터가 핵심입니다. 이걸 붙이면 이 함수가 LangChain의 "도구"로 등록됩니다. 에이전트가 "연차 조회 도구를 써야겠다"고 판단하면 이 함수를 호출합니다.

직원 번호("E002")로 검색할 수도 있고, 이름("이서연")으로 검색할 수도 있습니다. LLM이 "이서연"이라는 이름을 넘기면, SQL의 `LIKE %이서연%`으로 매칭합니다.

**도구 2: sales_sum — 매출 합계 조회**

```python
@tool
def sales_sum(dept: str = "", start_date: str = "", end_date: str = "") -> dict:
    """부서별 또는 전체 매출 합계를 조회한다."""
    start = start_date or "2024-11-01"
    end = end_date or "2024-12-31"

    rows = _run_query(f"""
        SELECT e.department, e.name AS employee_name,
               SUM(s.amount) AS total_amount, COUNT(*) AS record_count
        FROM sales s
        JOIN employees e ON s.emp_no = e.emp_no
        WHERE s.sale_date BETWEEN %s AND %s {dept_filter}

    if rows:
```

부서명, 기간으로 필터링할 수 있습니다. 기본값이 정해져 있어서, 그냥 "매출 얼마야?"라고만 물으면 전체 기간의 전체 부서 매출을 보여줍니다.

> **`@tool` 데코레이터란?** Python의 데코레이터 문법입니다. 함수 위에 `@tool`을 붙이면 LangChain이 그 함수의 이름, 설명(docstring), 파라미터 정보를 자동으로 읽어서 LLM에게 "이런 도구가 있어"라고 알려줍니다. LLM은 이 정보를 보고 어떤 도구를 쓸지 판단합니다. docstring이 곧 도구의 사용 설명서인 셈이에요.

### 설명 ③ agent.py — ReAct 에이전트

이야기 파트에서 비유한 "스스로 판단하는 비서"를 구현한 모듈입니다.

**시스템 프롬프트 (비서 매뉴얼):**

```python
SYSTEM_PROMPT = """당신은 사내 HR 및 업무 질문에 답변하는 AI 어시스턴트입니다.

사용 가능한 도구:
- leave_balance: 직원 연차 잔여 조회 (emp_no 또는 이름으로 검색)
- sales_sum: 매출 합계 조회 (부서, 기간 필터 가능)
- list_employees: 직원 목록 조회 (부서 필터 가능)
- search_documents: 사내 문서 검색 (절차, 정책, 안내 등)

규칙:
1. 정형 데이터(숫자/통계/목록)는 DB 조회 도구를 사용하세요.
2. 비정형 질문(절차/정책/설명)은 search_documents를 사용하세요.
3. 복합 질문은 두 종류의 도구를 모두 사용하세요.
4. 답변은 반드시 한국어로 작성하세요.
5. 도구 실행 결과의 핵심 정보만 추출하여 자연스러운 문장으로 답변하세요."""
```

Ch.4의 프롬프트가 "문서에서만 답하라"였다면, 이번에는 "이런 도구가 있으니 상황에 맞게 써라"입니다. 비서에게 대본(프롬프트)을 줄 때, 연장통의 목록까지 같이 알려주는 거예요.

**AgentExecutor 생성:**

```python
def _build_agent_executor(self):
    from langchain.agents import AgentExecutor, create_tool_calling_agent
    from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

    # ① 프롬프트 구성
    prompt = ChatPromptTemplate.from_messages([
        ("system", SYSTEM_PROMPT),
        MessagesPlaceholder(variable_name="chat_history", optional=True),
        ("human", "{input}"),
        MessagesPlaceholder(variable_name="agent_scratchpad"),
    ])

    # ② Tool Calling Agent 생성
    agent = create_tool_calling_agent(
        llm=self._llm,
        tools=ALL_TOOLS,
        prompt=prompt,
    )

    # ③ AgentExecutor 래핑
    return AgentExecutor(
        agent=agent,
        tools=ALL_TOOLS,
        return_intermediate_steps=True,
        max_iterations=10,
    )
```

한 줄씩 봅시다.

①의 `agent_scratchpad`가 핵심입니다. 이건 비서의 "메모지"예요. 에이전트가 도구를 실행할 때마다 "어떤 도구를 골랐고, 결과가 뭐였는지"가 여기에 기록됩니다. 에이전트는 이 메모지를 보면서 "다음에 뭘 해야 하나?"를 판단합니다.

②의 `create_tool_calling_agent`가 ReAct 패턴을 구현합니다. LLM에게 "이 도구들이 있으니, 질문에 답하기 위해 필요한 도구를 골라서 써라"고 지시합니다. LLM이 `leave_balance`를 호출하겠다고 결정하면, 실제로 함수가 실행되고 결과가 돌아옵니다.

③의 `max_iterations=10`은 안전장치입니다. 에이전트가 무한 루프에 빠지지 않도록, 최대 10번까지만 도구를 쓸 수 있게 제한합니다.

**에이전트 실행 흐름:**

```python
def run(self, query: str) -> dict:
    # ① 질문 유형 분류
    query_type = self._router.classify_query(query)

    # ② 에이전트 실행
    result = self._agent_executor.invoke({"input": query})
    answer = result.get("output", "답변을 생성하지 못했습니다.")
    steps = result.get("intermediate_steps", [])

    # ③ 중간 단계에서 정형/비정형 데이터 추출
    structured_data, unstructured_data = self._parse_result(steps)

    return {
        "answer": answer,
        "query_type": query_type,
        "structured_data": structured_data,
        "unstructured_data": unstructured_data,
        "steps": self._serialize_steps(steps),
    }
```

①에서 라우터가 질문 종류를 판별합니다. ②에서 에이전트가 실제로 도구를 골라 실행합니다. `intermediate_steps`에는 에이전트가 "어떤 도구를 썼고, 결과가 뭐였는지"가 기록되어 있습니다. ③에서 이 중간 단계를 정형 데이터(DB 결과)와 비정형 데이터(문서 결과)로 분류합니다.

> **`return_intermediate_steps=True`가 왜 필요한가요?** 에이전트가 어떤 도구를 어떤 순서로 사용했는지 추적하기 위해서입니다. "이서연의 연차 잔여일이랑 규정 알려줘"라는 질문에 에이전트가 `leave_balance` → `search_documents` 순서로 도구를 사용했다면, 이 정보가 `intermediate_steps`에 기록됩니다. 디버깅할 때도, 채팅 UI에서 "비서가 어떤 과정을 거쳤는지" 보여줄 때도 유용합니다.

### 더 알아보기

- **라우터 vs 에이전트 직접 판단**: 라우터 없이 에이전트에게 모든 판단을 맡길 수도 있습니다. 하지만 LLM 호출은 느리고 비용이 듭니다. 키워드로 빠르게 걸러낼 수 있는 질문은 라우터가 먼저 처리하고, 모호한 것만 에이전트에게 넘기는 게 효율적입니다.
- **ReAct vs 단순 체인**: Ch.4의 RAG 체인은 검색 → 프롬프트 → LLM → 파서라는 고정된 순서였습니다. ReAct 에이전트는 고정 순서가 아니라, LLM이 상황에 따라 도구를 선택합니다. 유연하지만 예측이 어려울 수 있습니다. 단순한 문서 검색만 필요하면 체인이 더 빠르고 안정적입니다.
- **Tool Calling vs Function Calling**: OpenAI에서 먼저 "Function Calling"이라는 이름으로 시작했고, 이후 "Tool Calling"으로 이름이 바뀌었습니다. LangChain에서는 `create_tool_calling_agent`를 사용합니다. 본질은 같습니다 — LLM이 "이 도구를 이 파라미터로 호출하겠다"고 결정하는 것.
- **에이전트의 한계**: 에이전트는 LLM의 판단에 의존합니다. LLM이 잘못된 도구를 선택하거나, 파라미터를 잘못 넘기면 엉뚱한 결과가 나옵니다. `handle_parsing_errors=True`로 파싱 오류 시 자동 재시도하게 했지만, 완벽하지는 않습니다.
- **도구를 더 추가하려면?** `@tool` 데코레이터로 새 함수를 만들고 `ALL_TOOLS` 리스트에 넣으면 됩니다. 예를 들어 이메일 전송, 캘린더 조회 같은 도구를 추가하면 비서의 능력이 확장됩니다. 시스템 프롬프트에 새 도구 설명만 추가하면 에이전트가 바로 사용할 수 있습니다.
