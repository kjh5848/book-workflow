## 이야기 파트

### 팀장의 한마디

입사한 지 세 달이 지난 어느 오후였습니다.

팀장이 슬랙 메시지를 하나 보냈습니다.

> "우리 회사 문서 너무 많은데. 직원 연차도 물어봐야 알고, 매출 자료도 여기저기 흩어져 있고. 물어보면 알려주는 거 하나 만들어봐. AI로."

딱 두 줄이었습니다.

AI 비서를 만들어라. 회사 자료를 물어보면 답해주는 것.

처음엔 "ChatGPT에 연결하면 되겠지?" 싶었습니다. 그런데 잠깐, ChatGPT가 우리 회사 직원 이름을 알까요? 우리 팀의 이번 달 매출을 알까요? 연차가 몇 개 남았는지 알까요?

당연히 모릅니다. ChatGPT는 우리 회사를 다닌 적이 없으니까요.

그렇다면 어디서부터 시작해야 할까요?

비서를 고용하면 제일 먼저 뭘 해야 하는지 생각해봤습니다. 업무를 익혀야 하죠. 회사 장부를 펼쳐놓고 "우리 직원은 누가 있고, 연차는 어떻게 쓰고, 매출은 어느 부서가 잘 하고 있는지" 보여줘야 합니다.

AI 비서도 마찬가지입니다. 먼저 **비서가 볼 수 있는 장부**가 있어야 합니다.

이번 챕터는 그 장부를 만드는 이야기입니다.

### 장부, 서랍장, 그리고 웨이터

회사 인사팀을 생각해보세요.

한쪽 벽에 커다란 서랍장이 있습니다. 서랍마다 이름표가 붙어 있어요. "직원 명단", "연차 기록", "매출 내역". 담당자가 필요한 서랍을 열어서 자료를 꺼내고, 수정하고, 다시 넣어두는 거죠.

이게 **데이터베이스(DB)** 입니다. 우리 코드에서는 PostgreSQL이라는 서랍장을 씁니다.

그런데 서랍장에 아무나 접근하면 안 되겠죠. 중요한 자료가 있으니까요. 그래서 인사팀 앞에는 창구 담당자가 있습니다. 직접 서랍장에 손 대는 게 아니라, 창구 담당자에게 요청하면 필요한 자료를 가져다 주는 거예요.

이게 **API** 입니다. 우리 코드에서는 FastAPI라는 창구를 씁니다.

그러면 창구 담당자가 하는 일은 뭘까요? 크게 네 가지입니다.

- **새 직원 등록** — 서랍에 새 자료 넣기
- **직원 정보 조회** — 서랍에서 자료 꺼내오기
- **정보 수정** — 서랍 안 자료 고치기
- **퇴사자 삭제** — 서랍에서 자료 꺼내 버리기

이 네 가지를 묶어서 **CRUD** 라고 부릅니다. Create(만들기), Read(읽기), Update(수정), Delete(삭제). 이름은 어렵지만 하는 일은 간단합니다. 장부에 할 수 있는 네 가지 일이에요.

여기서 하나 더. 창구 담당자가 가져다준 자료가 "이름: 홍길동, 부서: 영업팀" 같은 형식이어야 나중에 읽기 편하겠죠. 아무 형식이나 마구 적어오면 해석하기 어려우니까요. 그래서 자료를 주고받을 때 형식을 미리 정해두는데, 이걸 **스키마(Schema)** 라고 합니다. 우리 코드에서는 Pydantic이 이 형식을 담당합니다.

정리하면 이렇습니다:

- 서랍장 → PostgreSQL (데이터를 저장하는 곳)
- 창구 담당자 → FastAPI (데이터를 주고받는 창구)
- 창구에서 하는 네 가지 일 → CRUD
- 자료 양식 → Pydantic 스키마

### 엑셀 파일의 한계

팀장 요청을 받고 기존 자료를 찾아봤습니다.

직원 명단은 엑셀 파일이었습니다. `직원명단_최종.xlsx`. 열어보니 컬럼이 제각각이고, 어떤 줄은 직급이 빠져 있고, 사번이 중복된 행도 보였습니다. "EMP003" 이 두 명이에요.

연차 기록은 또 다른 파일이었습니다. `연차현황_2025_v3.xlsx`. 직원 이름으로 연결되어 있는데, 인사팀에서 개명한 직원 이름을 아직 안 바꿔서 연결이 끊겨 있었습니다.

매출 자료는 팀별로 따로 관리하고 있었습니다. 개발팀 파일, 영업팀 파일, 마케팅팀 파일이 각각 있는데, 집계 기준도 달랐습니다.

이 자료들을 AI 비서에게 보여줄 수 있을까요? 물론 보여줄 수는 있습니다. 그런데 "사번 EMP003은 누구야?"라고 물으면 두 명이 나오고, "연차 며칠 남았어?"라고 물으면 연결이 끊긴 데이터를 봐야 하니 답을 못 합니다.

비서가 제대로 일하려면, 먼저 장부가 정리되어 있어야 합니다.

그래서 이번 챕터에서 할 일은 명확합니다. 흩어진 엑셀 자료를 **제대로 된 데이터베이스** 로 옮기고, API 창구를 세워서 "직원 정보 줘", "매출 보여줘" 같은 요청에 깔끔하게 응답하도록 만드는 거예요.

### v0.1을 만드는 과정

첫 번째로 한 일은 데이터베이스를 띄우는 것이었습니다.

PostgreSQL을 설치하고 설정하고... 하면 시간이 꽤 걸립니다. 대신 Docker를 씁니다. Docker는 마치 IKEA 가구처럼, 조립 설명서(docker-compose.yml 파일)대로 명령 한 줄만 치면 데이터베이스가 뚝딱 생겨납니다. 나중에 필요 없으면 다시 내리면 되고, 설치 찌꺼기도 없어요.

터미널에서 `docker-compose up -d` 한 줄이면 PostgreSQL 데이터베이스가 실행됩니다. 미리 써둔 테이블 구조와 샘플 데이터도 함께 들어갑니다. 직원 5명, 연차 기록, 매출 내역 10건이 자동으로 채워져 있어요.

두 번째로 한 일은 FastAPI 서버를 실행하는 것이었습니다. `python run.py` 한 줄이면 끝입니다.

브라우저에서 `http://localhost:8000` 을 열면 자동으로 관리자 대시보드로 이동합니다. 여기서 뭘 볼 수 있을까요?

- 직원 수 / 연차 건수 / 매출 건수 / 이번 달 총 매출 — 한눈에 보이는 통계 카드
- 최근 매출 5건 목록
- 직원 관리 메뉴 — 조회, 등록, 수정, 삭제
- 연차 현황 메뉴 — 직원별 잔여 연차
- 매출 관리 메뉴 — 부서별 매출, 기간 조회

그리고 `http://localhost:8000/docs` 를 열면 Swagger UI가 뜹니다. 이건 API 창구 목록을 예쁘게 보여주는 화면이에요. 버튼 하나로 "개발팀 매출 조회", "EMP001 직원 정보 조회" 같은 요청을 직접 테스트해볼 수 있습니다.

### 작은 문제 하나

처음에는 사번 중복 문제가 있었습니다.

기존 엑셀 데이터를 그대로 옮겼더니, "EMP003" 두 명이 그대로 들어가버렸습니다. 에러 없이 통과됐어요.

"이러면 안 되는데..."

해결책은 간단했습니다. 데이터베이스 테이블을 만들 때 사번 컬럼에 "이 칸에는 같은 값이 두 번 들어올 수 없다"는 규칙을 걸면 됩니다. 중복 사번으로 등록하려 하면 데이터베이스가 자동으로 막아줘요.

이제 "EMP003"을 두 번 등록하려 하면 오류가 납니다. API에서는 "사번 EMP003이 이미 존재합니다"라는 메시지로 응답하고요.

엑셀로 관리할 때는 중복을 사람이 직접 확인해야 했습니다. 데이터베이스는 이런 규칙을 알아서 지켜줍니다.

### 브라우저에 뜬 대시보드

두 가지 명령을 실행하고 브라우저를 열었습니다.

직원 목록 화면에는 5명의 이름, 사번, 부서, 직급, 입사일이 깔끔하게 줄 서 있습니다. 검색창에 "개발"을 치면 개발팀 직원만 필터링됩니다.

연차 현황에는 각 직원의 총 연차, 사용 연차, 잔여 연차가 한눈에 보입니다. 잔여 연차는 데이터베이스가 자동으로 계산해줘요. 사람이 직접 뺄셈할 필요가 없습니다.

매출 화면에는 영업팀, 개발팀, 마케팅팀의 매출 내역과 부서별 합계가 보입니다.

v0.1이 완성됐습니다. 아직 AI는 없습니다. 비서도 없습니다. 하지만 **비서가 볼 장부** 는 생겼습니다.

다음 챕터에서는 이 장부를 ChatGPT에게 보여주고, 질문에 답하게 만들어보겠습니다. 그런데 그 과정에서 생각지 못한 문제를 만나게 됩니다. ChatGPT가 장부를 읽고 이상한 답을 내놓거든요.

---

### 이것만은 기억하자

- **AI 비서를 만들기 전에, 비서가 볼 장부부터 만들자.** 데이터가 없으면 AI도 답을 못 한다.
- 데이터베이스는 서랍장, API는 창구 담당자, CRUD는 창구에서 하는 네 가지 일이다.
- 데이터베이스를 쓰면 엑셀로는 못 막던 중복, 형식 오류 같은 문제를 규칙으로 자동으로 막아준다.
- 다음 챕터에서는 이 장부를 ChatGPT에게 보여주고 질문을 던져본다. 그런데... 뭔가 이상한 일이 벌어진다.

### 설명 ① database.py — DB 연결 방식

`database.py`는 FastAPI와 PostgreSQL을 연결하는 다리 역할을 합니다. 핵심은 `get_connection()` 함수입니다.

```python
@contextmanager
def get_connection():
    conn = None
    try:
        conn = psycopg2.connect(get_dsn())   # ① DB 연결
        yield conn                            # ② 연결 객체 제공
        conn.commit()                         # ③ 성공하면 저장
    except psycopg2.OperationalError:
        if conn:
            conn.rollback()                  # ④ 실패하면 되돌리기
        raise RuntimeError("DB 연결 실패")
    finally:
        if conn:
            conn.close()                     # ⑤ 항상 연결 닫기
```

중요한 점은 `with get_connection() as conn:` 형태로 쓴다는 것입니다. `with` 블록이 끝나면 연결이 자동으로 닫힙니다. 연결을 깜빡하고 닫지 않아서 생기는 문제를 막아주는 패턴이에요.

DB 접속 정보(호스트, 사용자, 비밀번호)는 `.env` 파일에서 읽어옵니다. 코드 안에 직접 비밀번호를 적는 건 보안상 좋지 않아요.

### 설명 ② crud.py — CRUD 함수

`crud.py`는 실제로 SQL 쿼리를 실행하는 함수들이 모여 있습니다. 직원, 연차, 매출 각각에 대해 CRUD 함수가 있습니다. 직원 조회 함수 하나를 살펴봐요.

```python
def get_all_employees(conn, name_filter=None, dept_filter=None):
    conditions = []
    params = []

    if name_filter:
        conditions.append("name ILIKE %s")
        params.append(f"%{name_filter}%")   # ① 부분 일치 검색
    if dept_filter:
        conditions.append("dept ILIKE %s")
        params.append(f"%{dept_filter}%")

    where_clause = ("WHERE " + " AND ".join(conditions)) if conditions else ""
    sql = f"SELECT ... FROM employee {where_clause} ORDER BY id"

    with conn.cursor() as cur:
        cur.execute(sql, params)             # ② SQL 실행
        rows = cur.fetchall()               # ③ 결과 가져오기

    return [_row_to_employee(r) for r in rows]
```

눈여겨볼 점은 `%s` 로 파라미터를 넣는다는 것입니다. 직접 문자열을 붙이면 SQL 인젝션이라는 보안 문제가 생길 수 있어요. psycopg2가 파라미터를 안전하게 처리해줍니다.

`ILIKE`는 대소문자 구분 없이 검색하는 PostgreSQL 기능입니다. "개발"을 검색해도, "개발팀"에 속한 직원이 나옵니다.

### 설명 ③ api.py — REST API 엔드포인트

`api.py`는 외부에서 들어오는 HTTP 요청을 받아서 crud.py 함수를 호출하고, 결과를 JSON으로 돌려주는 역할을 합니다. 직원 등록 API를 예로 볼게요.

```python
@router.post("/employees", status_code=201)
def api_create_employee(body: EmployeeCreate):
    try:
        with get_connection() as conn:
            emp = crud.create_employee(
                conn, body.emp_no, body.name,
                body.dept, body.position, body.hire_date,
            )
    except psycopg2.errors.UniqueViolation:
        raise HTTPException(
            status_code=409,
            detail=f"사번 '{body.emp_no}'이(가) 이미 존재합니다."
        )
    return EmployeeResponse(...)
```

요청이 오면 → DB에 저장 → 성공하면 JSON 반환, 사번 중복이면 409 오류 반환. 이게 API의 흐름입니다.

HTTP 상태 코드는 대화에서 표정 같은 거예요. 201은 "잘 만들었어요", 404는 "찾을 수 없어요", 409는 "충돌 났어요", 503은 "지금 서비스 안 돼요".

### 참고 — schemas.py / views.py / templates/

**schemas.py**: FastAPI 요청/응답 형식을 Pydantic으로 정의합니다. 예를 들어 직원 등록 요청에는 사번, 이름, 부서, 직급, 입사일이 반드시 있어야 한다고 명시해요. 형식이 맞지 않으면 FastAPI가 자동으로 400 오류를 돌려줍니다.

**views.py + templates/**: 브라우저에서 보이는 Admin 화면을 담당합니다. FastAPI가 HTML 파일(Jinja2 템플릿)을 직접 렌더링해서 돌려줍니다. API(`/api/*`)와 UI(`/admin/*`)가 같은 서버에서 동시에 동작하는 구조예요.

### 데이터베이스 구조

이번 챕터에서 만든 테이블 3개의 관계를 간단히 보면:

```
employee (직원)
  ├── id, emp_no(사번-UNIQUE), name, dept, position, hire_date
  │
leave_balance (연차) ── employee와 연결
  ├── id, employee_id, year, total_days, used_days
  └── remaining_days (= total_days - used_days, 자동 계산)
  │
sales (매출)
  └── id, dept, sale_date, amount, item
```

`remaining_days`는 PostgreSQL의 **Generated Column** 기능으로, `total_days - used_days`를 자동으로 계산합니다. 연차를 사용할 때마다 잔여 연차를 직접 빼줄 필요가 없어요. DB가 알아서 해줍니다.

### 더 알아보기

- **REST API 설계 원칙**: GET(조회), POST(생성), PATCH(부분 수정), DELETE(삭제) 메서드를 구분하는 이유와 HTTP 상태 코드 의미 → FastAPI 공식 문서 참조
- **Pydantic 검증**: `Field(ge=0)` (0 이상), `max_length=50` 같은 검증 규칙으로 잘못된 데이터가 들어오는 것을 막을 수 있습니다
- **Docker 기본**: `docker-compose up -d`로 시작, `docker-compose down`으로 종료, `docker-compose logs`로 로그 확인
- **PostgreSQL UNIQUE 제약**: 테이블 컬럼에 `UNIQUE`를 걸면 중복 데이터 삽입을 DB 수준에서 막습니다. 애플리케이션 코드가 아니라 데이터베이스가 직접 보호하기 때문에, 어떤 경로로 데이터가 들어오든 중복이 생기지 않습니다
