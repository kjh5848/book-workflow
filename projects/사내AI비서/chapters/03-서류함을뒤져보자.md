# Ch.3: 서류함을 뒤져보자 — v0.3

> 이번 버전: v0.2 → v0.3
> 한 줄 요약: 사내 문서를 잘게 자르고, 분류 번호를 붙여서, 검색 가능한 도서관에 넣었다.
> 핵심 개념: 문서 파싱, 청킹, 임베딩, 벡터DB

---

## 이야기 파트

### 서류함 앞에 서다

v0.2에서 RAG라는 방향이 잡혔습니다. "문서 전체를 넣지 말고, 질문에 맞는 조각만 찾아서 LLM에게 건네주자." 개념은 이해했습니다.

그런데 막상 시작하려니 첫 번째 질문이 떠올랐습니다.

"조각을 찾으려면, 먼저 조각이 있어야 하는 거 아닌가?"

맞습니다. v0.2에서 RAG를 돌릴 때는 사내 문서 텍스트를 미리 준비해둔 상태였습니다. 그 텍스트가 어디서 왔는지는 신경 쓰지 않았어요. 그런데 실제 상황에서는 사내 문서가 깔끔한 텍스트로 존재하지 않습니다. HR팀은 PDF로 취업규칙을 보내고, 보안팀은 Word로 보안규정을 공유하고, 재무팀은 Excel로 매출 현황을 정리합니다.

회사의 서류함을 열어보니, 안에는 온갖 형태의 서류가 뒤섞여 있었습니다.

---

### 서류함, 색인 카드, 분류 번호, 도서관

여기서 비유를 하나 꺼내보겠습니다. 이번 챕터에서 하는 일 전체를 한 장면으로 그려볼게요.

회사에 서류함이 하나 있습니다. 서류함 안에는 취업규칙, 보안규정, 매출 보고서, 예산 기안서 같은 문서가 가득 들어 있어요. 누군가 "연차 규정이 어떻게 돼?"라고 물으면, 서류함을 열고 관련 문서를 찾아야 합니다.

그런데 이 서류함에는 문제가 있습니다.

첫째, **서류 형식이 제각각**입니다. 어떤 건 PDF로 인쇄된 공문이고, 어떤 건 Word 파일이고, 어떤 건 Excel 표입니다. 사람 눈으로는 다 읽을 수 있지만, 컴퓨터가 읽으려면 형식마다 다른 방법으로 텍스트를 꺼내야 합니다. 이게 **파싱(Parsing)** 입니다. 서류함에서 문서를 하나씩 꺼내서, 종이 위의 글자를 텍스트로 옮겨 적는 과정이에요.

둘째, **문서가 너무 깁니다**. 취업규칙이 20페이지짜리인데, "연차 규정"은 3페이지에 있어요. 나머지 17페이지는 이 질문과 관계가 없습니다. 문서 통째로 보관하면 나중에 검색할 때 효율이 떨어집니다. 그래서 문서를 작은 조각으로 자릅니다. 마치 책의 내용을 색인 카드에 한 장씩 옮겨 적는 것처럼요. 이게 **청킹(Chunking)** 입니다. 색인 카드 한 장에 500자 정도를 적고, 카드 사이에 약간의 겹침을 둬서 내용이 끊기지 않게 합니다.

셋째, **비슷한 내용을 찾을 방법이 필요합니다**. 색인 카드가 100장이 있을 때, "연차 규정"이라는 질문과 관련된 카드를 어떻게 찾을까요? 사람이라면 카드를 한 장씩 읽어보겠지만, 컴퓨터는 다른 방식을 씁니다. 각 카드의 내용을 숫자 목록(벡터)으로 변환합니다. 비슷한 의미의 텍스트는 비슷한 숫자 목록을 가지게 됩니다. 마치 도서관에서 비슷한 주제의 책에 비슷한 분류 번호를 붙이는 것처럼요. 이게 **임베딩(Embedding)** 입니다.

마지막으로, 분류 번호가 붙은 색인 카드들을 정리해서 보관할 장소가 필요합니다. 일반 데이터베이스는 "이름이 김민준인 사람"처럼 정확한 조건으로 찾습니다. 하지만 우리는 "연차 관련 내용"처럼 **의미가 비슷한 것**을 찾아야 합니다. 이런 검색에 특화된 저장소가 **벡터DB** 입니다. 이 책에서는 ChromaDB를 씁니다. 분류 번호가 붙은 카드들을 체계적으로 보관하고, 질문이 들어오면 가장 비슷한 카드를 꺼내주는 도서관이에요.

정리하면 이렇습니다.

- **서류함** = `data/docs/` 폴더 (사내 문서 6종)
- **서류함 열기** = 문서 파싱 (PDF, Word, Excel → 텍스트)
- **색인 카드 만들기** = 청킹 (500자씩 자르기)
- **분류 번호 붙이기** = 임베딩 (텍스트 → 768차원 벡터)
- **도서관에 보관** = ChromaDB에 저장

이 네 단계를 거치면, "연차 규정"이라고 물었을 때 관련 색인 카드가 나옵니다.

---

### 서류함을 열었더니

자, 시작합니다. 먼저 서류함을 열어봅시다.

`data/docs/` 폴더에 사내 문서 6개가 들어 있습니다. HR팀의 취업규칙과 정보보안서약서(PDF), 보안팀의 보안규정(Word), 재무팀의 매출 현황과 예산 기안서(Excel), 그리고 운영팀의 런칭 전략(PDF). 형식도 제각각이고 내용도 제각각입니다.

컴퓨터에게 이 문서들을 읽으라고 시킵니다. pypdf로 PDF를 열고, python-docx로 Word를 열고, openpyxl로 Excel을 엽니다. 형식별로 다른 도구가 필요한 거죠. 마치 서류함에서 PDF 공문은 복사기로 스캔하고, Word 파일은 프린터로 출력하고, Excel 표는 따로 옮겨 적는 것처럼.

그런데 여기서 첫 번째 난관을 만났습니다.

PDF를 열었는데 텍스트가 이상합니다. 사람이 보기에는 멀쩡한 문서인데, 추출해보니 글자 사이에 공백이 잔뜩 들어가 있거나, 표의 열이 뒤섞여 나옵니다. 이미지로 된 PDF는 아예 빈 텍스트가 나왔습니다 — 텍스트가 아니라 그림이니까요.

"이게 뭐야? 사람 눈에는 똑같은 문서인데, 컴퓨터가 읽으면 이 모양인 거야?"

네, 그렇습니다. PDF는 원래 "화면에 보이는 대로 인쇄하는" 형식이라, 텍스트 순서나 구조를 보장하지 않습니다. Word는 그나마 단락과 표 구조가 있어서 깔끔하게 나옵니다. Excel은 셀 단위로 읽으면 되니까 비교적 수월하고요.

완벽하지는 않지만, 충분히 읽을 수 있는 수준의 텍스트가 나왔습니다. 이미지 기반 PDF 문제는 나중에 Ch.7에서 OCR로 해결합니다. 지금은 텍스트가 있는 문서를 대상으로 진행합니다.

---

### 색인 카드를 만들다

문서에서 텍스트를 꺼냈으니, 이제 잘게 잘라야 합니다.

여기서 한 가지 의문이 들 수 있습니다. "왜 자르는 거야? 문서 하나를 통째로 저장하면 안 돼?"

v0.2에서 이미 경험한 문제입니다. 취업규칙 전체를 LLM에게 보내면 토큰이 폭발합니다. 그리고 문서 전체에서 검색하면, 연차 규정과 관계없는 내용이 95%를 차지해서 검색 정확도가 떨어집니다.

그래서 문서를 500자씩 자릅니다. 500자면 한국어로 약 한 문단 정도입니다. 한 색인 카드에 한 문단을 담는 거죠.

그런데 딱 500자에서 잘라버리면 문장이 중간에 끊깁니다. "연차는 입사 1년 후 15일이 부여되며, 사용하지 않은 연차는" — 여기서 잘리면 다음 카드에 있는 "다음 해로 이월되지 않습니다"를 놓칩니다.

이걸 방지하려고 **오버랩(overlap)** 을 둡니다. 앞 카드의 마지막 100자를 다음 카드의 첫 100자에 겹치게 적는 거예요. 이러면 경계에서 끊긴 문맥이 다음 카드에서 이어집니다.

500자 청킹, 100자 오버랩. 이게 이 책에서 쓰는 기본 설정입니다.

6개 문서를 이렇게 잘랐더니, 총 수십 장의 색인 카드가 만들어졌습니다. 각 카드에는 "어떤 파일에서 왔는지, 몇 페이지인지"라는 꼬리표(메타데이터)도 붙어 있습니다. 나중에 검색 결과에서 출처를 보여줄 때 이 꼬리표를 씁니다.

---

### 분류 번호를 붙이다

색인 카드가 만들어졌습니다. 이제 핵심 질문입니다.

"연차 사용 규정"이라는 질문이 들어왔을 때, 수십 장의 카드 중에서 관련된 걸 어떻게 찾지?

사람이라면 카드를 하나씩 읽어보면 됩니다. 하지만 카드가 100장, 1000장, 10000장이면요?

여기서 **임베딩** 이 등장합니다.

다시 도서관 비유를 써볼게요. 도서관에서 책을 찾을 때, 책장을 하나씩 둘러보지 않잖아요. 분류 번호를 봅니다. "300번대는 사회과학, 340번대는 법학, 344번대는 노동법." 비슷한 주제의 책은 비슷한 번호를 가지고 있어서, 번호만 비교하면 관련 책을 빠르게 찾을 수 있습니다.

임베딩도 같은 원리입니다. "연차 사용 규정"이라는 텍스트를 숫자 목록으로 바꿉니다. 768개의 숫자로 이루어진 목록이에요. "연차 휴가 신청 방법"이라는 텍스트도 768개 숫자로 바꾸면, 두 목록이 비슷한 값을 가집니다. 의미가 비슷하니까요.

반면에 "이번 달 매출 현황"을 숫자로 바꾸면, "연차 사용 규정"과는 전혀 다른 값이 나옵니다. 의미가 다르니까요.

이 변환을 해주는 게 **임베딩 모델**입니다. 이 책에서는 `ko-sroberta-multitask`라는 한국어 특화 모델을 씁니다. 한국어 문장의 의미를 잘 잡아내도록 학습된 모델이에요.

색인 카드 수십 장에 전부 분류 번호(768차원 벡터)를 붙였습니다. 이제 질문이 들어오면, 질문에도 분류 번호를 붙이고, 가장 비슷한 번호를 가진 카드를 찾으면 됩니다.

---

### 도서관에 넣다

분류 번호까지 붙였으면, 이 카드들을 잘 보관해야 합니다. 여기서 **ChromaDB** 가 등장합니다.

ChromaDB는 벡터 전용 데이터베이스입니다. 일반 데이터베이스가 "이름 = 김민준"처럼 정확한 조건으로 검색한다면, ChromaDB는 "이 벡터와 가장 비슷한 벡터를 찾아줘"라는 방식으로 검색합니다.

카드에 붙은 분류 번호(벡터)와 텍스트, 메타데이터(출처 파일, 페이지 번호)를 함께 저장합니다. 그리고 검색할 때는 질문의 벡터를 넣으면, 코사인 유사도라는 계산법으로 가장 비슷한 카드를 골라줍니다.

ChromaDB를 선택한 이유는 간단합니다. 설치가 쉽고, Python 코드 몇 줄이면 바로 쓸 수 있고, 파일 형태로 로컬에 저장되니까 별도의 서버가 필요 없습니다. 사내 AI 비서를 처음 만들어보는 단계에서는 이만한 게 없어요.

---

### "연차 규정"을 검색해보다

자, 모든 준비가 끝났습니다. 파이프라인을 돌려봅시다.

1. 문서 6개를 파싱해서 텍스트를 추출했습니다.
2. 텍스트를 500자씩 잘라서 색인 카드를 만들었습니다.
3. 각 카드에 768차원 벡터(분류 번호)를 붙였습니다.
4. 벡터와 텍스트를 ChromaDB에 저장했습니다.

이제 CLI 검색 도구를 실행합니다.

> "연차 사용 규정"

결과가 나왔습니다. 유사도 85%로 가장 높은 카드가 떠오릅니다. HR팀의 취업규칙에서 가져온 카드인데, "연차는 입사 1년 후 15일이 부여되며..."라는 내용이 담겨 있습니다. 출처도 표시됩니다: `HR_취업규칙_v1.0.pdf`, 3페이지.

두 번째 카드는 유사도 78%로 연차 신청 절차에 대한 내용이었고, 세 번째 카드는 정보보안서약서의 보안 교육 이수 관련 내용이었습니다 — 약간 빗나갔지만, 유사도가 낮으니 참고만 하면 됩니다.

"비밀번호 변경 주기"를 검색하면 보안규정의 관련 문단이 나오고, "상반기 매출"을 검색하면 재무팀의 매출 현황표 내용이 나옵니다.

서류함이 도서관이 됐습니다.

---

### 아직 "답변"은 아니다

그런데 여기서 한 가지. CLI에서 나오는 건 **텍스트 조각**입니다. 답변이 아닙니다.

"연차 사용 규정"을 검색하면 "연차는 입사 1년 후 15일이 부여되며, 3년 이상 근속 시 매 2년마다 1일 가산..."이라는 원문이 나옵니다. 이걸 사람이 읽고 해석해야 합니다.

"내 연차가 며칠 남았어?"라고 물으면, 검색 결과에 연차 규정 텍스트가 나오지만, "당신의 잔여 연차는 7일입니다"라고 대답해주지는 않습니다. 그건 LLM이 해야 할 일이에요.

v0.3은 RAG의 **R(Retrieval)** — 검색 부분을 만든 겁니다. **G(Generation)** — LLM이 검색 결과를 보고 자연어로 답하는 부분은 다음 챕터에서 다룹니다.

지금 단계에서의 느낌은 이겁니다. 도서관에서 관련 책을 찾아왔는데, 그 책을 읽고 요약해줄 사람이 아직 없는 상태. 책상 위에 관련 자료가 올라와 있는 것까지는 성공입니다. 이 자료를 읽고 친절하게 답해주는 비서는 Ch.4에서 만듭니다.

---

### 이것만은 기억하자

- **사내 문서는 형식이 제각각이다.** PDF, Word, Excel을 각각 다른 도구로 파싱해야 한다.
- **문서를 통째로 저장하면 검색이 부정확하다.** 500자 단위로 잘라서 색인 카드(청크)를 만들어야 한다.
- **임베딩은 텍스트에 분류 번호를 붙이는 것이다.** 비슷한 의미의 텍스트는 비슷한 숫자를 가진다. 이 숫자로 "의미가 비슷한 것"을 찾는다.
- **ChromaDB는 벡터 전용 도서관이다.** 분류 번호로 가장 비슷한 카드를 빠르게 찾아준다.
- 다음 챕터에서는 이 검색 결과에 LLM을 연결한다. 텍스트 조각이 아니라, 자연어 답변이 나오기 시작한다. 비서가 드디어 말을 하기 시작한다.

| 서류함에서 문서 꺼내 글자를 텍스트로 옮기기 | 파싱 (Parsing) | 구조화된 파일(PDF, DOCX, XLSX)에서 텍스트 데이터를 추출하는 과정. 형식마다 다른 라이브러리(pypdf, python-docx, openpyxl)를 사용한다. |
| 책 내용을 색인 카드에 한 장씩 옮겨 적기 | 청킹 (Chunking) | 긴 텍스트를 고정 크기(이 책에서는 500자) 단위로 분할하는 과정. 오버랩(100자)을 두어 경계에서 문맥이 끊기는 것을 방지한다. |
| 색인 카드에 분류 번호 붙이기 | 임베딩 (Embedding) | 텍스트를 고정 길이의 실수 벡터(이 책에서는 768차원)로 변환하는 과정. 의미적으로 유사한 텍스트일수록 벡터 값이 가까워진다. |
| 한국어 분류 번호 체계 | ko-sroberta-multitask | 한국어 특화 임베딩 모델. SRoBERTa 기반으로 문장 유사도 태스크에 파인튜닝되어 한국어 문장의 의미를 768차원 벡터로 변환한다. |
| 분류 번호가 비슷한 카드 찾기 | 코사인 유사도 (Cosine Similarity) | 두 벡터 간 방향의 유사도를 측정하는 방법. 1에 가까울수록 의미가 비슷하고, 0에 가까울수록 관련이 없다. |
| 분류 번호로 책을 찾는 도서관 | 벡터DB / ChromaDB | 벡터(숫자 목록)를 저장하고, 입력 벡터와 가장 유사한 벡터를 빠르게 검색하는 데이터베이스. ChromaDB는 로컬 파일 기반으로 별도 서버 없이 사용 가능하다. |
| 카드 뒷면의 꼬리표 (어떤 파일, 몇 페이지) | 메타데이터 (Metadata) | 청크에 부착되는 부가 정보. 원본 파일명, 페이지 번호, 부서명 등을 포함하여 검색 결과에서 출처 추적에 사용된다. |

### 설명 ③ chunker.py — 500자 청킹 + 오버랩

텍스트를 고정 크기로 자르는 모듈입니다.

```python
# 핵심 함수: 텍스트를 500자씩 자르되, 100자씩 겹치게
def split_text_into_chunks(text, chunk_size=500, overlap=100):
    chunks = []
    step = chunk_size - overlap   # 400자씩 이동
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunk = text[start:end].strip()
        if chunk:
            chunks.append(chunk)
        start += step
    return chunks
```

동작 원리를 그림으로 보면 이렇습니다.

```
원본 텍스트: [==========A==========][==========B==========][=======C=======]

청크 1: [==========A==========]
                    ↕ 100자 겹침
청크 2:             [==========B==========]
                                ↕ 100자 겹침
청크 3:                         [=======C=======]
```

500자 청크를 만들 때 400자씩 이동(`step = 500 - 100`)해서, 이전 청크의 끝 100자가 다음 청크의 시작 100자와 겹칩니다.

각 청크에는 메타데이터가 붙습니다.

```python
{
    "id": "hr_취업규칙_v1_0_text_p003_c0005",  # 문서ID_타입_페이지_청크번호
    "text": "연차는 입사 1년 후 15일이...",
    "metadata": {
        "doc_id": "hr_취업규칙_v1_0",
        "file_name": "HR_취업규칙_v1.0.pdf",
        "page": 3,
        "chunk_index": 5
    }
}
```

> **왜 500자인가요?** 한국어 500자는 약 1~2문단입니다. 너무 짧으면(100자) 문맥이 부족하고, 너무 길면(2000자) v0.2에서 본 것처럼 관련 없는 내용이 섞입니다. 500자는 "하나의 주제에 대한 충분한 문맥"과 "검색 정확도"의 균형점입니다. Ch.7에서 다른 크기와 비교 실험을 합니다.

### 설명 ④ store.py — 임베딩 + ChromaDB 저장

임베딩 모델로 텍스트를 벡터로 바꾸고, ChromaDB에 저장하는 모듈입니다.

```python
# 1. 임베딩 모델 로드
model = SentenceTransformer("jhgan/ko-sroberta-multitask")
# → 768차원 벡터를 반환하는 한국어 특화 모델

# 2. 텍스트를 벡터로 변환 (배치 단위)
embeddings = model.encode(texts, normalize_embeddings=True)
# → 62개 청크 × 768차원 = 62개 벡터

# 3. ChromaDB에 저장
client = chromadb.PersistentClient(path="./data/chroma_db")
collection = client.get_or_create_collection(
    name="metacoding_documents",
    metadata={"hnsw:space": "cosine"}   # 코사인 유사도 사용
)
collection.upsert(ids=ids, documents=texts,
                  embeddings=embeddings, metadatas=metas)
```

**임베딩 과정**을 좀 더 풀어보면:

1. `"연차는 입사 1년 후 15일이 부여되며"` 라는 텍스트가 들어옵니다.
2. ko-sroberta 모델이 이 문장의 의미를 분석합니다.
3. `[0.0234, -0.1567, 0.0891, ...]` 768개의 실수로 변환합니다.
4. 이 숫자 목록이 이 청크의 "분류 번호"입니다.

**검색 과정**:

```python
def search_chroma(query, top_k=5):
    # 질문도 같은 모델로 벡터 변환
    query_embedding = model.encode([query])
    # ChromaDB에서 가장 가까운 벡터 k개 찾기
    results = collection.query(
        query_embeddings=query_embedding,
        n_results=top_k
    )
    return results
```

질문 벡터와 저장된 벡터를 코사인 유사도로 비교해서, 가장 가까운 k개를 돌려줍니다.

> **upsert란?**: update + insert의 합성어입니다. 같은 ID의 청크가 이미 있으면 덮어쓰고, 없으면 새로 넣습니다. 파이프라인을 다시 돌려도 데이터가 중복되지 않습니다.

> **배치 처리**: 62개 청크를 한 번에 임베딩하면 메모리가 부족할 수 있어서, 64개씩 나눠서 처리합니다(`BATCH_SIZE = 64`). 실제로는 청크 수가 수천 개가 될 수 있으니 배치 처리가 필수입니다.

### 더 알아보기

- **임베딩 모델은 왜 ko-sroberta인가요?**: 이 모델은 한국어 문장 유사도 태스크에 특화돼 있습니다. 영어 모델(예: all-MiniLM)을 쓰면 한국어 의미 파악이 부정확합니다. ko-sroberta는 한국어 문장 쌍을 대규모로 학습해서, "연차 사용 규정"과 "휴가 신청 방법"이 비슷한 의미라는 걸 잘 잡아냅니다.
- **768차원이라는 숫자가 중요한가요?**: 차원 수는 모델마다 다릅니다. 768은 이 모델이 텍스트의 의미를 768개의 특성으로 표현한다는 뜻입니다. 차원이 높을수록 정밀하지만 저장 공간과 계산 비용이 늘어납니다. 768차원은 정확도와 효율의 균형점으로 널리 쓰이는 크기입니다.
- **ChromaDB 말고 다른 벡터DB는 없나요?**: Pinecone(클라우드), Weaviate(자체 호스팅), Milvus(대규모) 등이 있습니다. ChromaDB는 로컬 파일 기반이라 설치와 사용이 가장 간단합니다. 프로덕션에서 수백만 건을 처리해야 한다면 다른 벡터DB를 고려해야 하지만, 학습 단계에서는 ChromaDB가 충분합니다.
- **청킹 방식이 Fixed-size뿐인가요?**: 아닙니다. 문장 단위, 문단 단위, 의미 단위(Semantic Chunking) 등 다양한 방식이 있습니다. Fixed-size는 가장 단순하고 예측 가능합니다. Ch.7에서 Semantic Chunking과 비교 실험을 합니다.
