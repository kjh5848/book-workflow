# Ch.6: 비서에게 매뉴얼을 주자 — v0.6

> 이번 버전: v0.5 → v0.6
> 한 줄 요약: 일은 하는데 체계가 없던 비서에게, 메모 패드·업무 일지·근무 규정서를 줬더니 제대로 된 직원이 됐다.
> 핵심 개념: 응답 캐시(TTL), 구조화 로깅, 환경 변수 설정

---

## 이야기 파트

### 비서가 일은 하는데…

Ch.5가 끝나고 팀장에게 비서를 넘겨줬습니다. 문서 질문도 되고, DB 조회도 되고, 복합 질문도 척척 답하는 그 비서를요.

처음 며칠은 팀장이 꽤 만족스러워했습니다. "연차 며칠 남았어?" 물으면 바로 숫자가 나오고, "보안 규정 알려줘" 하면 출처까지 달린 답변이 나오니까요.

그런데 일주일쯤 지나서 팀장이 다시 불렀습니다.

> "이거 매일 아침에 '이번 달 매출 얼마야?' 물어보는데, 왜 매번 5초씩 걸려요?"

맞는 말입니다. 팀장이 아침마다 같은 질문을 합니다. 비서는 매번 DB를 조회하고, LLM을 불러서 답변을 생성하고, 5초를 씁니다. 어제 아침과 오늘 아침 사이에 매출 숫자가 바뀌었을까요? 아마 안 바뀌었을 겁니다. 똑같은 답변을 매번 처음부터 만들고 있는 거예요.

또 다른 날, 이런 일이 있었습니다.

> "아까 비서한테 뭔가 물어봤는데 엉뚱한 답변이 나왔어요. 무슨 질문했는지, 어떤 과정을 거쳤는지 알 수 있어요?"

알 수 없었습니다. 비서가 무슨 도구를 썼는지, 몇 번 LLM을 호출했는지, 왜 그런 답변이 나왔는지 — 아무 기록이 없었어요. 뒤를 돌아봤더니 아무것도 없는 거예요.

그리고 마지막 문제.

> "이거 Ollama 말고 OpenAI로 바꿔보고 싶은데, 어디 수정하면 돼요?"

코드를 뒤졌습니다. `agent_config.py`에도 있고, `.env` 파일도 봐야 하고… 어디를 어떻게 건드려야 하는지 한눈에 보이지 않았습니다. 설정이 코드 여기저기에 흩어져 있었거든요.

비서가 일은 합니다. 그런데 **체계**가 없어요.

---

### 진짜 비서라면

실제 회사에서 신입 비서를 뽑으면 어떻게 되는지 생각해봅시다.

처음엔 업무 능력만 보고 뽑습니다. 일 잘하면 되지, 뭐가 더 필요하겠어요?

그런데 시간이 지나면서 문제가 생깁니다.

자주 받는 질문이 있습니다. "김 부장님 전화번호요?" 하루에 다섯 번씩 받는 질문이에요. 매번 연락처 앱을 열어서 찾습니다. 5초. 그냥 포스트잇에 적어서 모니터 앞에 붙여두면 1초면 되는데 말이죠.

그리고 기록이 없습니다. 어제 누가 전화 왔었는지, 어떤 서류를 어디로 보냈는지 — 다 머릿속에만 있어요. 나중에 "그거 어떻게 됐지?" 물어보면 "아, 그게…" 하면서 기억을 더듬습니다.

마지막으로 설정 문제가 있습니다. 비서가 아프면 어떻게 하죠? 대신할 사람이 왔을 때 "우리 회사 이메일 비밀번호는 이거고, 회의실 예약은 이 시스템에서 하고, 점심 배달은 이 업체에…" 하나씩 알려줘야 합니다. 다음 번에 또 바뀔 때마다 처음부터 다시.

이걸 해결하는 방법이 세 가지입니다.

**첫째, 메모 패드**. 자주 받는 질문과 답변을 적어서 책상 앞에 붙여둔다. "이번 달 매출요? 잠깐요…" 대신, 메모를 보고 바로 답한다. 단, 일주일 지난 메모는 버린다 — 숫자가 달라졌을 수 있으니까.

**둘째, 업무 일지**. 퇴근 전에 오늘 한 일을 적는다. "이서연 대리님 연차 조회 3회, 매출 보고 2회, 보안 규정 안내 1회. 평균 응답 시간 2.3초." 나중에 "그때 왜 그런 답이 나왔지?" 궁금하면 일지를 펼쳐보면 된다.

**셋째, 근무 규정서**. "LLM은 OpenAI를 쓴다. API 키는 이 칸에 적혀 있다. 메모는 1시간치만 보관한다. 로그는 파일로 저장한다." 코드는 건드리지 않고, 규정서만 바꾸면 된다.

v0.6에서 비서에게 이 세 가지를 줍니다.

---

### 포스트잇을 붙이다

첫 번째 문제부터 해결해봅시다. 팀장이 매일 아침 같은 질문을 합니다. 매번 DB를 조회하고 LLM을 돌리는 게 낭비입니다.

생각해보면, 같은 질문에는 같은 답이 나옵니다 — 적어도 한동안은요. "이번 달 매출 얼마야?" 오전 9시에 물어본 답변은 오전 10시에도 유효합니다. 그런데 이틀 전 답변은 이미 달라졌을 수 있어요.

그래서 **TTL(유효 시간)** 이라는 개념이 필요합니다. 포스트잇에 유통기한을 적어두는 거예요.

비서가 질문을 받으면 이런 순서로 일합니다:

1. 이 질문, 포스트잇에 있나 확인한다
2. 있으면 → 포스트잇을 읽고 바로 답한다 (0.1초)
3. 없으면 → 직접 찾아서 답하고, 포스트잇에 새로 적어둔다
4. 포스트잇이 1시간이 넘었으면 → 버리고 새로 찾는다

"이번 달 매출 얼마야?"를 아침 9시에 처음 물어봤습니다. 비서가 DB를 뒤져서 답하고, 포스트잇에 적었습니다. 10시에 같은 질문이 들어왔습니다. 비서는 포스트잇을 보고 0.1초 만에 답합니다. 1시간 뒤인 11시에 같은 질문이 들어왔습니다. 포스트잇이 만료됐으니 다시 DB를 조회합니다.

이게 **캐시(Cache)** 입니다. 비서의 메모 패드.

---

### 업무 일지를 쓰다

두 번째 문제입니다. 비서가 뭘 했는지 기록이 없어요.

어느 날 비서가 이상한 답변을 했다고 가정해봅시다. 이서연 대리의 연차를 물어봤는데, "규정에 따르면 연차는 15일입니다"라는 규정 설명을 했어요. 잔여 일수를 물어봤는데 규정을 읊은 겁니다.

왜 그런 거죠? 라우터가 질문을 잘못 분류했을까요? 도구를 잘못 선택했을까요? LLM이 이상한 답을 생성했을까요? 아무도 모릅니다. 기록이 없으니까요.

진짜 비서라면 업무 일지를 씁니다. "14:32 — 이서연 대리 연차 조회 요청. 라우터 판단: structured. 연차조회 도구 실행. 결과: 12일 잔여. 답변 생성 완료. 소요: 2.1초." 이게 있으면 나중에 들여다볼 수 있어요.

v0.6의 비서는 이걸 자동으로 합니다. 모든 질문-답변 사이클을 로그(log)로 기록해요. 그냥 기록하는 게 아니라, 구조화된 형식으로요.

두 가지 선택지가 있습니다.

**일반 텍스트 일지**: "2026-01-15 09:30:12 INFO agent_config 질문 수신: 이번 달 매출 얼마야" 같은 형식. 사람이 읽기는 쉬운데, 나중에 검색하거나 집계하기 어렵습니다.

**JSON 일지**: `{"timestamp": "2026-01-15T09:30:12Z", "level": "INFO", "message": "질문 수신", "route": "db", "latency_ms": 2100}` 같은 형식. 사람이 읽기는 조금 딱딱한데, 나중에 "오늘 DB 조회가 몇 번이었지?" 같은 질문에 바로 집계할 수 있습니다.

v0.6에서는 환경 변수로 선택할 수 있습니다. 개발할 때는 텍스트로 편하게 보고, 나중에 운영할 때는 JSON으로 남겨두는 거예요.

그리고 **Langfuse** 라는 선택 사항도 있습니다. 이건 외부 서비스인데, LLM 호출 이력을 대시보드에서 시각적으로 볼 수 있게 해줍니다. "이 질문에 어떤 프롬프트가 들어갔는지", "평균 응답 시간이 얼마인지" 같은 걸 차트로 볼 수 있어요. 필수는 아닙니다. 키를 설정하면 자동으로 켜지고, 안 설정하면 조용히 꺼집니다.

### 근무 규정서를 만들다

세 번째 문제입니다. 설정이 코드 안에 들어가 있어요.

"OpenAI에서 Ollama로 바꾸고 싶어요." → 코드를 열어야 합니다.
"캐시 시간을 1시간에서 30분으로 줄이고 싶어요." → 코드를 열어야 합니다.
"로그를 파일로도 저장하고 싶어요." → 코드를 열어야 합니다.

코드는 "어떻게 동작하는지"를 담는 곳이어야 합니다. "어떤 값으로 동작하는지"는 코드 밖에 있어야 해요. 그 값들을 담는 파일이 **.env 파일** 입니다.

비서 근무 규정서라고 생각하세요. 근무 규정서에는 이런 게 적혀 있어요.

> LLM 공급자: Ollama (로컬 서버 사용)
> 서버 주소: http://localhost:11434
> 사용 모델: deepseek-r1:8b
> 메모 유효 시간: 3600초 (1시간)
> 메모 최대 개수: 1000개
> 업무 일지 형식: 일반 텍스트
> 업무 일지 파일: (파일 저장 안 함, 화면에만 표시)

이 규정서를 바꾸면 비서가 그에 따라 달라집니다. 코드는 손대지 않고요.

OpenAI로 바꾸고 싶으면? 규정서에서 "LLM_PROVIDER"를 "openai"로 바꾸고 API 키를 적으면 됩니다. 5초면 됩니다. Ollama로 돌아오고 싶으면? 다시 "ollama"로 바꾸면 됩니다.

---

### 모두 조립하다

메모 패드(캐시)도 생겼고, 업무 일지(로깅)도 생겼고, 근무 규정서(환경 변수)도 만들었습니다. 이 세 가지를 v0.5의 비서(라우터 + 도구 + 에이전트)에 붙여서 조립하는 역할을 **agent_config.py** 가 합니다.

건물을 짓는다고 생각해봅시다. 벽돌(LLM), 배관(라우터), 인테리어(에이전트), 보안 시스템(캐시), 관리사무소(모니터링) — 각각의 부품을 하나의 건물로 엮어주는 게 설계도입니다. `agent_config.py` 가 그 설계도예요.

사용자가 `main.py`를 실행하면 이런 순서로 비서가 준비됩니다:

1. 근무 규정서(.env) 읽기 → "Ollama 사용하는구나, 모델은 deepseek-r1:8b구나"
2. LLM 만들기 → Ollama에 접속
3. 라우터 만들기 → 질문 분류기 준비
4. 도구 목록 준비 → 연장통 4개 장착
5. 에이전트 조립 → 라우터 + 도구 + LLM을 AgentExecutor에 연결
6. 메모 패드(캐시) 연결
7. 업무 일지(모니터링) 연결

준비 완료. 이제 질문을 받을 수 있습니다.

---

### 실행해봤더니

v0.6을 데모 모드로 실행해봤습니다. 5개의 시나리오 질문이 순서대로 실행됩니다.

"영업팀 직원 목록을 보여줘" — 비서가 DB를 조회해서 영업팀 3명을 답합니다. 3.2초. "이서연의 휴가 잔여일이 몇 일이야?" — 장부에서 12일 잔여를 찾아줍니다. 2.8초. "이번 달 전체 매출 합계는?" — 3,680만 원. "연차 사용 규정이 어떻게 돼?" — 서류함에서 관련 문단을 검색합니다. "이서연의 휴가 잔여일과 연차 규정을 함께 알려줘" — 에이전트가 두 도구를 차례로 사용합니다.

다섯 개가 끝나고 마지막에 통계가 뜹니다. 총 호출 5건, 총 토큰 2,840개, 평균 응답 3.5초, 캐시 적중률 0%. 처음 실행이니 캐시에 아무것도 없었습니다.

같은 질문을 한 번 더 해봤습니다. "이서연의 휴가 잔여일이 몇 일이야?"

이번에는 "이전 응답을 재사용했습니다"라는 메시지가 뜨면서 **0.01초** 만에 답이 나왔습니다. 포스트잇을 보고 바로 대답한 겁니다.

---

### 이것만은 기억하자

- **캐시는 비서의 메모 패드다.** 같은 질문이 또 들어오면, DB나 LLM을 다시 부르지 않고 메모를 보고 바로 답한다. 단, 메모에는 유통기한(TTL)이 있다.
- **로그는 나중에 "왜 그랬지?"를 추적하는 유일한 수단이다.** 비서가 무엇을 했는지 기록해두지 않으면, 이상한 답변이 나왔을 때 이유를 알 수가 없다.
- **설정은 코드 밖에 있어야 한다.** .env 파일이 근무 규정서다. 코드를 건드리지 않고 규정서만 바꾸면 비서의 동작이 바뀐다.
- 다음 챕터에서는 비서의 **실력 자체를 올립니다.** 같은 질문에 더 정확한 답을 하도록 — RAG 파이프라인의 청킹, 검색, 재정렬을 튜닝합니다. 지금은 100개를 가져와서 3개를 쓰는 상황인데, 처음부터 딱 맞는 3개를 가져오게 만드는 거예요.

## 기술 파트

### 용어 정리

| 이야기 속 비유 | 진짜 용어 | 정식 정의 |
|--------------|----------|----------|
| 메모 패드 | 캐시 (Cache) | 자주 사용되는 데이터를 미리 저장해두고 빠르게 재사용하는 저장소. 이 챕터에서는 LLM 응답 결과를 메모리에 보관한다. |
| 포스트잇 유통기한 | TTL (Time-To-Live) | 캐시 항목이 유효한 시간(초). 이 시간이 지나면 자동으로 만료되어 다음 요청 시 새로 계산한다. |
| 업무 일지 | 로그 (Log) | 프로그램이 실행되는 동안 발생하는 이벤트를 시간순으로 기록한 파일 또는 출력. |
| 구조화된 업무 일지 | 구조화 로깅 (Structured Logging) | 로그를 JSON 형식처럼 파싱 가능한 구조로 기록하는 방식. 검색, 필터링, 집계가 용이하다. |
| 근무 규정서 | 환경 변수 / .env 파일 | 코드 외부에서 프로그램의 동작을 제어하는 키-값 쌍의 설정 모음. 코드를 수정하지 않고 실행 환경을 바꿀 수 있다. |
| 설계도 | AgentConfig / agent_config.py | LLM, 라우터, 도구, 캐시, 모니터링을 하나로 조립하고 AgentExecutor를 생성하는 구성 모듈. |

### 설명 ① agent_config.py — 에이전트 표준 구성

이야기 파트에서 비유한 "설계도"입니다. LLM, 라우터, 도구, 에이전트를 하나로 조립합니다.

**LLM 생성 — 환경 변수에 따라 자동 선택:**

```python
def _build_llm():
    provider = os.getenv("LLM_PROVIDER", "ollama").lower()

    if provider == "openai":
        from langchain_openai import ChatOpenAI
        return ChatOpenAI(
            model=os.getenv("OPENAI_MODEL", "gpt-4o-mini"),
            openai_api_key=os.getenv("OPENAI_API_KEY"),
            timeout=AGENT_TIMEOUT_SECONDS,    # 60초
            max_retries=RETRY_MAX_ATTEMPTS,   # 3번
        )
    else:
        from langchain_ollama import ChatOllama
        return ChatOllama(
            base_url=os.getenv("OLLAMA_BASE_URL", "http://localhost:11434"),
            model=os.getenv("OLLAMA_MODEL", "deepseek-r1:8b"),
        )
```

`.env`에서 `LLM_PROVIDER`만 바꾸면 이 함수가 다른 LLM을 만들어 돌려줍니다. OpenAI와 Ollama 중 어느 쪽을 쓰든 이후 코드는 동일합니다.

`timeout`과 `max_retries`도 눈여겨보세요. LLM 서버가 느리거나 일시적으로 불안정할 때, 60초 안에 응답하지 않으면 포기하고 3번까지 다시 시도합니다. v0.5에서는 이 설정이 없었어요.

**AgentExecutor 조립:**

```python
executor = AgentExecutor(
    agent=agent,
    tools=self.tools,
    max_iterations=10,
    max_execution_time=60,      # 60초 타임아웃
    handle_parsing_errors=True, # 파싱 오류 자동 재시도
    return_intermediate_steps=True,
)
```

v0.5의 AgentExecutor와 비교하면 `max_execution_time`이 추가됐습니다. 에이전트가 도구를 10번 넘게 호출하거나 60초 안에 답을 못 내면 강제로 멈춥니다. 무한 루프 방지 안전장치입니다.

**재시도 로직:**

```python
def _run_with_retry(self, query, chat_history=None):
    for attempt in range(1, RETRY_MAX_ATTEMPTS + 1):
        try:
            return self.agent_executor.invoke({"input": query, ...})
        except Exception as exc:
            if attempt < RETRY_MAX_ATTEMPTS:
                time.sleep(RETRY_DELAY_SECONDS)  # 2초 대기 후 재시도
    return {"output": "3회 재시도 후 처리에 실패했습니다.", ...}
```

Ollama 서버가 잠깐 느려지는 경우처럼 일시적 오류일 때, 2초 기다렸다가 다시 시도합니다. 3번 모두 실패하면 사용자에게 실패 메시지를 돌려줍니다. v0.5에서는 오류가 나면 바로 에러가 터졌는데, 이제는 부드럽게 처리됩니다.

### 설명 ② cache.py — TTL 응답 캐시

이야기 파트에서 비유한 "메모 패드"를 구현한 모듈입니다.

**캐시 키 생성 방식:**

같은 질문인지 어떻게 알 수 있을까요? "이서연의 연차 며칠 남았어?"와 "이서연의 연차 며칠 남았어?"는 같은 문장이지만, 단순히 문자열 비교를 합니다. SHA-256 해시로 변환해서 키로 씁니다.

```python
def _make_key(self, query: str) -> str:
    raw = f"{query}::"
    return hashlib.sha256(raw.encode("utf-8")).hexdigest()
```

"이서연의 연차 며칠 남았어?" → `a3f8c2d1...` (64자 해시)

이 해시가 메모 패드의 "이름표"입니다. 질문이 들어오면 이름표를 만들어서, 같은 이름표가 있으면 메모를 꺼냅니다.

**캐시 조회:**

```python
def get(self, query: str) -> Optional[Any]:
    key = self._make_key(query)
    entry = self._store.get(key)

    if entry is None:
        return None  # 메모 없음

    value, expires_at = entry
    if time.time() > expires_at:
        del self._store[key]
        return None  # 유통기한 만료

    return value  # 메모 반환
```

`_store`는 `{키: (값, 만료시각)}` 딕셔너리입니다. 가져올 때 현재 시각과 만료시각을 비교해서, 만료됐으면 삭제하고 없는 것처럼 돌려줍니다.

**최대 크기 관리:**

```python
def set(self, query: str, value: Any) -> None:
    if len(self._store) >= self.max_size:
        # 가장 오래된(가장 빨리 만료되는) 항목 제거
        oldest_key = min(self._store, key=lambda k: self._store[k][1])
        del self._store[oldest_key]

    expires_at = time.time() + self.ttl
    self._store[key] = (value, expires_at)
```

메모 패드가 1,000개가 넘으면 가장 오래된 포스트잇을 떼어냅니다. 책상이 무한정 넓을 수 없으니까요.

> **캐시 적중률(Hit Rate)이란?** 전체 요청 중에 캐시에서 답한 비율입니다. 적중률이 50%라면, 절반은 캐시에서 바로 답하고 절반은 새로 처리했다는 뜻입니다. 같은 질문이 반복될수록 적중률이 올라갑니다.

### 설명 ③ monitoring.py — 로깅 + 토큰 추적

**구조화 로깅 설정:**

```python
def setup_logging(level="INFO", use_json=False, log_file=None):
    if use_json:
        formatter = JsonFormatter()  # JSON 형식
    else:
        formatter = logging.Formatter(
            "%(asctime)s | %(levelname)-8s | %(name)s | %(message)s"
```

### 더 알아보기

- **캐시 무효화**: "캐시 무효화(cache invalidation)는 컴퓨터 과학의 두 가지 어려운 문제 중 하나"라는 오래된 농담이 있습니다. 언제 캐시를 버려야 하는가? TTL 방식은 단순하지만, 더 복잡한 시나리오 — 예를 들어 "직원 데이터가 DB에서 업데이트됐을 때 즉시 캐시를 무효화하고 싶다면" — 는 별도 설계가 필요합니다. v0.6은 TTL만 씁니다.

- **Redis 캐시**: v0.6의 캐시는 메모리에만 저장됩니다. 프로그램을 재시작하면 사라집니다. 실제 운영 환경에서는 Redis 같은 외부 캐시 서버를 쓰면 재시작해도 캐시가 유지됩니다.

- **로그 레벨 전략**: `LOG_LEVEL=DEBUG`로 설정하면 모든 세부 동작이 출력됩니다 — 라우팅 판단 근거, 캐시 키 해시, LLM 호출 파라미터까지. 개발 중에는 DEBUG, 운영 중에는 INFO 또는 WARNING으로 설정하는 게 일반적입니다. DEBUG 로그가 너무 많으면 오히려 필요한 정보를 찾기 어렵습니다.

- **싱글톤 패턴**: `get_agent()`를 여러 번 호출해도 에이전트가 하나만 만들어집니다. LLM 연결, ChromaDB 로드 같은 비용이 큰 초기화 작업을 한 번만 하기 위해서입니다. 파일 맨 아래의 `_agent_instance: Optional[ConnectHRAgent] = None` 변수가 이 역할을 합니다.

---

> **v0.6 완성**: 비서에게 메모 패드(TTL 캐시)를 줬습니다 — 같은 질문엔 즉시 답하고, 오래된 메모는 자동으로 버립니다. 업무 일지(구조화 로깅)를 줬습니다 — 비서가 무엇을 했는지 남기고, 나중에 추적할 수 있습니다. 근무 규정서(.env)를 줬습니다 — 코드 없이 설정만 바꿔서 LLM을 교체하거나 동작을 조정할 수 있습니다. 비서가 이제 일만 잘하는 게 아니라 제대로 된 직원처럼 일합니다. 마지막 챕터에서는 이 비서의 실력 자체를 올립니다. 지금은 "연차 규정 알려줘"라고 물으면 관련 문단 5개를 검색해서 LLM에 넣습니다. 그 5개가 정말 딱 맞는 5개인지, 더 좋은 방법이 있는지 — RAG 파이프라인을 직접 뜯어고칩니다.
