# 스킬 + 검토 모드 구조 v3 (집필 워크플로우)

> **이 문서는 `워크플로우-설계-v3.md`와 한 쌍이다.**
> 워크플로우-설계-v3.md가 "무엇을 어떤 순서로 하는가(STEP)"를 정의하고,
> 이 문서는 "어떤 도구와 체크리스트를 쓰는가(스킬+검토 모드)"를 정의한다.

---

## 설계 원칙

- **에이전트는 없다.** v1은 품질 에이전트 3개를 뒀고, v2는 8개로 확장했다. v3는 에이전트를 전부 제거했다.
- **스킬은 도구다.** 하나의 작업만 수행하고 결과를 돌려준다. 판단하지 않는다. (22개)
- **검토 모드는 체크리스트다.** v1의 품질 에이전트 3개를 체크리스트로 전환했다. AI가 특정 관점을 의식적으로 점검하게 만든다. (3개)

### 왜 에이전트를 뺐는가

| v2 (에이전트 8개) | v3 (에이전트 0개) | 이유 |
|---|---|---|
| 씨앗-에이전트 | STEP 1 | "에이전트"가 아니라 **워크플로우 STEP**이다 |
| 코드분해-에이전트 | STEP 2 | 같은 LLM이 이름표만 바꿔 단 것 |
| 설계-에이전트 | STEP 3~4 | 나눠도 같은 모델이 수행 → 분담 의미 없음 |
| 집필-에이전트 | STEP 5 | 나누면 컨텍스트 전달 비용만 생김 |
| 마무리-에이전트 | STEP 6~7 | |
| 인사이트-에이전트 | [인사이트 검토] | 에이전트 → **체크리스트**로 전환 |
| 의도감시-에이전트 | [의도감시 검토] | |
| 감수-에이전트 | [감수 검토] | |

### 1. 인사이트 검토

> **언제**: STEP 1~5, 변수를 수집할 때마다
> **뭘 하나**: 저자가 놓치고 있는 부분을 짚어주는 추가 질문

#### 참조 스킬

- `A4.의존성-매퍼` → "이 기술을 쓰려면 저 개념이 먼저 필요한데, 순서가 맞아?"
- `B5.난이도-곡선기` → "여기서 난이도가 뛰는데, 독자가 버틸 수 있을까?"
- `B6.갭-분석기` → "이 도메인에서 보통 다루는 OO이 빠져있는데, 의도적으로 뺀 거야?"
- `D6.분량-계산기` → "이 분량이면 100페이지 권장을 넘어. 저자가 리뷰할 때 뭘 빼면 좋을까?"

#### 체크리스트

```
□ 저자 답변에서 모호하거나 구체화가 필요한 부분이 있는가?
□ 코드 분석 결과와 저자 답변 사이에 충돌이 있는가?
□ 이전 STEP 답변과 현재 답변 사이에 모순이 있는가?
□ 분량(100p) 관점에서 현재 범위가 현실적인가?
□ 독자 관점에서 빠진 선행 개념이 있는가?
□ 저자가 아직 결정하지 않은 채 넘어간 부분이 있는가?
```

#### 동작 방식

```
저자 답변 수신
    ↓
답변 + 코드 분석 결과 + 이전 답변 맥락 분석
    ↓
"이건 생각해봤어?" 형태의 인사이트 질문 생성
    ↓
questions/pending/에 질문 저장 (대화에서 묻히지 않게)
    ↓
저자 반응 → 반영 or "넘어가자"
    ↓
답변 완료 시 questions/done/으로 이동
```

#### questions/ 폴더 — 질문 장바구니

대화 속에서 질문하면 묻혀버린다.
**인사이트 검토에서 생성한 질문**과 **저자가 직접 메모한 질문** 모두 여기에 쌓는다.

**누가 넣나:**
- **인사이트 검토**: 워크플로우 진행 중 발견한 "저자가 놓친 부분" 자동 생성
- **저자**: 작업 중 떠오른 아이디어, 나중에 확인할 것, 결정을 미룬 것 등 직접 파일 생성

```
questions/
├── pending/                    ← 아직 답하지 않은 질문
│   ├── 001-async-범위.md              (인사이트 검토가 생성)
│   ├── 002-트랜잭션-선행개념.md         (인사이트 검토가 생성)
│   ├── 003-에러처리-챕터-넣을까.md      (저자가 직접 생성)
│   └── 004-부록에-docker-넣기.md       (저자가 직접 생성)
└── done/                       ← 답변 완료 + 반영된 질문
    ├── 001-async-범위.md
    └── ...
```

각 질문 파일 형식:

```markdown
# 001-async-범위

## 출처
인사이트 검토 / STEP 2 코드 분석 중
(또는: 저자 메모 / STEP 3 진행 중 떠오름)

## 질문
완성 코드에 @Async가 있는데, 비동기 처리를 다룰 거야?
다룬다면 독자가 쓰레드 개념을 알아야 하는데,
그건 배경지식에 포함되는 거야? 아니면 챕터를 써야 해?

## 영향 범위
이 답변에 따라 STEP 4(목차)에서 챕터 추가 여부가 달라짐

## 저자 답변
(pending 상태에서는 비어있음)

## 반영 결과
(done으로 이동할 때 기록: 어떤 산출물에 어떻게 반영했는지)
```

#### 인사이트 질문 유형

**코드 기반 인사이트**:
```
"완성 코드에 @Async가 있던데, 비동기 처리를 다룰 거야?
 다룬다면 독자가 쓰레드 개념을 알아야 하는데,
 그건 배경지식에 포함되는 거야? 아니면 챕터를 써야 해?"
```

**구조 기반 인사이트**:
```
"v0.3에서 JPA를 도입하고 v0.4에서 인증을 붙이는데,
 독자 입장에서 두 개가 동시에 새로우면 힘들어.
 사이에 'JPA에 익숙해지는' 쉬어가기 버전을 넣는 건 어때?"
```

**의도 기반 인사이트**:
```
"'JWT를 이해시키는 게 목적'이라고 했는데,
 코드에 리프레시 토큰 로직이 있거든.
 리프레시 토큰은 '이해' 범위야 '구현' 범위야?
 이해라면 비유로만, 구현이라면 실습 코드가 필요해."
```

**분량 기반 인사이트**:
```
"지금 목차가 12챕터인데, 100페이지에 넣으려면 챕터당 8페이지야.
 코드까지 넣으면 빡빡해. 빼도 되는 챕터가 있을까?
 아니면 2개 챕터를 합칠 수 있을까?"
```

### 2. 의도감시 검토

> **언제**: STEP 5, 각 챕터 초안 완료 직후
> **뭘 하나**: 챕터가 의도에서 벗어나지 않았는지 검증

#### 참조 스킬

- `D5.의도-대조기` → 의도 밖 내용 탐지, 깊이 초과 탐지
- `D1.용어-탐지기` → 실무 수준 용어가 교육용이어야 할 곳에 등장하는지
- `D4.포맷-검증기` → 코드가 교육용인지, 상수 규칙을 지켰는지

#### 체크리스트 (7항목)

```
□ 의도 일치     이 챕터의 내용이 "책의 의도"에 부합하는가?
□ 범위 이탈     "의도 밖 범위"에 해당하는 내용이 슬며시 들어오지 않았는가?
□ 코드 성격     예제 코드가 교육용인가? (실무 코드가 아닌가?)
□ 깊이 적절     이 책의 목적(이해 vs 구현)에 맞는 깊이인가?
□ 비유 독립     이야기 파트의 비유가 기술 파트 없이도 이해되는가?
□ 핵심 유지     챕터 핵심 메시지가 책 전체 핵심 메시지와 연결되는가?
□ 상수 준수     챕터 내부 구조(이야기→기술), 비유 길이, 코드 분류를 지켰는가?
```

#### 출력 예시

```
✅ 의도 일치: OK — "인증이 뭔지 이해시키기"에 부합

⚠️ 범위 이탈 감지:
   → 23번째 줄에서 Spring Security의 FilterChain을 설명하고 있음.
     의도 밖 범위(Spring Security 연동)에 해당.
     제안: 삭제하거나 "나중에 Security를 쓰면 이런 게 자동으로 된다" 한 줄로 축소

⚠️ 코드 성격 경고:
   → AuthService.java에서 BCrypt 해싱, salt 처리까지 구현함.
     이건 실무 코드에 가까움. 교육용이라면 단순 비교로 충분.
     제안: password.equals(input)으로 단순화,
     "실무에서는 반드시 암호화를 쓴다" 한 줄 언급

✅ 깊이 적절: OK
✅ 비유 독립: OK
✅ 핵심 유지: OK
✅ 상수 준수: OK
```

### 3. 감수 검토 (3인 편집장)

> **언제**: 각 STEP 산출물 완성 후, 저자 확인 전
> **뭘 하나**: 3가지 관점에서 질문으로 감수

#### 3인 편집장

| 편집장 | 관점 | 핵심 질문 스타일 |
|--------|------|-----------------|
| **기술 감수자** | 기술 정확성, 빠진 개념, 순서 | "독자가 OO를 모르는데 여기서 OO이 나와도 될까?" |
| **독자 대변인** | 난이도 곡선, 진입장벽, 완주 가능성 | "여기서 독자가 포기하지 않을까?" |
| **이야기 편집장** | 서사 흐름, 비유 적절성, 몰입도 | "이 도입이 설명으로 시작하는데, 경험담으로 열면 어떨까?" |

#### 참조 스킬

- **기술 감수자**: `B5.난이도-곡선기` 결과 보고 기술 의존성 질문
- **독자 대변인**: `D6.분량-계산기` 결과 보고 완주 가능성 질문
- **이야기 편집장**: `D2.톤-검사기` 결과 보고 몰입도 질문

#### 동작 방식

```
STEP 산출물 완성
    ↓
3인 편집장이 각자 2~3개 질문
    ↓
저자가 질문에 답하면서 수정 여부 결정
    ↓
"넘어가자"도 가능
```

#### STEP별 감수 포인트

| STEP | 기술 감수자 | 독자 대변인 | 이야기 편집장 |
|------|------------|-----------|-------------|
| 1. 씨앗 | 핵심 메시지가 기술적으로 정확한가? | — | 차별점이 독자에게 와닿는 이야기가 되나? |
| 2. 코드 해부 | 빠진 핵심 개념은? 기술 의존성은? | 첫 버전이 너무 어렵지 않나? | — |
| 3. 시나리오+버전 | 버전 순서가 기술 의존성에 맞나? | 시나리오에 감정이입 가능? 포기 구간은? | 전환점이 극적인가? 긴장감 있나? |
| 4. 뼈대 | 빠진 기술은? 순서 최적화 제안 | 챕터 수가 적절? 완주 가능? | 챕터 제목이 호기심을 유발하나? |
| 5. 챕터 | 기술 설명이 정확? 더 좋은 예시? | 이 챕터만 읽어도 이해되나? | 도입 이야기가 몰입감 있나? |
| 6. 프롤로그 | 핵심 개념이 빠지진 않았나? | 프롤로그만 읽고 책을 사고 싶어지나? | 하나의 완결된 이야기인가? |
| 7. 마무리 | 부록에 넣을 심화 자료 제안 | 에필로그이 독자에게 용기를 주나? | 서문이 진정성 있나? |

## 스킬 카탈로그 (22개, 3단계 차등 정의)

스킬은 **하나의 작업만 수행하고 결과를 돌려주는 원자적 도구**다.
판단하지 않는다. 여러 STEP에서 조합해서 사용한다.

### 정의 기준

| Tier | 수준 | 개수 | 어떤 스킬 |
|------|------|------|----------|
| **Tier 1** | 상세 (입력/출력/규칙/예시) | 7개 | 규칙이 없으면 결과가 불안정한 스킬 |
| **Tier 2** | 간략 (한 줄 + 입출력) | 9개 | 입출력만 알면 되는 스킬 |
| **Tier 3** | 목록 (한 줄 설명) | 6개 | AI가 자연스럽게 수행하는 스킬 |

---

### Tier 1 — 상세 정의 (7개)

규칙이 없으면 결과가 불안정한 스킬. 명시적 규칙이 필요하다.

---

#### C1. 비유-생성기

> 기술 용어를 일상 비유로 변환한다.

| 항목 | 내용 |
|------|------|
| **입력** | 기술 용어 + 맥락 |
| **출력** | 비유 (길이 제한 없음) |

**규칙**:
- 독자가 100% 아는 일상적인 것만 사용한다
- **기술 용어로 비유 금지**: "세션은 JWT와 비슷하다" ❌ / "세션은 놀이공원 팔찌" ✅
- 길이 제한 없음. 비유가 길어져도 괜찮다
- 용어의 정식 설명이 길어지면 기술 파트로 넘긴다

```
입력: "세션(Session)" + 인증 맥락
출력: "세션은 놀이공원 팔찌예요. 입구에서 한 번 확인하면 팔찌를 채워주고,
      이후엔 팔찌만 보여주면 돼요."

입력: "API" + 웹 개발 맥락
출력: "API는 식당 웨이터예요. 주문을 받아서 주방에 전달하고,
      음식이 나오면 가져다주죠."

입력: "트랜잭션" + DB 맥락
출력: "은행 송금이에요. 내 통장에서 빠지고 상대 통장에 들어가는 게
      둘 다 되거나 둘 다 안 되거나. 반만 되는 건 없어요."
```

#### B3. 코드-태거

> 코드 파일에 [실습]/[설명]/[참고] 태그를 붙인다.

| 항목 | 내용 |
|------|------|
| **입력** | 파일 목록 + 해당 챕터의 핵심 주제 |
| **출력** | 파일별 태그 + 태그 근거 |

**분류 규칙**:

| 태그 | 조건 | 독자 액션 |
|------|------|----------|
| [실습] | 이번 챕터 핵심 주제 코드 | 직접 작성 |
| [설명] | 중요하지만 핵심은 아닌 코드 | 읽고 이해 |
| [참고] | 이 챕터 주제가 아닌 코드 | 파일명 + 한 줄 |

```
입력: v0.5 파일 목록 + 핵심 주제 "인증"
출력:
  LoginController.java  → [실습] (인증 핵심)
  AuthService.java      → [실습] (인증 핵심)
  login.html            → [실습] (인증 핵심)
  SessionConfig.java    → [설명] (중요하지만 직접 작성은 아님)
  User.java             → [설명] (엔티티 설명 필요)
  PostController.java   → [참고] (v0.3에서 작성, 변경 없음)
  Post.java             → [참고] (v0.2에서 작성)
```

#### B6. 갭-분석기

> 도메인 표준 커리큘럼과 현재 목차를 비교하여 누락된 주제를 찾아낸다.

| 항목 | 내용 |
|------|------|
| **입력** | 현재 `planning/outline.md` + 도메인 키워드 + `planning/seed.md` |
| **출력** | 누락 주제 목록 (필수/권장/선택 우선순위) |

**우선순위 규칙**:

| 등급 | 조건 |
|------|------|
| [필수] | 이거 없으면 독자가 핵심 내용을 이해할 수 없음 |
| [권장] | 있으면 책의 차별점이 됨, 독자 만족도 상승 |
| [선택] | 심화/부록 처리 가능, 분량 고려 시 생략 가능 |

**동작**:
1. 도메인 표준 커리큘럼 도출 (일반적으로 이 주제 입문서가 다루는 것)
2. 현재 목차와 비교
3. seed.md 의도 필터 적용 + 우선순위 분류
4. 저자 승인 후에만 반영

```
입력: outline.md (스프링부트 웹 개발) + seed.md
출력:
  [필수] 의존성 주입 — 이해 없이 Spring을 쓸 수 없음
  [권장] 예외 처리 — 실무 차별점, "실수→해결" 소재로 좋음
  [선택] 테스트 — 분량 고려 시 부록 처리 가능
  [선택] 설정 관리 — seed.md 의도(이해) 범위 초과, 생략 권장
```

#### D3. 파트-분리-검증기

> 이야기 파트와 기술 파트가 제대로 분리되었는지 검증한다.

| 항목 | 내용 |
|------|------|
| **입력** | 챕터 전체 텍스트 |
| **출력** | 분리 위반 목록 |

**검사 규칙**:
- 이야기 파트에 코드 블록이 있으면 위반 → 기술 파트로 이동
- 이야기 파트에서 비유로 사용한 용어가 기술 파트에 정식 정의 없으면 위반
- 기술 파트에 [실습]/[설명]/[참고] 분류가 없으면 위반
- "이것만은 기억하자" + 다음 예고가 없으면 위반

```
입력: Ch.3 전체
출력:
  ⚠️ 이야기 파트 15줄: 코드 블록 발견 → 기술 파트로 이동
  ⚠️ 이야기 파트 "세션" 비유 사용 → 기술 파트에 정식 정의 누락
  ✅ 기술 파트에 [실습]/[설명]/[참고] 분류 표시됨
  ✅ "이것만은 기억하자" + 다음 예고 있음
```

#### D4. 포맷-검증기

> 산출물이 상수(구조, 형식, 규칙)를 준수하는지 검증한다.

| 항목 | 내용 |
|------|------|
| **입력** | 산출물 텍스트 + 검증 대상 상수 목록 |
| **출력** | 상수별 준수/위반 결과 |

**체크리스트**:
- 챕터 구조: 이야기 파트 → 기술 파트 순서
- 코드 성격: 교육용 코드인가? (실무 코드 아닌가?)
- 용어 정리: 기술 파트 첫 머리에 있는가?
- 코드 분류: [실습]/[설명]/[참고] 표시가 있는가?
- 비유 길이: 제한 없이 자유롭게 사용했는가?
- 톤: 대화체를 유지하는가?

```
입력: Ch.3 + 상수 체크리스트
출력:
  ✅ 챕터 구조: 이야기 파트 → 기술 파트 순서
  ⚠️ 코드 성격: AuthService에 BCrypt 구현 (실무 코드)
     → 교육용 코드로 단순화 필요
  ✅ 용어 정리: 기술 파트 첫 머리에 있음
  ✅ 코드 분류: [실습]/[설명]/[참고] 표시됨
```

#### D5. 의도-대조기

> 텍스트가 seed.md의 의도/범위와 일치하는지 대조한다.

| 항목 | 내용 |
|------|------|
| **입력** | 텍스트 + planning/seed.md (의도 + 의도 밖 범위) |
| **출력** | 의도 일치/이탈 결과 + 이탈 위치 |

**대조 규칙**:
- seed.md의 "의도 밖 범위"에 해당하는 내용이 포함되면 이탈
- seed.md의 "책의 의도"(이해 vs 구현)보다 깊은 수준이면 깊이 초과
- 챕터 핵심 메시지가 seed.md의 "핵심 메시지"와 연결되지 않으면 경고

```
입력: Ch.3 + seed.md (의도: JWT 이해, 밖: Spring Security)
출력:
  ✅ 의도 일치: JWT 동작 원리 설명 — 부합
  ⚠️ 범위 이탈: 23줄 Spring Security FilterChain 설명
     → seed.md "의도 밖 범위"에 해당
  ⚠️ 깊이 초과: 45줄 리프레시 토큰 구현 코드
     → seed.md "이해"가 목적인데 "구현" 수준
  ✅ 핵심 메시지 연결: "JWT는 신분증" — 책 전체 메시지와 연결됨
```
