# D7 규칙-충돌-탐지기 + D8 프로젝트-버전-비교기 설계서

> 작성일: 2026-03-04
> 상태: 설계 완료 — 구현 대기
> 변경: 22스킬 + 3검토모드 + 0에이전트 → 22스킬 + 3검토모드 + **2횡단에이전트**

## 1. 배경

### 1.1 현재 문제

| 문제 | 현재 상황 | 필요 |
|------|----------|------|
| **규칙 충돌 감지 부재** | 사용자 요구사항이 기존 규칙과 충돌해도 감지 수단 없음 | 매 대화마다 자동으로 규칙 위반 사전 경고 |
| **프로젝트 버전 비교 부재** | 책 전체를 새로 쓸 때 이전 버전과 비교할 도구 없음 | 프로젝트 단위 구조적 비교 리포트 |

### 1.2 왜 에이전트인가 (스킬이 아닌 이유)

v3의 스킬 정의: **"하나의 작업만 수행하고 결과를 돌려주는 원자적 도구. 판단하지 않는다."**

D7/D8은 이 정의에 맞지 않는다:

| | 기존 스킬 (22개) | D7/D8이 해야 할 일 |
|---|---|---|
| **판단** | 안 함 | **판단이 핵심** (충돌인가? 개선인가?) |
| **실행 시점** | STEP 안에서 호출됨 | STEP 밖 — 매 턴(D7) / 프로젝트 전환(D8) |
| **자율성** | 수동 호출 | D7은 **항상 켜져 있어야** 함 |
| **컨텍스트** | 메인 AI와 공유 | **별도 컨텍스트**가 효율적 |

→ "스킬도 검토 모드도 아닌" 새 개념: **횡단 에이전트** 도입

### 1.3 아키텍처 변경

```
v3 기존:
┌─────────────────────────────────┐
│  하나의 AI                       │
│  22 스킬 + 3 검토 모드           │
│  STEP 1 → 2 → 3 → 4 → 5 → 6 → 7 │
└─────────────────────────────────┘

v3 확장:
┌─────────────────────────────────┐
│  하나의 AI                       │
│  22 스킬 + 3 검토 모드           │
│  STEP 1 → 2 → 3 → 4 → 5 → 6 → 7 │
│                                  │
│  ┌──────────┐  ┌──────────┐     │
│  │ D7 에이전트│  │ D8 에이전트│     │
│  │ (매 턴)   │  │ (프로젝트) │     │
│  │ 규칙 충돌  │  │ 버전 비교  │     │
│  └──────────┘  └──────────┘     │
│      ↑ 횡단: STEP 밖에서 독립 실행  │
└─────────────────────────────────┘
```

**변경되는 것**: 에이전트 2개 추가
**변경되지 않는 것**: 22 스킬, 3 검토 모드, 7 STEP 흐름 (전부 그대로)

## 4. D7. 규칙-충돌-탐지기 (횡단 에이전트)

### 4.1 역할

사용자가 매 대화에서 입력하는 요구사항/지시가 기존 스킬 규칙 및 상수와 **충돌하는지 사전 탐지**한다.

- 충돌이 없으면 **침묵** (오버헤드 최소화)
- 충돌이 있으면 **작업 시작 전에 경고** 출력

### 4.2 서브에이전트 명세

```yaml
이름: D7-규칙-충돌-탐지기
유형: 횡단 에이전트 (Task 서브에이전트)
모델: claude-haiku-4-5      # 매 턴 실행 → 경량 필수
실행: 백그라운드             # 메인 작업과 병렬 가능
입력:
  - 사용자 프롬프트 텍스트
  - 규칙 체크리스트 (아래 4.3 참조)
출력:
  - 충돌 없음 → "NO_CONFLICT" (메인 AI가 무시)
  - 충돌 있음 → 구조화된 충돌 리포트
```

### 4.3 규칙 체크리스트 (에이전트에게 전달)

에이전트가 매 턴 참조하는 규칙 목록. `conflict-detection.md`에 정리.

| 우선순위 | 출처 | 규칙 수 | 예시 |
|---------|------|---------|------|
| **필수** | CLAUDE.md 상수 | ~12개 | 톤=대화체, 코드 성격=교육용, 이야기 파트에 코드 없음 |
| **필수** | Tier 1 스킬 핵심 규칙 | 7개 | C1: 기술 용어로 비유 금지, D3: 코드 블록 금지 |
| **권장** | 검토 모드 체크리스트 | ~20개 | 의도감시 7항목, 인사이트 6항목 |
| **참고** | Tier 2/3 스킬 입출력 규약 | 15개 | A5: 두 버전 필요 |

### 4.4 실행 흐름

```
사용자 프롬프트 입력 (매 턴)
    ↓
메인 AI: Task 서브에이전트 발동 (haiku, 백그라운드)
    │          ↓
    │     [D7 에이전트] 프롬프트 vs 규칙 체크리스트 대조
    │          ↓
    │     충돌 없음 → "NO_CONFLICT" 반환
    │     충돌 있음 → 충돌 리포트 반환
    ↓
메인 AI: 서브에이전트 결과 확인
    ↓
"NO_CONFLICT" → 원래 작업 그대로 진행 (침묵)
충돌 리포트   → 작업 시작 전에 경고 출력 + 선택지 제시
```

**핵심**: 백그라운드 실행이므로 메인 AI가 작업 준비를 병렬로 할 수 있음. 충돌이 발견되면 작업을 멈추고 경고.

### 4.4.1 실행 보장 메커니즘

D7은 "매 대화 자동 실행"이 핵심이다. 이를 보장하기 위한 구현 전략:

**구현 방법: CLAUDE.md 워크플로우 규칙으로 강제**

```markdown
## 워크플로우 진행 규칙

0. **[최우선] 매 대화 규칙 검증 (D7)**:
   사용자 프롬프트를 받으면, 어떤 작업보다 먼저
   D7 횡단 에이전트를 Task 서브에이전트(haiku, 백그라운드)로 실행한다.
   이 규칙은 "상수"와 같은 수준이다 — 질문하지 않고 자동 적용한다.
```

**보장 수준에 대한 솔직한 설명:**

| 비교 대상 | 보장 수준 | 실제 준수율 |
|----------|----------|-----------|
| 상수 ("톤=대화체") | CLAUDE.md 규칙 | ~95% (매우 잘 지켜짐) |
| **D7 자동 실행** | CLAUDE.md 규칙 (동일) | ~95% (동일 수준 기대) |
| 스킬 호출 | STEP 가이드에 명시 | ~90% |

CLAUDE.md의 "상수"들(톤=대화체, 코드 성격=교육용 등)이 잘 지켜지는 것과 같은 메커니즘.
D7을 규칙 #0(최우선)으로 배치하여 다른 규칙보다 먼저 읽히도록 한다.

**한계**: 기술적으로 100% 강제가 아닌 "매우 강한 지시".
하지만 현재 아키텍처에서 이것이 가장 실용적이고 신뢰도 높은 방법이다.

### 4.5 출력 형식

#### 충돌 없음 (침묵 — 아무것도 출력하지 않음)

#### 충돌 있음

```
🔍 규칙 충돌 검사 결과

⚠️ 충돌 감지 (N건)
1. [충돌] "이야기 파트에 코드 넣어줘"
   ↔ D3.파트-분리-검증기: "이야기 파트에 코드 블록 금지"
   → 제안: 기술 파트로 이동하거나, D3 규칙을 예외 처리

2. [충돌] "실무 코드로 예제 작성"
   ↔ CLAUDE.md 상수: "코드 성격 = 교육용 (실무 코드 금지)"
   → 제안: 교육용으로 단순화하거나, 상수를 수정

❓ 주의 (M건)
1. [주의] "챕터당 30페이지"
   ↔ D6.분량-계산기: "100p 권장" 기준 초과 가능
   → 확인 필요: 전체 분량 제한을 완화할 것인지

📋 검사 대상: 상수 N개, 스킬 규칙 M개, 검토 체크리스트 K개
```

#### `규칙 검사` 수동 명령 시

```
📋 규칙 현황 정리

── 현재 적용 중인 규칙 ──────────────────
[상수] 12개 — 문체 5, 구조 5, 독자 3 (정상)
[스킬 규칙] 7개 — Tier 1 핵심 규칙 (정상)
[검토 체크리스트] 20개 — 인사이트 6, 의도감시 7, 감수 7 (정상)

── 예외 허용된 규칙 ──────────────────
1. [2026-03-01] "코드 비중 높임"
   원래: 코드 비중=낮음 → 허용: CH08만 코드 비중 높임
   기록: answers.md #Q15

── 잠재 충돌 (미해결) ──────────────────
없음
```

### 4.6 핵심 규칙

| # | 규칙 | 설명 |
|---|------|------|
| 1 | **충돌/주의 2단계 분류** | 명시적 위반 = 충돌(⚠️), 잠재적 긴장 = 주의(❓) |
| 2 | **출처 명시 필수** | 어떤 스킬/상수와 충돌하는지 반드시 표시 |
| 3 | **제안 필수** | 충돌마다 해결 방안 1개 이상 제시 |
| 4 | **자동 차단 안 함** | 경고만 출력. 최종 결정은 저자 |
| 5 | **누적 기록** | 예외 허용된 규칙은 `answers.md`에 기록 → 이후 검토에서 참조 |
| 6 | **침묵 원칙** | 충돌 없으면 아무것도 출력하지 않음 |

### 4.7 충돌 분류 기준

| 분류 | 조건 | 예시 |
|------|------|------|
| **충돌 ⚠️** | 기존 규칙과 정면으로 모순 | "코드를 이야기 파트에" ↔ D3 금지 |
| **주의 ❓** | 직접 모순은 아니나 긴장 관계 | "챕터 20개" ↔ 100p 권장 |

## 5. D8. 프로젝트-버전-비교기 (횡단 에이전트)

### 5.1 역할

프로젝트 전체를 새로 쓸 때(예: `사내AI비서` → `사내AI비서_v2`) 이전 프로젝트와 **구조적으로 비교**하여 무엇이 바뀌었고, 무엇이 개선/퇴보했는지 리포트를 생성한다.

### 5.2 서브에이전트 명세

```yaml
이름: D8-프로젝트-버전-비교기
유형: 횡단 에이전트 (Task 서브에이전트)
모델: claude-opus-4-6         # 정밀 판단 필요, 비빈번 실행
실행: 포그라운드              # 결과를 기다려야 다음 진행 가능
입력:
  - 이전 프로젝트 경로 (projects/사내AI비서/)
  - 새 프로젝트 경로 (projects/사내AI비서_v2/)
출력:
  - 6계층 비교 리포트
  - review/version-comparison-report.md 파일 저장
```

### 5.3 트리거

```
프로젝트 전체 재작성 시작 (새 프로젝트 폴더 생성)
    ↓
메인 AI: 이전 프로젝트 폴더 존재 확인 (이름 유사성 or 사용자 지정)
    ↓ 존재함
메인 AI: Task 서브에이전트 발동 (opus, 포그라운드)
    ↓
[D8 에이전트] 양쪽 프로젝트의 산출물 읽기 + 비교
    ↓
통합 리포트 반환 → 메인 AI가 사용자에게 출력
    ↓
권장 사항 + 주의점 제시
```

- **자동 트리거**: `새 책 만들기`에서 이전 프로젝트 이름 기반 매칭 시
- **수동 명령**: `버전 비교 [이전 프로젝트] [새 프로젝트]`

### 5.4 비교 계층 (6단계)

| # | 비교 계층 | 비교 대상 | 방법 |
|---|----------|----------|------|
| 1 | **의도 비교** | seed-v*.md 간 | 6개 질문 답변 항목별 대조 |
| 2 | **코드 비교** | code-analysis-v*.md 간 | 기술 스택, 모듈 수, 분석 범위 대조 |
| 3 | **구조 비교** | scenario-v*.md + outline-v*.md 간 | 챕터 목록/순서/내용 대조 |
| 4 | **본문 비교** | chapters/*.md 간 | 동일 주제 챕터 매칭 → 내용 변경점 |
| 5 | **분량 비교** | D6 기준 적용 | 이전/이후 예상 페이지 수 비교 |
| 6 | **품질 판단** | 전체 | 개선/퇴보/유지 종합 평가 |

### 5.5 산출물 의존 관계 (영향도 분석용)

```
seed
 └→ code-analysis
     └→ scenario
         └→ outline
             ├→ chapters (본문)
             ├→ prologue
             └→ epilogue
```

### 5.6 챕터 매칭 전략

프로젝트가 통째로 바뀌면 챕터 번호가 달라질 수 있다. 아래 기준으로 매칭:

| 우선순위 | 매칭 기준 | 설명 |
|---------|----------|------|
| 1 | **제목 유사도** | 제목이 비슷한 챕터끼리 매칭 |
| 2 | **다루는 기능** | 같은 코드 모듈/기능을 다루는 챕터 매칭 |
| 3 | **매칭 불가 (신규)** | v2에만 있는 완전히 새로운 챕터 → "신규" 표시 |
| 4 | **매칭 불가 (삭제)** | v1에만 있는 챕터 → "삭제" 표시 + 이유 확인 |
