# 책 집필 워크플로우 설계 v3

기술 서적(100페이지 권장)을 이야기처럼 쓰는 워크플로우.
저자(도메인 전문가)와 AI가 대화하며 책을 완성한다.

> **코드 워크플로우**는 완성 코드를 **만드는** 단계다 (별도 설계).
> 이 문서의 STEP 2에서 분석하는 **완성 코드 = 코드 워크플로우의 산출물**이다.

---

## 1. 설계 철학

### 단일 LLM 원칙

이 워크플로우는 **하나의 AI(Claude)가 처음부터 끝까지 관통**한다.
에이전트를 나누면 "역할 분담"처럼 보이지만, 실제로는 같은 LLM이 다른 이름표를 달고 있을 뿐이다.
이름표를 바꿔 달아도 기억이 이어지지 않으면 의미가 없고, 이어지면 굳이 나눌 필요가 없다.

### 세 가지 개념만 쓴다

| 개념 | 정체 | 역할 |
|------|------|------|
| **STEP** | 흐름 | 1~7번까지 순서대로 진행하는 워크플로우 단계 |
| **스킬** | 도구 | 하나의 작업만 수행하고 결과를 돌려주는 원자적 도구 (22개) |
| **검토 모드** | 체크리스트 | 산출물 품질을 검증하는 관점과 질문 목록 (3개) |

- **STEP은 흐름이다.** "누가"가 아니라 "언제, 무엇을" 정의한다.
- **스킬은 도구다.** 판단하지 않는다. 입력을 받아 출력을 돌려줄 뿐이다.
- **검토 모드는 체크리스트다.** AI가 특정 관점을 의식적으로 점검하게 만든다.

### 코드 워크플로우와의 관계

```
┌─────────────────────────┐
│    코드 워크플로우         │
│    (별도 설계)            │
│                           │
│  완성 코드를 만드는 단계    │
│  (코드 작성, 문법, 컨벤션)  │
│                           │
│  산출물: 완성 코드          │
└────────────┬──────────────┘
             │
             ▼ (완성 코드 전달)
┌─────────────────────────┐
│    집필 워크플로우         │
│    (이 문서)              │
│                           │
│  완성 코드를 분석하고       │
│  책을 쓰는 단계            │
│                           │
│  STEP 2에서               │
│  A1~A5 스킬로 분석 수행     │
└─────────────────────────┘
```

코드 워크플로우가 만든 **완성 코드**를 집필 워크플로우의 **STEP 2**에서 분석 스킬(A1~A5)로 해부한다.
코드를 작성하는 스킬(문법, 컨벤션, 빌드 등)은 코드 워크플로우에만 존재하며 이 문서에 포함하지 않는다.

## 2. 전체 흐름 (7 STEP)

```
Phase 1 ── 의도 확립
  STEP 1. 씨앗              "이 책은 뭐다"

Phase 2 ── 재료 파악
  STEP 2. 코드 해부          "재료가 뭐가 있지"

Phase 3 ── 이야기 설계
  STEP 3. 시나리오 + 버전 설계  "어떤 순서로 이야기하지"
  STEP 4. 뼈대 세우기          "목차와 코드 실습 배치"

Phase 4 ── 집필
  STEP 5. 챕터 집필 (반복)      "쓴다"

Phase 5 ── 완성
  STEP 6. 프롤로그 + 로드맵     "숲을 보여준다"
  STEP 7. 마무리              "서문, 맺음말, 부록"
```

### 왜 이 순서인가

| STEP | 이름 | 왜 이 순서? |
|------|------|------------|
| 1. 씨앗 | 의도 확립 | **의도가 필터다.** 의도를 모르면 코드를 분석해도 어디에 집중할지 모른다 |
| 2. 코드 해부 | 코드 분석 | 의도 필터를 가지고 코드를 봐야 "이건 다루고 이건 안 다룬다"를 판단할 수 있다 |
| 3. 시나리오 + 버전 | 이야기 설계 | 시나리오가 있어야 버전을 쪼갤 수 있다. 코드를 알아야 현실적인 시나리오가 나온다 |
| 4. 뼈대 세우기 | 구조 설계 | 시나리오 + 버전 + 코드가 있어야 목차를 짤 수 있다 |
| 5. 챕터 집필 | 쓰기 | 뼈대가 있어야 쓴다 |
| 6. 프롤로그 | 숲 보기 | 모든 챕터가 끝나야 전체를 조망하는 프롤로그를 쓸 수 있다 |
| 7. 마무리 | 닫기 | 책의 처음(서문)과 끝(맺음말)을 닫는다 |

### 전체 흐름 다이어그램

```
┌─────────────────────────────┐
│     하나의 AI (Claude)        │
│                               │
│  [스킬 22개] 도구함           │
│  [검토 모드 3개] 체크리스트    │
│                               │
│  STEP 1~7 순서대로 진행       │
└─────────────────────────────┘

STEP 1 씨앗 ─── planning/seed.md
  │  스킬: C2 / 검토: 인사이트+감수
  v
STEP 2 코드 해부 ─── planning/code-analysis.md
  │  스킬: A1~A4 / 검토: 인사이트+감수
  v
STEP 3 시나리오+버전 ─── planning/scenario.md + versions/
  │  스킬: A5,B1,B2 / 검토: 인사이트+감수
  v
STEP 4 뼈대 ─── planning/outline.md
  │  스킬: B3~B6,C4,D6 / 검토: 인사이트+감수
  v
STEP 5 챕터 집필 ─── chapters/*.md (반복)
  │  스킬: C1~C3,C5,D1~D4
  │  검토: 인사이트+의도감시+감수 (3개 모두)
  v
STEP 6 프롤로그 ─── book/front/prologue.md, roadmap.md
  │  스킬: C2 / 검토: 감수
  v
STEP 7 마무리 ─── book/front/preface.md, book/back/*.md
     스킬: C2,C4,D6 / 검토: 감수
```

## 3. 상수 (Constants) — 절대 질문하지 않는다

이 시스템이 만드는 책의 정체성. 바뀌면 시스템 자체가 달라진다.
저자에게 묻지 않고 모든 단계에서 자동으로 적용한다.

### 문체 상수

| 상수 | 값 | 왜 상수인가 |
|------|-----|-----------|
| 포맷 | 짧은 쿡북 (100p 권장, 초과 허용) | 이 시스템의 정체성. 두꺼운 교과서가 아님. 초과분은 저자가 리뷰하며 빼기 작업 |
| 스타일 | 스토리텔링 | 핵심 컨셉. 설명이 아니라 이야기 |
| 톤 | 대화체 | "독자에게 말 거는 느낌" 고정 |
| 언어 수준 | 쉽게 (비유 필수) | 어려운 용어는 반드시 비유로 먼저 |
| 비유 길이 | 제한 없음 | 비유는 자유롭게. 용어 설명이 길어지면 기술 파트로 이동 |

### 구조 상수

| 상수 | 값 | 왜 상수인가 |
|------|-----|-----------|
| 챕터 내부 구조 | 이야기 파트 → 기술 파트 | 모든 챕터가 이 구조를 따른다 |
| 코드 분류 | 실습 / 설명 / 참고 3단계 | 모든 코드를 다 쓰게 하지 않는 원칙 |
| 성장 방식 | 버전별 성장 (챕터 = 한 버전) | 매 챕터에서 프로젝트가 한 단계 올라간다 |
| 코드 성격 | 교육용 (실무 코드 금지) | 실무 코드면 난이도 급상승 |
| 용어 정리 | 비유 → 기술 파트에서 정식 정의 필수 | 비유로 넘긴 용어를 방치하지 않는다 |

### 독자 상수

| 상수 | 값 | 왜 상수인가 |
|------|-----|-----------|
| 독자 수준 | 배경지식 있음, 이 책의 주제만 모름 | 매번 물어봤자 답은 같다 |
| 책 유형 | 개념서 | 코드보다 개념과 흐름 중심 |
| 코드 비중 | 낮음 | 이야기 파트에 코드 없음. 기술 파트에서만 |

### 책 구조 상수 (독자가 읽는 순서)

```
프롤로그    ← 코드 없이 전체 개념을 이야기로
로드맵      ← 버전별 성장 안내
본문 Ch.1~N ← 각 챕터: [이야기 파트] → [기술 파트]
서문 / 맺음말 / 부록
```

### 챕터 내부 구조 상수

```
┌─────────────────────────────────────┐
│  이야기 파트 (코드 없음)              │
│  - 도입 (경험, 상황)                 │
│  - 개념 (비유 1~2문장)               │
│  - 시나리오 진행 (이번 버전에서 뭘 바꾸나)│
│  - "이것만은 기억하자" + 다음 예고     │
├─────────────────────────────────────┤
│  기술 파트 (이야기 끝난 후)            │
│  - 용어 정리 (비유→정식 정의)          │
│  - 파일 계층 구조 [실습/설명/참고]      │
│  - 실습 코드 + 설명 코드              │
│  - 더 알아보기                       │
└─────────────────────────────────────┘
```

### 글쓰기 규칙 상수

1. **이야기 먼저**: 모든 챕터는 이야기 파트로 시작한다
2. **비유는 자유롭게**: 길이 제한 없음. 용어의 정식 설명이 길어지면 기술 파트로 넘긴다
3. **기술은 뒤에서**: 이야기 파트가 끝난 후 기술 파트에서 상세 설명
4. **비유 → 정의 필수**: 비유로 넘어간 용어는 기술 파트에서 반드시 정식 정의
5. **한 챕터 한 버전**: 챕터가 끝나면 결과물이 한 단계 성장
6. **점진적 깊이**: 쉬운 것 → 어려운 것
7. **코드 최소화**: 이야기 파트에 코드 없음. 기술 파트에서만
8. **챕터 닫기**: "이것만은 기억하자" + 다음 버전 예고
9. **기술은 저자가 결정**: AI가 제안, 저자가 최종 결정
10. **실패 먼저**: 정답부터 알려주지 않는다. 실수 → 에러 → 해결 순서

## 4. 변수 (Variables) — 인사이트 질문으로 채운다

단순히 "뭐야?"라고 묻지 않는다.
**저자가 놓치고 있는 걸 짚어주면서** 묻는다.

### STEP 1 변수: 씨앗

| 변수 | 기본 질문 | 인사이트 질문 (저자가 놓친 부분) | 예시 |
|------|----------|------------------------------|------|
| **주제** | "뭐에 대한 책이야?" | — (단순 질문으로 충분) | "스프링부트 웹 개발" |
| **집필 동기** | "왜 이 책을 쓰고 싶어?" | "혹시 비슷한 책이 이미 있는데도 쓰려는 거라면, 그 책의 뭐가 아쉬웠어? 그게 차별점이 될 수 있어" | "기존 책들은 코드만 나열. 난 '왜'를 설명하고 싶어" |
| **저자 경험** | "이 분야 경험이 어때?" | "경험 중에 실패담이 있으면 그게 최고의 챕터 소재야. 이 기술 때문에 삽질한 적 있어?" | "세션 안 끊어서 서버 메모리 터진 적 있어" |
| **핵심 메시지** | "이 책에서 제일 하고 싶은 말?" | "독자가 이 책 다 읽고 동료에게 한 마디로 설명한다면 뭐라고 할까? 그게 핵심 메시지야" | "JWT는 결국 '신분증을 매번 보여주는 것'이다" |
| **책의 의도** | "독자가 뭘 이해하게 하고 싶어?" | "'이해'와 '할 수 있다'는 다르거든. 이 책은 '아 이게 이거구나' 하게 만드는 건지, '직접 만들 수 있게' 하는 건지 명확히 하자. 이게 이후 모든 결정의 필터야" | 이해: "JWT가 뭔지, 왜 쓰는지" / 실행: "JWT 인증을 직접 구현" |
| **의도 밖 범위** | "그러면 이건 안 다루는 거지?" | "완성 코드에 OO이 있던데, 이것도 의도 안이야 밖이야? 의도 밖이면 코드 분석에서부터 빠져. 과감하게 쳐내자" | "Spring Security, OAuth2, 프로덕션 배포는 밖" |

### STEP 2 변수: 코드 해부

| 변수 | 기본 질문 | 인사이트 질문 | 예시 |
|------|----------|-------------|------|
| **핵심 기능 목록** | (AI가 코드 분석 후 제시) | "이 중에 독자에게 안 보여줘도 되는 건? 보여주고 싶은 욕심이 있어도 의도 밖이면 빼자. 100페이지 권장이야" | 회원가입, 로그인, CRUD, 댓글 → 댓글 뺌 |
| **기술 스택** | (AI가 코드에서 추출) | "코드에 `@Transactional`이 있는데, 트랜잭션을 설명할 거야? 한다면 JPA보다 앞에 와야 할 수도 있어" | Spring Boot, JPA, H2, Thymeleaf |
| **의도 밖 코드** | (AI가 의도 필터로 분류) | "이 코드들은 의도 밖이라 분석에서 뺄게. 혹시 다시 넣고 싶은 거 있어?" | AOP, Security 관련 코드 제외 |

### STEP 3 변수: 시나리오 + 버전

| 변수 | 기본 질문 | 인사이트 질문 | 예시 |
|------|----------|-------------|------|
| **시나리오** | "책 전체를 관통하는 프로젝트/상황이 있어?" | "'블로그를 만든다'가 아니라 왜 만드는지 동기가 있어야 해. '자기 생각을 세상에 공유하고 싶은 개발자가 블로그를 처음부터 만들어가는 이야기' 처럼. 동기가 있어야 독자가 감정이입해" | "주니어가 사수 없이 블로그를 만들어가는 여정" |
| **전환점** | "시나리오에서 제일 중요한 전환점은?" | "전환점은 '아하!' 순간이야. '쉬운 방법으로 했다가 한계를 느끼고 제대로 된 방법을 배우는 지점'이 좋아" | "서버 재시작했더니 데이터가 다 날아간 순간" |
| **버전 분해** | "기능들을 어떤 순서로 붙여갈까?" (AI 제안) | "이 순서로 따라갈 때 중간에 포기할 지점이 있나? v0.3→v0.4 사이 난이도가 뛰면 쉬어가는 버전을 끼워야 해" | v0.1 Hello → v0.2 목록(ArrayList) → v0.3 목록(JPA) |
| **버전별 성과** | "각 버전 끝에 독자 눈에 뭐가 보여?" | "눈에 보이는 게 있어야 계속 따라와. 내부 리팩터링만 하는 버전은 독자가 지쳐" | v0.1: "Hello World" / v0.2: 글 목록 페이지 |

### STEP 4 변수: 뼈대 세우기

| 변수 | 기본 질문 | 인사이트 질문 | 예시 |
|------|----------|-------------|------|
| **코드 실습 분류** | (AI가 버전별로 분류 제안) | "이 버전 핵심이 '인증'이면 JPA 코드를 실습시킬 필요 없어. 이번 버전의 주인공 코드만 [실습]이고 나머지는 [설명]이나 [참고]" | v0.4 인증: LoginController[실습], Post[참고] |
| **여정 맵** | "독자가 제일 먼저 알아야 할 건?" | "난이도 곡선을 그려봐. 계속 올라가기만 하면 지쳐. 어려운 챕터 다음엔 쉬운 챕터. 산 오르다가 쉬어가는 평지처럼" | 출발→기초(쉬움)→심화(어려움)→실습(쉬움)→실전 |
| **목차** | (AI가 여정 기반으로 제안) | "챕터 제목이 '이거 재밌겠다' 느낌이어야 해. 'HTTP 세션'이 아니라 '우리 집에 자물쇠 달기'. 그리고 7챕터 이하가 좋아. 많으면 시작 전에 지쳐" | Part1: 첫 발 떼기 / Part2: 살 붙이기 |
| **기술 매핑** | (AI가 코드 기반으로 매핑) | "완성 코드와 다르게 가르치고 싶은 부분 있어? 코드는 처음부터 JPA지만, 책에서는 ArrayList 먼저 → JPA 전환으로 갈 수도 있어" | Ch.2: ArrayList → Ch.3: JPA로 전환 |

### STEP 5 변수: 챕터별 (챕터마다 반복 수집)

| 변수 | 기본 질문 | 인사이트 질문 | 예시 |
|------|----------|-------------|------|
| **도입 이야기** | "이 개념을 처음 접했을 때 어땠어?" | "독자가 첫 문장에서 '나도 그랬는데?'라고 느끼게 해야 해. 추상적 말고 구체적 순간. 언제, 어디서, 뭘 하다가?" | "밤 11시, 배포 직전에 세션이 뻗었다" |
| **비유** | "이 개념을 쉽게 비유하면?" | "독자가 비유 대상을 모르면 소용없어. '세션은 JWT와 비슷하다'는 비유가 아니야. 독자가 100% 아는 일상적인 것으로" | "API는 식당 웨이터" |
| **실패담** | "이걸 쓰면서 실수한 적 있어?" | "완벽한 코드를 바로 보여주면 복사하는 거지 배우는 게 아니야. 먼저 실수하고, 에러 만나고, 왜 그런지 파헤치는 게 진짜 학습이야" | "세션 타임아웃 안 걸어서 메모리 터진 이야기" |
| **한줄 기억** | "이 챕터에서 딱 하나만 기억하라면?" | "이 문장이 독자 머리에 박혀야 해. 기술 용어 말고 비유로. 내일 동료한테 자랑할 수 있는 한 마디" | "로그인은 문 열기, 인가는 방 골라 들어가기" |

## 5. 검토 모드 (3개)

검토 모드는 **에이전트가 아니다.** AI가 산출물을 만든 후 특정 관점의 체크리스트를 의식적으로 점검하는 것이다.
"모자를 바꿔 쓴다"고 생각하면 된다. 글을 쓰는 모자 → 검토하는 모자로 바꿔 쓰고 체크리스트를 돌린다.

### 5.1 인사이트 검토

> **언제**: STEP 1~5, 변수를 수집할 때마다
> **뭘 하나**: 저자가 놓치고 있는 부분을 짚어주는 추가 질문

#### 동작 방식

```
저자 답변 수신
    ↓
답변 + 코드 분석 결과 + 이전 답변 맥락 분석
    ↓
"이건 생각해봤어?" 형태의 인사이트 질문 생성
    ↓
questions/pending/에 질문 저장 (대화에서 묻히지 않게)
    ↓
저자 반응 → 반영 or "넘어가자"
    ↓
답변 완료 시 questions/done/으로 이동
```

#### questions/ 폴더 — 질문 장바구니

대화 속에서 질문하면 묻혀버린다.
**인사이트 검토에서 생성한 질문**과 **저자가 직접 메모한 질문** 모두 여기에 쌓는다.

**누가 넣나:**
- **인사이트 검토**: 워크플로우 진행 중 발견한 "저자가 놓친 부분" 자동 생성
- **저자**: 작업 중 떠오른 아이디어, 나중에 확인할 것, 결정을 미룬 것 등 직접 파일 생성

```
questions/
├── pending/                    ← 아직 답하지 않은 질문
│   ├── 001-async-범위.md              (인사이트 검토에서 생성)
│   ├── 002-트랜잭션-선행개념.md         (인사이트 검토에서 생성)
│   ├── 003-에러처리-챕터-넣을까.md      (저자가 직접 생성)
│   └── 004-부록에-docker-넣기.md       (저자가 직접 생성)
└── done/                       ← 답변 완료 + 반영된 질문
    ├── 001-async-범위.md
    └── ...
```

각 질문 파일 형식:

```markdown
# 001-async-범위

## 출처
인사이트 검토 / STEP 2 코드 분석 중
(또는: 저자 메모 / STEP 3 진행 중 떠오름)

## 질문
완성 코드에 @Async가 있는데, 비동기 처리를 다룰 거야?
다룬다면 독자가 쓰레드 개념을 알아야 하는데,
그건 배경지식에 포함되는 거야? 아니면 챕터를 써야 해?

## 영향 범위
이 답변에 따라 STEP 4(목차)에서 챕터 추가 여부가 달라짐

## 저자 답변
(pending 상태에서는 비어있음)

## 반영 결과
(done으로 이동할 때 기록: 어떤 산출물에 어떻게 반영했는지)
```

**왜 필요한가:**
- 인사이트 질문이 나온 시점에 바로 답하지 못할 수 있다
- 저자가 작업 중 떠오른 생각을 잊지 않게 메모해둘 수 있다
- pending 목록을 보면 "아직 결정 안 한 것들"이 한눈에 보인다
- done으로 옮길 때 "어디에 반영했는지"까지 기록하므로 추적 가능
- 저자가 직접 만든 질문도 AI가 다음 STEP에서 참고할 수 있다

#### 인사이트 질문 유형

**코드 기반 인사이트**:
```
"완성 코드에 @Async가 있던데, 비동기 처리를 다룰 거야?
 다룬다면 독자가 쓰레드 개념을 알아야 하는데,
 그건 배경지식에 포함되는 거야? 아니면 챕터를 써야 해?"
```

**구조 기반 인사이트**:
```
"v0.3에서 JPA를 도입하고 v0.4에서 인증을 붙이는데,
 독자 입장에서 두 개가 동시에 새로우면 힘들어.
 사이에 'JPA에 익숙해지는' 쉬어가기 버전을 넣는 건 어때?"
```

**의도 기반 인사이트**:
```
"'JWT를 이해시키는 게 목적'이라고 했는데,
 코드에 리프레시 토큰 로직이 있거든.
 리프레시 토큰은 '이해' 범위야 '구현' 범위야?
 이해라면 비유로만, 구현이라면 실습 코드가 필요해."
```

**분량 기반 인사이트**:
```
"지금 목차가 12챕터인데, 100페이지에 넣으려면 챕터당 8페이지야.
 코드까지 넣으면 빡빡해. 빼도 되는 챕터가 있을까?
 아니면 2개 챕터를 합칠 수 있을까?"
```

#### 인사이트 검토 체크리스트

```
□ 저자 답변에서 모호하거나 구체화가 필요한 부분이 있는가?
□ 코드 분석 결과와 저자 답변 사이에 충돌이 있는가?
□ 이전 STEP 답변과 현재 답변 사이에 모순이 있는가?
□ 분량(100p) 관점에서 현재 범위가 현실적인가?
□ 독자 관점에서 빠진 선행 개념이 있는가?
□ 저자가 아직 결정하지 않은 채 넘어간 부분이 있는가?
```

### 5.2 의도감시 검토

> **언제**: STEP 5, 각 챕터 초안 완료 직후
> **뭘 하나**: 챕터가 의도에서 벗어나지 않았는지 검증

#### 체크리스트 (7항목)

```
□ 의도 일치     이 챕터의 내용이 "책의 의도"에 부합하는가?
□ 범위 이탈     "의도 밖 범위"에 해당하는 내용이 슬며시 들어오지 않았는가?
□ 코드 성격     예제 코드가 교육용인가? (실무 코드가 아닌가?)
□ 깊이 적절     이 책의 목적(이해 vs 구현)에 맞는 깊이인가?
□ 비유 독립     이야기 파트의 비유가 기술 파트 없이도 이해되는가?
□ 핵심 유지     챕터 핵심 메시지가 책 전체 핵심 메시지와 연결되는가?
□ 상수 준수     챕터 내부 구조(이야기→기술), 비유 길이, 코드 분류를 지켰는가?
```

#### 사용 스킬

- `D5.의도-대조기` → 의도 밖 내용 탐지, 깊이 초과 탐지
- `D1.용어-탐지기` → 실무 수준 용어가 교육용이어야 할 곳에 등장하는지
- `D4.포맷-검증기` → 코드가 교육용인지, 상수 규칙을 지켰는지

#### 출력 예시

```
✅ 의도 일치: OK — "인증이 뭔지 이해시키기"에 부합

⚠️ 범위 이탈 감지:
   → 23번째 줄에서 Spring Security의 FilterChain을 설명하고 있음.
     의도 밖 범위(Spring Security 연동)에 해당.
     제안: 삭제하거나 "나중에 Security를 쓰면 이런 게 자동으로 된다" 한 줄로 축소

⚠️ 코드 성격 경고:
   → AuthService.java에서 BCrypt 해싱, salt 처리까지 구현함.
     이건 실무 코드에 가까움. 교육용이라면 단순 비교로 충분.
     제안: password.equals(input)으로 단순화,
     "실무에서는 반드시 암호화를 쓴다" 한 줄 언급

✅ 깊이 적절: OK
✅ 비유 독립: OK
✅ 핵심 유지: OK
✅ 상수 준수: OK
```

### 5.3 감수 검토 (3인 편집장)

> **언제**: 각 STEP 산출물 완성 후, 저자 확인 전

| 편집장 | 관점 | 핵심 질문 스타일 |

## 6. 각 STEP 상세

---

### STEP 1. 씨앗 — "이 책은 뭐다"

> **목적**: 의도 필터 확보. 이후 모든 결정의 기준.

**스킬**: `C2.요약기`
**검토**: [인사이트 검토] + [감수 검토]
**산출물**: `planning/seed.md`

#### 흐름

```
1. "뭐에 대한 책이야?" → 주제
2. "왜 이 책을 쓰고 싶어?" → 집필 동기 (인사이트 질문 포함)
3. "이 분야 경험이 어때?" → 저자 경험 (실패담 유도)
4. "이 책에서 제일 하고 싶은 말?" → 핵심 메시지
5. "독자가 뭘 이해하게 하고 싶어?" → 책의 의도 (이해 vs 구현 명확화)
```

| **Tier 1** | 상세 (입력/출력/규칙/예시) | 7개 | 규칙이 없으면 결과가 불안정한 스킬 |

### Tier 1 — 상세 정의 (7개)

규칙이 없으면 결과가 불안정한 스킬. 명시적 규칙이 필요하다.

---

#### C1. 비유-생성기

> 기술 용어를 일상 비유로 변환한다.

| 항목 | 내용 |
|------|------|
| **입력** | 기술 용어 + 맥락 |
| **출력** | 비유 (길이 제한 없음) |

**규칙**:
- 독자가 100% 아는 일상적인 것만 사용한다
- **기술 용어로 비유 금지**: "세션은 JWT와 비슷하다" ❌ / "세션은 놀이공원 팔찌" ✅
- 길이 제한 없음. 비유가 길어져도 괜찮다
- 용어의 정식 설명이 길어지면 기술 파트로 넘긴다

```
입력: "세션(Session)" + 인증 맥락
출력: "세션은 놀이공원 팔찌예요. 입구에서 한 번 확인하면 팔찌를 채워주고,
      이후엔 팔찌만 보여주면 돼요."

입력: "API" + 웹 개발 맥락
출력: "API는 식당 웨이터예요. 주문을 받아서 주방에 전달하고,
      음식이 나오면 가져다주죠."

입력: "트랜잭션" + DB 맥락
출력: "은행 송금이에요. 내 통장에서 빠지고 상대 통장에 들어가는 게
      둘 다 되거나 둘 다 안 되거나. 반만 되는 건 없어요."
```

---

#### B3. 코드-태거

> 코드 파일에 [실습]/[설명]/[참고] 태그를 붙인다.

| 항목 | 내용 |
|------|------|
| **입력** | 파일 목록 + 해당 챕터의 핵심 주제 |
| **출력** | 파일별 태그 + 태그 근거 |

**분류 규칙**:

| 태그 | 조건 | 독자 액션 |
|------|------|----------|
| [실습] | 이번 챕터 핵심 주제 코드 | 직접 작성 |
| [설명] | 중요하지만 핵심은 아닌 코드 | 읽고 이해 |
| [참고] | 이 챕터 주제가 아닌 코드 | 파일명 + 한 줄 |

```
입력: v0.5 파일 목록 + 핵심 주제 "인증"
출력:
  LoginController.java  → [실습] (인증 핵심)
  AuthService.java      → [실습] (인증 핵심)
  login.html            → [실습] (인증 핵심)
  SessionConfig.java    → [설명] (중요하지만 직접 작성은 아님)
  User.java             → [설명] (엔티티 설명 필요)
  PostController.java   → [참고] (v0.3에서 작성, 변경 없음)
  Post.java             → [참고] (v0.2에서 작성)
```

#### B6. 갭-분석기

> 도메인 표준 커리큘럼과 현재 목차를 비교하여 누락된 주제를 찾아낸다.

| 항목 | 내용 |
|------|------|
| **입력** | 현재 `planning/outline.md` + 도메인 키워드 + `planning/seed.md` |
| **출력** | 누락 주제 목록 (필수/권장/선택 우선순위) |

**우선순위 규칙**:

| 등급 | 조건 |
|------|------|
| [필수] | 이거 없으면 독자가 핵심 내용을 이해할 수 없음 |
| [권장] | 있으면 책의 차별점이 됨, 독자 만족도 상승 |
| [선택] | 심화/부록 처리 가능, 분량 고려 시 생략 가능 |

**동작**:
1. 도메인 표준 커리큘럼 도출 (일반적으로 이 주제 입문서가 다루는 것)
2. 현재 목차와 비교
3. seed.md 의도 필터 적용 + 우선순위 분류
4. 저자 승인 후에만 반영

```
입력: outline.md (스프링부트 웹 개발) + seed.md
출력:
  [필수] 의존성 주입 — 이해 없이 Spring을 쓸 수 없음
  [권장] 예외 처리 — 실무 차별점, "실수→해결" 소재로 좋음
  [선택] 테스트 — 분량 고려 시 부록 처리 가능
  [선택] 설정 관리 — seed.md 의도(이해) 범위 초과, 생략 권장
```

#### D3. 파트-분리-검증기

> 이야기 파트와 기술 파트가 제대로 분리되었는지 검증한다.

| 항목 | 내용 |
|------|------|
| **입력** | 챕터 전체 텍스트 |
| **출력** | 분리 위반 목록 |

**검사 규칙**:
- 이야기 파트에 코드 블록이 있으면 위반 → 기술 파트로 이동
- 이야기 파트에서 비유로 사용한 용어가 기술 파트에 정식 정의 없으면 위반
- 기술 파트에 [실습]/[설명]/[참고] 분류가 없으면 위반
- "이것만은 기억하자" + 다음 예고가 없으면 위반

```
입력: Ch.3 전체
출력:
  ⚠️ 이야기 파트 15줄: 코드 블록 발견 → 기술 파트로 이동
  ⚠️ 이야기 파트 "세션" 비유 사용 → 기술 파트에 정식 정의 누락
  ✅ 기술 파트에 [실습]/[설명]/[참고] 분류 표시됨
  ✅ "이것만은 기억하자" + 다음 예고 있음
```

#### D4. 포맷-검증기

> 산출물이 상수(구조, 형식, 규칙)를 준수하는지 검증한다.

| 항목 | 내용 |
|------|------|
| **입력** | 산출물 텍스트 + 검증 대상 상수 목록 |
| **출력** | 상수별 준수/위반 결과 |

**체크리스트**:
- 챕터 구조: 이야기 파트 → 기술 파트 순서
- 코드 성격: 교육용 코드인가? (실무 코드 아닌가?)
- 용어 정리: 기술 파트 첫 머리에 있는가?
- 코드 분류: [실습]/[설명]/[참고] 표시가 있는가?
- 비유 길이: 제한 없이 자유롭게 사용했는가?
- 톤: 대화체를 유지하는가?

```
입력: Ch.3 + 상수 체크리스트
출력:
  ✅ 챕터 구조: 이야기 파트 → 기술 파트 순서
  ⚠️ 코드 성격: AuthService에 BCrypt 구현 (실무 코드)
     → 교육용 코드로 단순화 필요
  ✅ 용어 정리: 기술 파트 첫 머리에 있음
  ✅ 코드 분류: [실습]/[설명]/[참고] 표시됨
```

#### D5. 의도-대조기

> 텍스트가 seed.md의 의도/범위와 일치하는지 대조한다.

| 항목 | 내용 |
|------|------|
| **입력** | 텍스트 + planning/seed.md (의도 + 의도 밖 범위) |
| **출력** | 의도 일치/이탈 결과 + 이탈 위치 |

**대조 규칙**:
- seed.md의 "의도 밖 범위"에 해당하는 내용이 포함되면 이탈
- seed.md의 "책의 의도"(이해 vs 구현)보다 깊은 수준이면 깊이 초과
- 챕터 핵심 메시지가 seed.md의 "핵심 메시지"와 연결되지 않으면 경고

```
입력: Ch.3 + seed.md (의도: JWT 이해, 밖: Spring Security)
출력:
  ✅ 의도 일치: JWT 동작 원리 설명 — 부합
  ⚠️ 범위 이탈: 23줄 Spring Security FilterChain 설명
     → seed.md "의도 밖 범위"에 해당
  ⚠️ 깊이 초과: 45줄 리프레시 토큰 구현 코드
     → seed.md "이해"가 목적인데 "구현" 수준
  ✅ 핵심 메시지 연결: "JWT는 신분증" — 책 전체 메시지와 연결됨
```

#### D6. 분량-계산기

> 텍스트의 예상 페이지 수를 계산한다.

| 항목 | 내용 |
|------|------|
| **입력** | 텍스트 (또는 챕터 목록 + 예상 글자 수) |
| **출력** | 예상 페이지 수 + 100p 권장 기준 대비 현황 |

**기준**: 1페이지 ≈ 500자 (코드 블록은 별도 계산)

```
입력: Ch.1~Ch.7 + 프롤로그 + 로드맵
출력:
  | 항목 | 글자 수 | 코드 | 예상 페이지 |
  |------|--------|------|------------|
  | 프롤로그 | 3,000 | 0 | 6p |
  | 로드맵 | 1,500 | 0 | 3p |
  | Ch.1 | 5,000 | 500 | 11p |
  | Ch.2 | 6,000 | 800 | 14p |
  | ... | ... | ... | ... |
  | 합계 | | | 92p |

  ℹ️ 100p 권장 기준: 8p 여유 (초과 시 저자 리뷰에서 빼기 작업)
```

### Tier 2 — 간략 정의 (9개)

입출력만 알면 되는 스킬. 동작이 직관적이다.

---

#### A1. 구조-스캐너

> 프로젝트의 디렉토리/파일 구조를 트리로 출력한다.

| 입력 | 출력 |
|------|------|
| 프로젝트 경로 또는 GitHub URL | 디렉토리 트리 (파일명 + 한 줄 설명) |

---

#### A2. 기능-추출기

> 코드에서 사용자 관점의 기능 목록을 추출한다.

| 입력 | 출력 |
|------|------|
| 코드 파일들 | 기능 목록 (기능명 + 관련 파일 + 한 줄 설명) |

---

#### A3. 기술스택-탐지기

> 프로젝트에서 사용된 기술/라이브러리/프레임워크를 탐지한다.

| 입력 | 출력 |
|------|------|
| 빌드 파일 (pom.xml, build.gradle 등) + 코드 | 기술 스택 목록 (기술명 + 버전 + 용도) |

---

#### A4. 의존성-매퍼

> 코드 컴포넌트 간 의존 관계를 매핑한다.

| 입력 | 출력 |
|------|------|
| 코드 파일들 | 의존성 그래프 (A → B: "A가 B를 사용") + 의존성 메모 |

---

#### A5. diff-생성기

> 두 버전(또는 두 텍스트) 간 차이점을 요약한다.

| 입력 | 출력 |
|------|------|
| 버전A, 버전B (코드 또는 텍스트) | 신규/변경/삭제 목록 + 핵심 변경 요약 |

---

#### B1. 기능-정렬기

> 기능 목록을 학습 난이도/의존성 순서로 정렬한다.

| 입력 | 출력 |
|------|------|
| 기능 목록 + 의존성 그래프 | 난이도순 정렬된 기능 목록 + 정렬 근거 |

---

#### B2. 스냅샷-설계기

> 정렬된 기능을 버전별 스냅샷으로 묶는다.

| 입력 | 출력 |
|------|------|
| 정렬된 기능 목록 + 시나리오 흐름 | 버전 목록 (버전명 + 포함 기능 + 눈에 보이는 성과) |

---

#### B4. 계층-생성기

> 태그가 붙은 파일 목록을 디렉토리 트리로 시각화한다.

| 입력 | 출력 |
|------|------|
| 태그 붙은 파일 목록 | 트리 형태의 파일 계층 구조 (태그 + 한 줄 설명 포함) |

---

#### B5. 난이도-곡선기

> 챕터별 난이도를 시각적으로 표현한다.

| 입력 | 출력 |
|------|------|
| 챕터 목록 + 각 챕터의 핵심 개념 수/신규 기술 수 | 난이도 곡선 (텍스트 시각화) + 스파이크 경고 |

## 9. 폴더 구조

### 원칙

1. **폴더 분리**: 여러 파일이 생성되거나 주제가 다른 문서들은 폴더별로 분리한다
2. **파일명 버전**: 산출물은 수정할 때마다 파일명에 `-v1`, `-v2` 버전을 붙인다. 이전 버전은 삭제하지 않는다

### 산출물 버전 관리

모든 산출물은 **파일명에 버전**을 붙여 관리한다.
가장 높은 번호가 현재 버전이고, 이전 버전은 보존한다.

#### 버전이 올라가는 시점

| 시점 | 예시 | 설명 |
|------|------|------|
| 초안 생성 | `seed-v1.md` | AI가 처음 만든 산출물 |
| 검토 후 수정 | `seed-v2.md` | 인사이트/의도감시/감수 검토 반영 |
| 저자 피드백 반영 | `seed-v3.md` | 저자가 확인 후 수정 요청한 것 반영 |

#### 파일명 규칙

```
{이름}-v{번호}.md

예:
  seed-v1.md          ← 초안
  seed-v2.md          ← 감수 후 수정
  01-제목-v1.md        ← 챕터 초안
  01-제목-v2.md        ← 의도감시 수정
  01-제목-v3.md        ← 감수 후 최종
  prologue-v1.md       ← 프롤로그 초안
```

#### STEP별 버전 흐름

**STEP 1~4 (기획)**: 초안(v1) → 검토 반영(v2) → 저자 확인(v3)
**STEP 5 (챕터)**: 초안(v1) → 의도감시 수정(v2) → 감수 반영(v3) → 저자 확인(v4)
**STEP 6~7 (마무리)**: 초안(v1) → 감수 반영(v2) → 저자 확인(v3)

> 매번 v3이나 v4까지 가는 건 아니다. 검토에서 문제가 없으면 v1이 최종이 될 수도 있다.
> 핵심은 **수정이 발생할 때마다 새 파일**을 만드는 것이다. 덮어쓰지 않는다.

## 11. 필요 없는 것 (왜 에이전트를 뺐는가)

### v2에서 뺀 것: 에이전트 8개 → 0개

| v2 (에이전트 8개) | v3 (에이전트 0개) | 왜 뺐나 |
|---|---|---|
| 씨앗-에이전트 | STEP 1. 씨앗 | "에이전트"가 아니라 **워크플로우 STEP**이다. STEP 설명으로 충분 |
| 코드분해-에이전트 | STEP 2. 코드 해부 | 같은 LLM이 이름표만 바꿔 달고 있을 뿐 |
| 설계-에이전트 | STEP 3~4. 설계 | 에이전트를 나눠도 기억이 이어지면 의미 없고, 안 이어지면 문제 |
| 집필-에이전트 | STEP 5. 챕터 집필 | |
| 마무리-에이전트 | STEP 6~7. 마무리 | |
| 인사이트-에이전트 | [인사이트 검토] 체크리스트 | 에이전트가 아니라 **관점 전환**이다. 모자를 바꿔 쓰는 것 |
| 의도감시-에이전트 | [의도감시 검토] 체크리스트 | 체크리스트만 있으면 AI가 자연스럽게 수행 |
| 감수-에이전트 | [감수 검토] 3인 편집장 체크리스트 | |

**핵심 논리**: 에이전트를 나누는 이유는 "역할 분담"인데, 하나의 LLM이 전체를 관통할 때:
- 나눠도 같은 모델이 수행한다 → 분담의 의미가 없다
- 나누면 컨텍스트 전달이 필요하다 → 오히려 비용이 생긴다
- STEP(흐름) + 스킬(도구) + 검토 모드(체크리스트)로 충분하다

### v1에서 이미 뺐던 것

| 뺀 것 | 이유 |
|-------|------|
| 독자 프로필/페르소나 수집 단계 | 독자 수준은 상수. 질문할 필요 없음 |
| config.md에 저자가 상수 편집 | 상수는 시스템이 고정. 저자가 건드릴 일 없음 |
| 상세 git 커밋 규칙 | 커밋은 자연스럽게. 규칙표 불필요 |
| 독자 여정을 별도 STEP으로 분리 | 목차 설계와 같이 해야 맥락이 이어짐 |
| 코드 실습 설계를 별도 STEP으로 분리 | 목차에 코드 분류를 넣어야 하므로 같은 STEP |
