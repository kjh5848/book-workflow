# 에이전트 구조 v1 (집필 워크플로우)

> **이 문서는 `워크플로우-설계-v1.md`와 한 쌍이다.**
> 워크플로우-설계-v1.md가 "무엇을 어떤 순서로 하는가(STEP)"를 정의하고,
> 이 문서는 "품질 관리를 누가 하는가(에이전트)"를 정의한다.

---

## 설계 원칙

- **STEP이 워크플로우다.** 씨앗→코드해부→설계→집필→마무리는 "에이전트"가 아니라 **흐름의 단계**다.
- **에이전트는 품질 관리만 한다.** 워크플로우를 진행하는 것은 AI+저자이고, 에이전트는 산출물을 **검증하는 관점**이다.
- **스킬은 별도 정의하지 않는다.** 비유-생성기, 요약기 등은 AI가 글 쓸 때 자연스럽게 하는 일이다. 별도 정의는 과잉 설계다.

---

## 전체 구조도

```
┌────────────────────────────────────────────────────────┐
│              워크플로우 STEP (7단계)                       │
│                                                          │
│  STEP1 ─→ STEP2 ─→ STEP3 ─→ STEP4 ─→ STEP5 ─→ STEP6~7 │
│  씨앗     코드해부   시나리오   뼈대     집필     마무리    │
└──────┬──────────┬──────────┬──────────┬──────────┬───────┘
       │          │          │          │          │
       ▼          ▼          ▼          ▼          ▼
┌────────────────────────────────────────────────────────┐
│              품질 에이전트 (3개)                           │
│                                                          │
│  인사이트-에이전트 │ 의도감시-에이전트 │ 감수-에이전트       │
│  (STEP 1~5)       │ (STEP 5)         │ (전 STEP)         │
└────────────────────────────────────────────────────────┘
```

**워크플로우 에이전트가 없는 이유:**
- "씨앗-에이전트", "코드분해-에이전트" 등은 결국 같은 AI가 이름표만 바꿔 단 것이다.
- STEP 설명으로 충분하다. 에이전트라는 이름을 붙여도 동작이 달라지지 않는다.

**스킬이 없는 이유:**
- "비유-생성기", "브릿지-작성기", "요약기" 등은 AI가 글 쓸 때 자연스럽게 하는 일이다.
- 별도로 정의하면 15~22개 스킬이 되는데, 이는 과잉 설계다.
- STEP 흐름 안에서 "비유로 설명해라", "요약해라" 등의 지시로 충분하다.

## 에이전트 상세 (3개)

---

### 1. 인사이트-에이전트

| 항목 | 내용 |
|------|------|
| **활성** | STEP 1~5, 변수를 수집할 때마다 |
| **역할** | 저자가 놓치고 있는 부분을 짚어주는 추가 질문 |
| **관리** | `questions/` 폴더 |

#### 동작 방식

```
저자 답변 수신
    ↓
답변 + 코드 분석 결과 + 이전 답변 맥락 분석
    ↓
"이건 생각해봤어?" 형태의 인사이트 질문 생성
    ↓
questions/pending/에 질문 저장 (대화에서 묻히지 않게)
    ↓
저자 반응 → 반영 or "넘어가자"
    ↓
답변 완료 시 questions/done/으로 이동
```

#### questions/ 폴더 — 질문 장바구니

대화 속에서 질문하면 묻혀버린다.
**인사이트-에이전트가 생성한 질문**과 **저자가 직접 메모한 질문** 모두 여기에 쌓는다.

**누가 넣나:**
- **인사이트-에이전트**: 워크플로우 진행 중 발견한 "저자가 놓친 부분" 자동 생성
- **저자**: 작업 중 떠오른 아이디어, 나중에 확인할 것, 결정을 미룬 것 등 직접 파일 생성

```
questions/
├── pending/                    ← 아직 답하지 않은 질문
│   ├── 001-async-범위.md              (인사이트-에이전트가 생성)
│   ├── 002-트랜잭션-선행개념.md         (인사이트-에이전트가 생성)
│   ├── 003-에러처리-챕터-넣을까.md      (저자가 직접 생성)
│   └── 004-부록에-docker-넣기.md       (저자가 직접 생성)
└── done/                       ← 답변 완료 + 반영된 질문
    ├── 001-async-범위.md
    └── ...
```

각 질문 파일 형식:

```markdown
# 001-async-범위

## 출처
인사이트-에이전트 / STEP 2 코드 분석 중
(또는: 저자 메모 / STEP 3 진행 중 떠오름)

## 질문
완성 코드에 @Async가 있는데, 비동기 처리를 다룰 거야?
다룬다면 독자가 쓰레드 개념을 알아야 하는데,
그건 배경지식에 포함되는 거야? 아니면 챕터를 써야 해?

## 영향 범위
이 답변에 따라 STEP 4(목차)에서 챕터 추가 여부가 달라짐

## 저자 답변
(pending 상태에서는 비어있음)

## 반영 결과
(done으로 이동할 때 기록: 어떤 산출물에 어떻게 반영했는지)
```

#### 인사이트 질문 유형

**코드 기반 인사이트**:
```
"완성 코드에 @Async가 있던데, 비동기 처리를 다룰 거야?
 다룬다면 독자가 쓰레드 개념을 알아야 하는데,
 그건 배경지식에 포함되는 거야? 아니면 챕터를 써야 해?"
```

**구조 기반 인사이트**:
```
"v0.3에서 JPA를 도입하고 v0.4에서 인증을 붙이는데,
 독자 입장에서 두 개가 동시에 새로우면 힘들어.
 사이에 'JPA에 익숙해지는' 쉬어가기 버전을 넣는 건 어때?"
```

**의도 기반 인사이트**:
```
"'JWT를 이해시키는 게 목적'이라고 했는데,
 코드에 리프레시 토큰 로직이 있거든.
 리프레시 토큰은 '이해' 범위야 '구현' 범위야?
 이해라면 비유로만, 구현이라면 실습 코드가 필요해."
```

**분량 기반 인사이트**:
```
"지금 목차가 12챕터인데, 100페이지에 넣으려면 챕터당 8페이지야.
 코드까지 넣으면 빡빡해. 빼도 되는 챕터가 있을까?
 아니면 2개 챕터를 합칠 수 있을까?"
```

### 2. 의도감시-에이전트

| 항목 | 내용 |
|------|------|
| **활성** | STEP 5, 각 챕터 초안 완료 직후 |
| **역할** | 챕터가 의도에서 벗어나지 않았는지 검증 |
| **참조** | `planning/seed.md` |

#### 체크리스트

```
□ 의도 일치     이 챕터의 내용이 "책의 의도"에 부합하는가?
□ 범위 이탈     "의도 밖 범위"에 해당하는 내용이 슬며시 들어오지 않았는가?
□ 코드 성격     예제 코드가 교육용인가? (실무 코드가 아닌가?)
□ 깊이 적절     이 책의 목적(이해 vs 구현)에 맞는 깊이인가?
□ 비유 독립     이야기 파트의 비유가 기술 파트 없이도 이해되는가?
□ 핵심 유지     챕터 핵심 메시지가 책 전체 핵심 메시지와 연결되는가?
□ 상수 준수     챕터 내부 구조(이야기→기술), 비유 길이, 코드 분류를 지켰는가?
```

#### 출력 예시

```
✅ 의도 일치: OK — "인증이 뭔지 이해시키기"에 부합

⚠️ 범위 이탈 감지:
   → 23번째 줄에서 Spring Security의 FilterChain을 설명하고 있음.
     의도 밖 범위(Spring Security 연동)에 해당.
     제안: 삭제하거나 "나중에 Security를 쓰면 이런 게 자동으로 된다" 한 줄로 축소

⚠️ 코드 성격 경고:
   → AuthService.java에서 BCrypt 해싱, salt 처리까지 구현함.
     이건 실무 코드에 가까움. 교육용이라면 단순 비교로 충분.
     제안: password.equals(input)으로 단순화,
     "실무에서는 반드시 암호화를 쓴다" 한 줄 언급

✅ 깊이 적절: OK
✅ 비유 독립: OK
✅ 핵심 유지: OK
✅ 상수 준수: OK
```

### 3. 감수-에이전트 (3인 편집장)

| 항목 | 내용 |
|------|------|
| **활성** | 전 STEP, 각 STEP 산출물 완성 후 저자 확인 전 |
| **역할** | 3가지 관점에서 질문으로 감수 |

#### 3인 편집장

| 편집장 | 관점 | 핵심 질문 스타일 |
|--------|------|-----------------|
| **기술 감수자** | 기술 정확성, 빠진 개념, 순서 | "독자가 OO를 모르는데 여기서 OO이 나와도 될까?" |
| **독자 대변인** | 난이도 곡선, 진입장벽, 완주 가능성 | "여기서 독자가 포기하지 않을까?" |
| **이야기 편집장** | 서사 흐름, 비유 적절성, 몰입도 | "이 도입이 설명으로 시작하는데, 경험담으로 열면 어떨까?" |

#### 동작 방식

```
STEP 산출물 완성
    ↓
3인 편집장이 각자 2~3개 질문
    ↓
저자가 질문에 답하면서 수정 여부 결정
    ↓
"넘어가자"도 가능
```

#### STEP별 감수 포인트

| STEP | 기술 감수자 | 독자 대변인 | 이야기 편집장 |
|------|------------|-----------|-------------|
| 1. 씨앗 | 핵심 메시지가 기술적으로 정확한가? | — | 차별점이 독자에게 와닿는 이야기가 되나? |
| 2. 코드 해부 | 빠진 핵심 개념은? 기술 의존성은? | 첫 버전이 너무 어렵지 않나? | — |
| 3. 시나리오+버전 | 버전 순서가 기술 의존성에 맞나? | 시나리오에 감정이입 가능? 포기 구간은? | 전환점이 극적인가? 긴장감 있나? |
| 4. 뼈대 | 빠진 기술은? 순서 최적화 제안 | 챕터 수가 적절? 완주 가능? | 챕터 제목이 호기심을 유발하나? |
| 5. 챕터 | 기술 설명이 정확? 더 좋은 예시? | 이 챕터만 읽어도 이해되나? | 도입 이야기가 몰입감 있나? |
| 6. 프롤로그 | 핵심 개념이 빠지진 않았나? | 프롤로그만 읽고 책을 사고 싶어지나? | 하나의 완결된 이야기인가? |
| 7. 마무리 | 부록에 넣을 심화 자료 제안 | 맺음말이 독자에게 용기를 주나? | 서문이 진정성 있나? |

#### 출력 예시 (STEP 4 목차 감수)

```markdown
## 편집장 감수

### 기술 감수자
> "Ch.3에서 JPA를 쓰는데, Ch.2에서 SQL을 한 번도 안 보여줬어.
>  독자가 '왜 SQL 안 쓰고 이걸 쓰지?' 의문을 가질 수 있어.
>  Ch.2 끝에 SQL로 한번 해보고 Ch.3에서 JPA로 바꾸면
>  동기부여가 되지 않을까?"

> "완성 코드에 AOP 코드가 있는데, 의도 밖이라 뺀 거 맞지?
>  근데 @Transactional을 설명하려면 AOP 개념이 살짝 필요할 수도 있어.
>  비유 수준으로라도 언급할지 결정해야 해."

### 독자 대변인
> "Ch.5→Ch.6 사이에 난이도가 갑자기 뛰어.
>  Ch.5는 CRUD인데 Ch.6이 갑자기 인증이야.
>  중간에 '지금까지 만든 것 정리' 쉬어가기 챕터가 있으면 좋겠어."

> "챕터가 9개인데, 독자가 완주할 수 있을까?
>  7개로 줄이거나, Part를 나눠서 '작은 완성'을 중간에 한 번 느끼게 하면 어때?"

### 이야기 편집장
> "Ch.1 제목이 '프로젝트 시작하기'인데 너무 교과서적이야.
>  '빈 화면에서 첫 글자가 나오기까지' 같은 제목은 어때?"

> "시나리오에 동기가 약해. '블로그를 만든다'인데,
>  왜 블로그인지 구체적 계기가 있으면 몰입감이 달라질 텐데?"
```

## STEP-에이전트 매핑

| STEP | 인사이트 | 의도감시 | 감수 |
|------|---------|---------|------|
| 1. 씨앗 | ✅ 놓친 부분 질문 | — | ✅ 의도 명확성 검증 |
| 2. 코드 해부 | ✅ 코드 기반 이슈 질문 | — | ✅ 빠진 개념, 기술 의존성 |
| 3. 시나리오+버전 | ✅ 전환점, 난이도, 포기 구간 | — | ✅ 서사 흐름, 감정이입 |
| 4. 뼈대 | ✅ 분량, 난이도 배분 | — | ✅ 완주 가능성, 누락 체크 |
| 5. 챕터 집필 | ✅ 챕터별 놓친 부분 | ✅ 의도 이탈/코드 성격/깊이 | ✅ 3인 편집장 리뷰 |
| 6. 프롤로그 | — | — | ✅ 프롤로그만으로 궁금해지는지 |
| 7. 마무리 | — | — | ✅ 진정성, 심화 자료 |

## v1에서 별도 정의하지 않는 것

| 항목 | 이유 |
|------|------|
| 워크플로우 에이전트 (씨앗, 코드분해, 설계, 집필, 마무리) | "에이전트"가 아니라 "워크플로우 STEP"이다. STEP 설명으로 충분 |
| 스킬 (비유-생성기, 요약기, 브릿지-생성기 등 15~22개) | AI가 글 쓸 때 자연스럽게 하는 일. 별도 정의는 과잉 설계 |

> v2에서는 워크플로우 에이전트 5개 + 품질 에이전트 3개 = 8개와 스킬 22개를 명시적으로 정의한다.
> v3에서는 에이전트를 다시 제거하고, 스킬 22개 + 검토 모드 3개 구조로 재편한다.
