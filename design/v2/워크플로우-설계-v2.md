# 책 집필 워크플로우 설계 v2

기술 서적(100페이지 권장)을 이야기처럼 쓰는 워크플로우.
저자(도메인 전문가)와 AI가 대화하며 책을 완성한다.

> **코드 워크플로우**는 완성 코드를 **만드는** 단계다 (별도 설계).
> 이 문서의 STEP 2에서 분석하는 **완성 코드 = 코드 워크플로우의 산출물**이다.
> 에이전트와 스킬의 상세 설계는 `에이전트-스킬-구조-v2.md`를 참조한다.

---

## 전체 흐름

```
Phase 1 ── 의도 확립
  STEP 1. 씨앗              "이 책은 뭐다"

Phase 2 ── 재료 파악
  STEP 2. 코드 해부          "재료가 뭐가 있지"

Phase 3 ── 이야기 설계
  STEP 3. 시나리오 + 버전 설계  "어떤 순서로 이야기하지"
  STEP 4. 뼈대 세우기          "목차와 코드 실습 배치"

Phase 4 ── 집필
  STEP 5. 챕터 집필 (반복)      "쓴다"

Phase 5 ── 완성
  STEP 6. 프롤로그 + 로드맵     "숲을 보여준다"
  STEP 7. 마무리              "서문, 맺음말, 부록"
```

### 왜 이 순서인가

| STEP | 이름 | 왜 이 순서? |
|------|------|------------|
| 1. 씨앗 | 의도 확립 | **의도가 필터다.** 의도를 모르면 코드를 분석해도 어디에 집중할지 모른다 |
| 2. 코드 해부 | 코드 분석 | 의도 필터를 가지고 코드를 봐야 "이건 다루고 이건 안 다룬다"를 판단할 수 있다 |
| 3. 시나리오 + 버전 | 이야기 설계 | 시나리오가 있어야 버전을 쪼갤 수 있다. 코드를 알아야 현실적인 시나리오가 나온다 |
| 4. 뼈대 세우기 | 구조 설계 | 시나리오 + 버전 + 코드가 있어야 목차를 짤 수 있다 |
| 5. 챕터 집필 | 쓰기 | 뼈대가 있어야 쓴다 |
| 6. 프롤로그 | 숲 보기 | 모든 챕터가 끝나야 전체를 조망하는 프롤로그를 쓸 수 있다 |
| 7. 마무리 | 닫기 | 책의 처음(서문)과 끝(맺음말)을 닫는다 |

## 상수 (Constants) — 절대 질문하지 않는다

이 시스템이 만드는 책의 정체성. 바뀌면 시스템 자체가 달라진다.
저자에게 묻지 않고 모든 단계에서 자동으로 적용한다.

### 문체 상수

| 상수 | 값 | 왜 상수인가 |
|------|-----|-----------|
| 포맷 | 짧은 쿡북 (100p 권장, 초과 허용) | 이 시스템의 정체성. 두꺼운 교과서가 아님. 초과분은 저자가 리뷰하며 빼기 작업 |
| 스타일 | 스토리텔링 | 핵심 컨셉. 설명이 아니라 이야기 |
| 톤 | 대화체 | "독자에게 말 거는 느낌" 고정 |
| 언어 수준 | 쉽게 (비유 필수) | 어려운 용어는 반드시 비유로 먼저 |
| 비유 길이 | 제한 없음 | 비유는 자유롭게. 용어 설명이 길어지면 기술 파트로 이동 |

### 구조 상수

| 상수 | 값 | 왜 상수인가 |
|------|-----|-----------|
| 챕터 내부 구조 | 이야기 파트 → 기술 파트 | 모든 챕터가 이 구조를 따른다 |
| 코드 분류 | 실습 / 설명 / 참고 3단계 | 모든 코드를 다 쓰게 하지 않는 원칙 |
| 성장 방식 | 버전별 성장 (챕터 = 한 버전) | 매 챕터에서 프로젝트가 한 단계 올라간다 |
| 코드 성격 | 교육용 (실무 코드 금지) | 실무 코드면 난이도 급상승 |
| 용어 정리 | 비유 → 기술 파트에서 정식 정의 필수 | 비유로 넘긴 용어를 방치하지 않는다 |

### 독자 상수

| 상수 | 값 | 왜 상수인가 |
|------|-----|-----------|
| 독자 수준 | 배경지식 있음, 이 책의 주제만 모름 | 매번 물어봤자 답은 같다 |
| 책 유형 | 개념서 | 코드보다 개념과 흐름 중심 |
| 코드 비중 | 낮음 | 이야기 파트에 코드 없음. 기술 파트에서만 |

### 책 구조 상수 (독자가 읽는 순서)

```
프롤로그    ← 코드 없이 전체 개념을 이야기로
로드맵      ← 버전별 성장 안내
본문 Ch.1~N ← 각 챕터: [이야기 파트] → [기술 파트]
서문 / 맺음말 / 부록
```

### 챕터 내부 구조 상수

```
┌─────────────────────────────────────┐
│  이야기 파트 (코드 없음)              │
│  - 도입 (경험, 상황)                 │
│  - 개념 (비유 1~2문장)               │
│  - 시나리오 진행 (이번 버전에서 뭘 바꾸나)│
│  - "이것만은 기억하자" + 다음 예고     │
├─────────────────────────────────────┤
│  기술 파트 (이야기 끝난 후)            │
│  - 용어 정리 (비유→정식 정의)          │
│  - 파일 계층 구조 [실습/설명/참고]      │
│  - 실습 코드 + 설명 코드              │
│  - 더 알아보기                       │
└─────────────────────────────────────┘
```

### 글쓰기 규칙 상수

1. **이야기 먼저**: 모든 챕터는 이야기 파트로 시작한다
2. **비유는 자유롭게**: 길이 제한 없음. 용어의 정식 설명이 길어지면 기술 파트로 넘긴다
3. **기술은 뒤에서**: 이야기 파트가 끝난 후 기술 파트에서 상세 설명
4. **비유 → 정의 필수**: 비유로 넘어간 용어는 기술 파트에서 반드시 정식 정의
5. **한 챕터 한 버전**: 챕터가 끝나면 결과물이 한 단계 성장
6. **점진적 깊이**: 쉬운 것 → 어려운 것
7. **코드 최소화**: 이야기 파트에 코드 없음. 기술 파트에서만
8. **챕터 닫기**: "이것만은 기억하자" + 다음 버전 예고
9. **기술은 저자가 결정**: AI가 제안, 저자가 최종 결정
10. **실패 먼저**: 정답부터 알려주지 않는다. 실수 → 에러 → 해결 순서

## 에이전트 — 8개 (워크플로우 5 + 품질 3)

### 워크플로우 에이전트 (5개) — 간략

각 STEP을 이끄는 에이전트. 대화를 주도하고 스킬을 조합해 산출물을 만든다.

| 에이전트 | 담당 | 역할 |
|---------|------|------|
| 씨앗-에이전트 | STEP 1 | 저자와 대화하며 의도 확립 |
| 코드분해-에이전트 | STEP 2 | 완성 코드를 의도 필터로 분석 |
| 설계-에이전트 | STEP 3~4 | 시나리오, 버전, 목차, 갭 분석 |
| 집필-에이전트 | STEP 5 | 챕터별 이야기+기술 파트 초안 |
| 마무리-에이전트 | STEP 6~7 | 프롤로그, 서문, 맺음말, 부록 |

> 각 에이전트의 사용 스킬과 동작 흐름 상세는 `에이전트-스킬-구조-v2.md` 참조.

### 품질 에이전트 (3개) — 상세

---

#### 1. 인사이트-에이전트

> **언제**: STEP 1~5, 변수를 수집할 때마다
> **뭘 하나**: 저자가 놓치고 있는 부분을 짚어주는 추가 질문

##### 동작 방식

```
저자 답변 수신
    ↓
답변 + 코드 분석 결과 + 이전 답변 맥락 분석
    ↓
"이건 생각해봤어?" 형태의 인사이트 질문 생성
    ↓
questions/pending/에 질문 저장 (대화에서 묻히지 않게)
    ↓
저자 반응 → 반영 or "넘어가자"
    ↓
답변 완료 시 questions/done/으로 이동
```

##### questions/ 폴더 — 질문 장바구니

대화 속에서 질문하면 묻혀버린다.
**인사이트-에이전트가 생성한 질문**과 **저자가 직접 메모한 질문** 모두 여기에 쌓는다.

**누가 넣나:**
- **인사이트-에이전트**: 워크플로우 진행 중 발견한 "저자가 놓친 부분" 자동 생성
- **저자**: 작업 중 떠오른 아이디어, 나중에 확인할 것, 결정을 미룬 것 등 직접 파일 생성

```
questions/
├── pending/                    ← 아직 답하지 않은 질문
│   ├── 001-async-범위.md              (인사이트-에이전트가 생성)
│   ├── 002-트랜잭션-선행개념.md         (인사이트-에이전트가 생성)
│   ├── 003-에러처리-챕터-넣을까.md      (저자가 직접 생성)
│   └── 004-부록에-docker-넣기.md       (저자가 직접 생성)
└── done/                       ← 답변 완료 + 반영된 질문
    ├── 001-async-범위.md
    └── ...
```

각 질문 파일 형식:

```markdown
# 001-async-범위

## 출처
인사이트-에이전트 / STEP 2 코드 분석 중
(또는: 저자 메모 / STEP 3 진행 중 떠오름)

## 질문
완성 코드에 @Async가 있는데, 비동기 처리를 다룰 거야?
다룬다면 독자가 쓰레드 개념을 알아야 하는데,
그건 배경지식에 포함되는 거야? 아니면 챕터를 써야 해?

## 영향 범위
이 답변에 따라 STEP 4(목차)에서 챕터 추가 여부가 달라짐

## 저자 답변
(pending 상태에서는 비어있음)

## 반영 결과
(done으로 이동할 때 기록: 어떤 산출물에 어떻게 반영했는지)
```

**왜 필요한가:**
- 인사이트 질문이 나온 시점에 바로 답하지 못할 수 있다
- 저자가 작업 중 떠오른 생각을 잊지 않게 메모해둘 수 있다
- pending 목록을 보면 "아직 결정 안 한 것들"이 한눈에 보인다
- done으로 옮길 때 "어디에 반영했는지"까지 기록하므로 추적 가능
- 저자가 직접 만든 질문도 AI가 다음 STEP에서 참고할 수 있다

##### 인사이트 질문 유형

**코드 기반 인사이트**:
```
"완성 코드에 @Async가 있던데, 비동기 처리를 다룰 거야?
 다룬다면 독자가 쓰레드 개념을 알아야 하는데,
 그건 배경지식에 포함되는 거야? 아니면 챕터를 써야 해?"
```

**구조 기반 인사이트**:
```
"v0.3에서 JPA를 도입하고 v0.4에서 인증을 붙이는데,
 독자 입장에서 두 개가 동시에 새로우면 힘들어.
 사이에 'JPA에 익숙해지는' 쉬어가기 버전을 넣는 건 어때?"
```

**의도 기반 인사이트**:
```
"'JWT를 이해시키는 게 목적'이라고 했는데,
 코드에 리프레시 토큰 로직이 있거든.
 리프레시 토큰은 '이해' 범위야 '구현' 범위야?
 이해라면 비유로만, 구현이라면 실습 코드가 필요해."
```

**분량 기반 인사이트**:
```
"지금 목차가 12챕터인데, 100페이지에 넣으려면 챕터당 8페이지야.
 코드까지 넣으면 빡빡해. 빼도 되는 챕터가 있을까?
 아니면 2개 챕터를 합칠 수 있을까?"
```

#### 2. 의도감시-에이전트

> **언제**: STEP 5, 각 챕터 초안 완료 직후
> **뭘 하나**: 챕터가 의도에서 벗어나지 않았는지 검증

##### 체크리스트

```
□ 의도 일치     이 챕터의 내용이 "책의 의도"에 부합하는가?
□ 범위 이탈     "의도 밖 범위"에 해당하는 내용이 슬며시 들어오지 않았는가?
□ 코드 성격     예제 코드가 교육용인가? (실무 코드가 아닌가?)
□ 깊이 적절     이 책의 목적(이해 vs 구현)에 맞는 깊이인가?
□ 비유 독립     이야기 파트의 비유가 기술 파트 없이도 이해되는가?
□ 핵심 유지     챕터 핵심 메시지가 책 전체 핵심 메시지와 연결되는가?
□ 상수 준수     챕터 내부 구조(이야기→기술), 비유 길이, 코드 분류를 지켰는가?
```

##### 출력 예시

```
✅ 의도 일치: OK — "인증이 뭔지 이해시키기"에 부합

⚠️ 범위 이탈 감지:
   → 23번째 줄에서 Spring Security의 FilterChain을 설명하고 있음.
     의도 밖 범위(Spring Security 연동)에 해당.
     제안: 삭제하거나 "나중에 Security를 쓰면 이런 게 자동으로 된다" 한 줄로 축소

⚠️ 코드 성격 경고:
   → AuthService.java에서 BCrypt 해싱, salt 처리까지 구현함.
     이건 실무 코드에 가까움. 교육용이라면 단순 비교로 충분.
     제안: password.equals(input)으로 단순화,
     "실무에서는 반드시 암호화를 쓴다" 한 줄 언급

✅ 깊이 적절: OK
✅ 비유 독립: OK
✅ 핵심 유지: OK
✅ 상수 준수: OK
```

#### 3. 감수-에이전트 (3인 편집장)

> **언제**: 각 STEP 산출물 완성 후, 저자 확인 전
> **뭘 하나**: 3가지 관점에서 질문으로 감수

##### 3인 편집장

| 편집장 | 관점 | 핵심 질문 스타일 |
|--------|------|-----------------|
| **기술 감수자** | 기술 정확성, 빠진 개념, 순서 | "독자가 OO를 모르는데 여기서 OO이 나와도 될까?" |
| **독자 대변인** | 난이도 곡선, 진입장벽, 완주 가능성 | "여기서 독자가 포기하지 않을까?" |
| **이야기 편집장** | 서사 흐름, 비유 적절성, 몰입도 | "이 도입이 설명으로 시작하는데, 경험담으로 열면 어떨까?" |

##### 동작 방식

```
STEP 산출물 완성
    ↓
3인 편집장이 각자 2~3개 질문
    ↓
저자가 질문에 답하면서 수정 여부 결정
    ↓
"넘어가자"도 가능
```

##### STEP별 감수 포인트

| STEP | 기술 감수자 | 독자 대변인 | 이야기 편집장 |
|------|------------|-----------|-------------|
| 1. 씨앗 | 핵심 메시지가 기술적으로 정확한가? | — | 차별점이 독자에게 와닿는 이야기가 되나? |
| 2. 코드 해부 | 빠진 핵심 개념은? 기술 의존성은? | 첫 버전이 너무 어렵지 않나? | — |
| 3. 시나리오+버전 | 버전 순서가 기술 의존성에 맞나? | 시나리오에 감정이입 가능? 포기 구간은? | 전환점이 극적인가? 긴장감 있나? |
| 4. 뼈대 | 빠진 기술은? 순서 최적화 제안 | 챕터 수가 적절? 완주 가능? | 챕터 제목이 호기심을 유발하나? |
| 5. 챕터 | 기술 설명이 정확? 더 좋은 예시? | 이 챕터만 읽어도 이해되나? | 도입 이야기가 몰입감 있나? |
| 6. 프롤로그 | 핵심 개념이 빠지진 않았나? | 프롤로그만 읽고 책을 사고 싶어지나? | 하나의 완결된 이야기인가? |

## 각 STEP 상세

---

### STEP 1. 씨앗 — "이 책은 뭐다"

> **목적**: 의도 필터 확보. 이후 모든 결정의 기준.

#### 흐름

```
1. "뭐에 대한 책이야?" → 주제
2. "왜 이 책을 쓰고 싶어?" → 집필 동기 (인사이트 질문 포함)
3. "이 분야 경험이 어때?" → 저자 경험 (실패담 유도)
4. "이 책에서 제일 하고 싶은 말?" → 핵심 메시지
5. "독자가 뭘 이해하게 하고 싶어?" → 책의 의도 (이해 vs 구현 명확화)
6. "그러면 이건 안 다루는 거지?" → 의도 밖 범위
    ↓
인사이트-에이전트: 놓친 부분 추가 질문
    ↓
AI 정리 → 저자 확인
    ↓
감수-에이전트: 의도 명확성 검증
    ↓
산출물: seed.md
```

#### 산출물: seed.md

```markdown
# 씨앗

## 주제
(한 줄)

## 집필 동기
(왜 이 책을 쓰고 싶은가)

## 저자 경험
(이 분야에서의 경험, 실패담 포함)

## 핵심 메시지
(독자가 동료에게 한 마디로 설명할 문장)

## 책의 의도
(뭘 이해시킬 것인가 — 모든 결정의 필터)

## 의도 밖 범위
(완성 코드에 있어도 다루지 않는 것들)

## 차별점
(기존 책 대비 이 책만의 포인트)
```

### STEP 2. 코드 해부 — "재료가 뭐가 있지"

> **목적**: 완성 코드를 의도 필터로 분석. 핵심 기능 추출.
> **전제**: 코드 워크플로우 산출물이 code/ 폴더에 있는 상태 또는 GitHub URL 제공.

#### 흐름

```
1. 완성 코드 구조/기능/기술 분석 (AI 자동)
2. "이 기능 목록 맞아?" → 핵심 기능 확인
3. 의도 필터 적용: 의도 밖 코드 분류 (AI 제안 → 저자 확인)
4. "이 기능 중에 안 보여줘도 되는 건?" → 범위 확정
    ↓
인사이트-에이전트: 코드에서 발견한 이슈 질문
    ↓
감수-에이전트: 빠진 개념, 기술 의존성 체크
    ↓
산출물: code-analysis.md
```

```markdown
# 코드 분석

(프로젝트 구조 트리)

## 핵심 기능 (의도 안)

|------|----------|----------|
| ... | ... | ... |

## 의도 밖 기능 (제외)
| 기능 | 관련 코드 | 제외 이유 |
|------|----------|----------|
| ... | ... | seed.md 의도 밖 범위에 해당 |
```

### STEP 3. 시나리오 + 버전 설계 — "어떤 순서로 이야기하지"

> **목적**: 책 전체를 관통하는 이야기 + 코드를 버전별로 쪼개기.
> 시나리오가 있어야 버전을 쪼갤 수 있으므로 한 STEP에서 같이 한다.

#### 흐름

```
--- 시나리오 ---
1. "책 전체를 관통하는 프로젝트/상황이 있어?" → 시나리오 (동기 포함)
2. "처음부터 끝까지 이야기해줘" → 전체 흐름
3. "제일 중요한 전환점은?" → 전환점
4. "마지막에 뭐가 완성돼?" → 최종 결과물

--- 버전 분해 (시나리오 기반) ---
5. "이 시나리오 순서대로 기능을 하나씩 붙여가면?" → AI가 버전 분해 제안
6. "각 버전 끝에 독자 눈에 뭐가 보여?" → 버전별 성과
7. "순서 바꾸고 싶은 거 있어?" → 저자 조정

--- 버전별 예제 코드 생성 ---
8. 확정된 버전 설계를 기반으로 versions/ 폴더에 버전별 예제 코드 생성
   - 완성 코드(code/)에서 해당 버전에 필요한 코드만 추출/축소
   - 교육용으로 단순화 (실무 코드 → 교육용 코드)
   - versions/v0.1/, versions/v0.2/, ... 각 폴더에 저장
    ↓
인사이트-에이전트: 전환점, 난이도 곡선, 포기 구간 질문
    ↓
감수-에이전트: 서사 흐름, 기술 의존성, 감정이입 체크
    ↓
산출물: scenario.md, versions/
```

#### 산출물: scenario.md

```markdown

### 한 줄 요약

### 전체 흐름
1. 시작: ...
2. 전개: ...
3. 전환점: ...
4. 완성: ...

### 최종 결과물
(예: "글쓰기, 로그인, 배포까지 되는 블로그")

---

## 버전 설계

| 버전 | 뭘 만드나 | 시나리오 속 위치 | 끝나면 뭐가 보이나 | 핵심 변경점 |
```

### STEP 4. 뼈대 세우기 — "목차와 코드 실습 배치"

> **목적**: 코드 실습 분류 + 독자 여정 + 목차 + 갭 분석 + 기술 매핑을 한 번에.

### STEP 5. 챕터 집필 — "쓴다" (챕터마다 반복)

> **목적**: 이야기 파트 + 기술 파트 초안 작성.
> 쓰고 → 의도 검증 → 감수 → 수정을 한 사이클로 반복.

#### 한 챕터의 흐름

```
--- 변수 수집 ---
1. "이번 버전에서 뭘 만들어?" → 버전 목표
2. "이전 버전에서 뭘 바꾸나?" → 변경점
3. "왜 해야 해? 안 하면 뭐가 문제야?" → 도입 이야기
4. "짧은 비유 하나만?" → 비유 (1~2문장)
5. "핵심 개념 3개 이하로?" → 핵심 개념
6. "실무에서 실패한 적은?" → 실패담
7. "이것만 기억해라 하나만?" → 한줄 기억
8. "기술 파트에서 더 설명할 건?" → 기술 파트 상세
    ↓
인사이트-에이전트: 챕터별 놓친 부분 질문
    ↓

--- 초안 작성 ---
AI가 이야기 파트 + 기술 파트 초안 작성
    ↓

--- 의도 검증 ---
의도감시-에이전트 발동: 의도 이탈/코드 성격/깊이 체크
    ↓
위반 시 수정
    ↓

--- 감수 ---
감수-에이전트(3인 편집장) 발동
    ↓
저자 확인 + 수정
    ↓

산출물: chapters/NN-제목.md
    ↓

다음 챕터로 (STEP 5 반복)
```

### STEP 6. 프롤로그 + 로드맵 — "숲을 보여준다"

> **목적**: 모든 챕터가 끝난 후, 코드 없이 전체를 한 눈에.

#### 흐름

```

2. 시나리오 흐름대로 하나의 이야기로 연결 → 프롤로그
3. 버전별 성장 과정을 시각적으로 → 로드맵
    ↓
감수-에이전트: 프롤로그만으로 책이 궁금해지는지
    ↓
산출물: book/00-prologue.md, book/00-roadmap.md
```

### STEP 7. 마무리 — "서문, 맺음말, 부록"

> **목적**: 책의 처음과 끝을 닫는다.

#### 흐름

```
1. "서문에 뭘 쓰고 싶어? 왜 이 책을 쓰게 됐는지"
2. "맺음말에서 독자에게 마지막으로 하고 싶은 말은?"
3. "더 공부하고 싶을 때 추천할 자료?" → 부록
4. 최종 제목 + 부제목 확정
    ↓
감수-에이전트: 진정성, 용기, 심화 자료
    ↓
산출물: book/preface.md, book/afterword.md, book/appendix.md
```

## 필요 없는 것 (원래 설계에서 뺀 것)

| 뺀 것 | 이유 |
|-------|------|
| 독자 프로필/페르소나 수집 단계 | 독자 수준은 상수. 질문할 필요 없음 |
| config.md에 저자가 상수 편집 | 상수는 시스템이 고정. 저자가 건드릴 일 없음 |
| 상세 git 커밋 규칙 | 커밋은 자연스럽게. 규칙표 불필요 |
| 독자 여정을 별도 STEP으로 분리 | 목차 설계와 같이 해야 맥락이 이어짐 |
| 코드 실습 설계를 별도 STEP으로 분리 | 목차에 코드 분류를 넣어야 하므로 같은 STEP |
