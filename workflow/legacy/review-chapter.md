# 챕터 검토 가이드

`검토 [챕터]` 명령 시 이 가이드를 따라 작성된 챕터를 검토한다.

## 입력

- `projects/[책이름]/chapters/[번호]-[챕터명].md` (검토 대상 챕터)
- `projects/[책이름]/answers.md` (독자 수준, 콘텐츠 전략 참조)
- `projects/[책이름]/outline.md` (페이지 배분, 챕터 위치 참조)

## 검토 체크리스트

아래 항목을 순서대로 검토하고, 각 항목에 대해 통과/개선 필요를 판단한다.

### 1. 난이도 적합성
- [ ] 대상 독자 수준에 맞는 설명 깊이인가?
- [ ] 사전 지식으로 명시한 내용은 별도 설명 없이 사용해도 되는가?
- [ ] 사전 지식에 없는 개념은 충분히 설명했는가?
- [ ] 갑자기 난이도가 올라가는 구간이 없는가?

### 2. 스토리텔링 품질
- [ ] 전문 용어가 처음 등장할 때 **비유**로 풀어 설명하고 있는가? (단순 정의 나열이 아닌지)
- [ ] 챕터당 최소 2~3개의 비유가 사용됐는가?
- [ ] 비유가 적절하고 독자가 쉽게 이해할 수 있는가?
- [ ] 교과서처럼 딱딱하지 않고 이야기하듯 읽히는가?
- [ ] 독자에게 말을 거는 문장이 적절히 포함됐는가? ("혹시~", "한번~")
- [ ] 시행착오 과정(에러 → 원인 → 해결)이 자연스럽게 포함됐는가?
- [ ] "왜" 이렇게 하는지에 대한 설명이 있는가? (What뿐 아니라 Why)
- [ ] 읽는 사람이 막힐 수 있는 지점에 보충 설명이 있는가?

### 3. 코드와 설명의 일치
- [ ] 코드 블록의 언어 태그가 올바른가?
- [ ] 본문에서 설명하는 코드와 실제 코드 블록이 일치하는가?
- [ ] 코드 실행 결과가 정확한가?
- [ ] GitHub 링크가 올바른 파일을 가리키는가?
- [ ] 코드의 수정/변경 사항이 명확히 표시됐는가?

### 4. 페이지 배분
- [ ] outline.md에서 배분된 페이지 수에 맞는가?
- [ ] 너무 짧거나 너무 긴 부분이 없는가?
- [ ] 불필요하게 반복되는 내용이 없는가?

### 5. 흐름의 자연스러움
- [ ] 도입부가 이전 챕터와 자연스럽게 연결되는가?
- [ ] 본문의 내용이 논리적 순서로 배치됐는가?
- [ ] 갑작스러운 주제 전환이 없는가?
- [ ] 마무리가 다음 챕터로 자연스럽게 이어지는가?

### 6. 구조 준수
- [ ] 도입 → 본문 → 정리 구조를 따르고 있는가?
- [ ] 답변의 "설명 방식"에 맞는 전개를 하고 있는가?
- [ ] 요약 섹션이 필요한 경우 포함돼 있는가?
- [ ] 연습 문제가 필요한 경우 포함돼 있는가?
- [ ] 트러블슈팅 섹션이 필요한 경우 포함돼 있는가?

### 7. 문체 일관성
- [ ] answers.md에서 설정한 문체를 일관되게 사용하는가?
- [ ] 존댓말/반말이 섞여 있지 않은가?
- [ ] 톤이 챕터 내내 일관적인가?

### 8. 오탈자/문법
- [ ] 오탈자가 없는가?
- [ ] 문법적으로 어색한 문장이 없는가?
- [ ] 마크다운 문법이 올바른가? (헤더, 코드 블록, 링크 등)

## 검토 결과 출력 형식

검토 결과를 아래 형식으로 사용자에게 보여준다:

```markdown
## 검토 결과: 챕터 [번호] - [제목]

### 종합 평가
[전체적인 평가를 2~3줄로 요약]

### 잘된 점
- [구체적인 칭찬 포인트]

### 개선이 필요한 부분
1. **[항목]**: [구체적인 문제점과 개선 방향]
   - 위치: [해당 섹션/라인]
   - 제안: [구체적인 수정 방안]

2. **[항목]**: ...

### 세부 체크리스트 결과
| 항목 | 결과 | 비고 |
|------|------|------|
| 난이도 적합성 | ✅/⚠️ | [메모] |
| 스토리텔링 품질 | ✅/⚠️ | [메모] |
| 코드-설명 일치 | ✅/⚠️ | [메모] |
| 페이지 배분 | ✅/⚠️ | [메모] |
| 흐름 자연스러움 | ✅/⚠️ | [메모] |
| 구조 준수 | ✅/⚠️ | [메모] |
| 문체 일관성 | ✅/⚠️ | [메모] |
| 오탈자/문법 | ✅/⚠️ | [메모] |
```

## 피드백 기록

검토 결과를 `projects/[책이름]/review/feedback-log.md`에 누적 기록한다:

```markdown
# 검토 기록

## [날짜] 챕터 [번호] - [제목]
- **검토 결과**: [통과 / 수정 필요]
- **주요 피드백**: [요약]
- **수정 완료 여부**: [완료 / 미완료]
```

## 후속 작업

1. 개선이 필요한 부분이 있으면 수정 여부를 사용자에게 묻는다.
2. 사용자가 수정을 원하면 해당 부분만 수정한 뒤 재검토한다.
3. 모든 항목이 통과되면 "챕터 N 검토 완료! 다음 챕터를 작성하시겠습니까?" 안내.
