# 챕터 초안 생성 가이드

`챕터 작성 [번호]` 명령 시 이 가이드를 따라 개별 챕터를 작성한다.

## 입력

- `projects/[책이름]/outline.md`에서 해당 챕터의 정보 (제목, 개요, 페이지, 키워드)
- `projects/[책이름]/answers.md`의 전체 답변 (독자 수준, 설명 방식, 문체 등)
- `projects/[책이름]/code/` 내 관련 코드 파일 (있는 경우)

## 스토리텔링 원칙

이 가이드의 모든 챕터는 **이야기**로 써야 한다. 아래 원칙을 항상 적용한다:

### 비유 활용법
- 모든 새로운 개념은 일상의 비유로 먼저 설명한다.
- 비유 → 실제 개념 → 코드 순서로 전개한다.
- 비유 예시 모음:
  - 변수 → "이름표가 붙은 상자"
  - API → "식당 웨이터" 또는 "창구 직원"
  - 서버 → "24시간 운영되는 가게"
  - 데이터베이스 → "거대한 엑셀 파일" 또는 "도서관 서고"
  - 함수 → "자판기: 동전을 넣으면(입력) 음료가 나온다(출력)"
  - 에러 → "길을 가다 만난 공사 중 표지판"
- 챕터마다 최소 2~3개의 비유를 사용한다.

### 내러티브 구조
각 챕터는 아래 이야기 흐름을 따른다:
1. **상황 제시**: 독자가 공감할 수 있는 상황이나 문제를 던진다.
2. **호기심 유발**: "이걸 어떻게 해결할 수 있을까요?"
3. **탐색 과정**: 해결 방법을 함께 찾아가는 여정 (시행착오 포함)
4. **해결과 깨달음**: 코드가 동작하는 순간의 성취감
5. **되돌아보기**: 배운 것을 정리하고 다음 모험을 예고

### 시행착오 패턴
처음부터 완벽한 답을 주지 않는다. 아래 패턴을 활용한다:
```
[처음 시도] → [에러 발생] → [원인 분석] → [수정] → [성공]
```
예시:
> "이 코드를 실행하면... 어라? `TypeError`가 뜨네요. 왜 그럴까요?
> 원인은 간단합니다. 문자열을 숫자로 변환하지 않았거든요.
> `int()`로 감싸주면..."

## 챕터 구조

모든 챕터는 아래 구조를 따른다:

### 1. 도입부 (상황 제시)
- 딱딱한 학습 목표 나열 대신, 독자가 공감할 수 있는 상황을 제시한다.
- 이전 챕터의 이야기와 자연스럽게 연결한다.
- 예시:
  ```markdown
  지난 챕터에서 우리는 개발 환경을 세팅했습니다.
  집을 지으려면 먼저 땅을 다져야 하듯, 코드를 짜려면 먼저 도구를 준비해야 하니까요.

  자, 이제 도구가 준비됐으니 본격적으로 첫 번째 요청을 보내볼까요?
  마치 새로 개통한 전화기로 첫 통화를 하는 것처럼, 조금 설레는 순간입니다.
  ```

### 2. 본문 (탐색과 해결의 여정)
답변의 "설명 방식"에 따라 전개하되, 항상 이야기하듯 진행한다:

#### 개념 먼저 → 실습
1. 일상 비유로 개념 소개 ("API는 마치 식당의 웨이터와 같습니다...")
2. 비유에서 실제 개념으로 연결 ("웨이터가 주문을 전달하듯, API는...")
3. 코드 제시 및 이야기체로 설명 ("자, 이 코드를 한번 살펴볼까요?")
4. 실행 결과 보여주기 ("실행하면... 짜잔! 이런 결과가 나타납니다")
5. 의도적 실수 → 해결 과정 ("만약 여기서 이렇게 바꾸면 어떻게 될까요?")

#### 바로 실습 → 개념 보충
1. 상황 제시 ("우리 앞에 이런 문제가 있습니다")
2. 완성 코드 먼저 제시 ("일단 이 코드를 그대로 실행해보세요")
3. 실행 결과에 대한 호기심 유발 ("동작하긴 하는데... 어떻게 된 걸까요?")
4. 비유로 동작 원리 설명 ("이 과정은 마치...")
5. 코드를 변형하며 이해 심화

#### 프로젝트 기반
1. 오늘의 미션 소개 ("오늘 우리가 만들 기능은...")
2. 큰 그림 먼저 보여주기 ("완성되면 이런 모습이 됩니다")
3. 단계별 구현 - 각 단계를 에피소드처럼 전개
4. 중간중간 에러를 만나고 해결하는 과정 포함
5. 기능 완성 후 성취감 공유 ("드디어 완성! 직접 테스트해볼까요?")

### 3. 코드 블록 규칙
- 코드 블록에는 반드시 언어 태그를 명시한다: ````python`, ````javascript` 등
- 코드가 길면 핵심 부분만 본문에 포함하고, 전체 코드는 GitHub 링크를 안내한다.
  ```markdown
  > 전체 코드는 [GitHub 레포](URL)의 `src/chatbot.py`에서 확인할 수 있습니다.
  ```
- code/ 폴더에 코드가 있으면 파일 경로를 명시한다.
- 코드에 주석을 달아 각 부분의 역할을 설명한다.
- answers.md의 "코드 수준"에 맞춰 코드의 완성도를 조절한다.

### 4. 마무리
- **핵심 정리**: answers.md에서 "챕터 끝 요약"이 "포함"이면, 이 챕터에서 배운 것을 3~5개 항목으로 정리한다.
  ```markdown
  ## 이 챕터에서 배운 것
  - API 키 발급 방법
  - 기본 요청 보내기
  - 응답 구조 이해하기
  ```
- **연습 문제**: answers.md에서 "연습 문제"가 "포함"이면, 1~3개 과제를 제시한다.
  ```markdown
  ## 직접 해보기
  1. [기본] 다른 프롬프트로 요청을 보내보세요.
  2. [응용] 응답에서 특정 필드만 추출해보세요.
  3. [도전] 여러 번 요청을 보내는 반복문을 작성해보세요.
  ```
- **다음 챕터 예고**: 다음 챕터에서 다룰 내용을 한 줄로 예고한다.
  ```markdown
  다음 챕터에서는 대화 기록을 유지하는 방법을 알아봅니다.
  ```

### 5. 트러블슈팅 (선택)
answers.md에서 "트러블슈팅/FAQ"가 "챕터별"이면 섹션을 추가한다:
```markdown
## 자주 묻는 질문

**Q: API 키가 인식되지 않아요.**
A: 환경 변수가 제대로 설정됐는지 확인해보세요. `echo $API_KEY` 명령으로 확인할 수 있습니다.
```

## 페이지 제한

- outline.md에서 해당 챕터에 배분된 페이지 수를 참고한다.
- 1페이지 ≈ 한글 기준 약 500자 (코드 포함)로 추산한다.
- 배분된 페이지를 초과하면 내용을 압축하거나 덜 중요한 부분을 부록으로 뺀다.

## 문체

- answers.md의 "글 톤/문체" 설정을 따르되, 어떤 문체든 **이야기하듯** 써야 한다.
- 전문 용어는 처음 등장 시 비유로 풀어 설명한다. 괄호 안 정의가 아니라, 한두 문장의 비유를 사용한다.
- 독자에게 말을 거는 문장을 자주 사용한다: "혹시 이런 경험 있으신가요?", "여기서 잠깐, 생각해보세요."
- 챕터 전체에서 일관된 톤을 유지한다.
- 절대 교과서처럼 쓰지 않는다. 옆자리 동료가 설명해주는 느낌으로.

## 출력

생성된 챕터는 `projects/[책이름]/chapters/[번호]-[챕터명].md`에 저장한다.
- 예: `chapters/01-시작하기.md`, `chapters/02-첫-번째-요청.md`
- 파일 생성 후 사용자에게 "챕터 N 초안이 완성됐습니다. 검토하시겠습니까?" 안내.
