# STEP 1. 씨앗 — "이 책은 뭐다"

> **목적**: 의도 필터 확보. 이후 모든 결정의 기준.
> **스킬**: A1.구조-스캐너, A3.기술스택-탐지기, C2.요약기
> **검토**: [인사이트 검토] + [감수 검토]
> **산출물**: `planning/seed.md`

## 코드 사전 스캔

씨앗 질문 **전에** 완성 코드를 가볍게 훑는다.
STEP 2(코드 해부)의 깊은 분석과는 다르다 — 여기서는 **질문 선택지의 재료**만 뽑는다.

### 목적

코드에서 자동으로 알 수 있는 것은 AI가 미리 파악하여:
1. 질문 선택지를 구체적으로 만든다 (막연한 추측이 아닌 실제 코드 기반)
2. 코드가 답해주는 것은 선택지에 미리 채운다
3. 저자에게는 **코드가 답 못하는 것만** 집중해서 묻는다

### 스캔 항목

| 항목 | 파악하는 것 | 질문에 반영 |
|------|-----------|-----------|
| 기술 스택 | 언어, 프레임워크, 주요 라이브러리 | Q1 선택지를 코드 실제 스택으로 채움 |
| 프로젝트 구조 | 폴더/파일 트리 | Q6 제외 범위 선택지에 실제 기능 목록 제공 |
| 주요 기능 | 엔드포인트, 핵심 클래스/함수 | Q5 이해 범위 선택지에 반영 |
| 외부 의존성 | API, DB, 서드파티 서비스 | Q6 제외 후보에 반영 |

### 흐름

```
코드 입력 (code/ 폴더 또는 GitHub URL)
    ↓
A1.구조-스캐너 → 프로젝트 트리 (가볍게)
A3.기술스택-탐지기 → 기술 스택 목록
    ↓
스캔 결과 요약을 저자에게 보여줌
  "코드를 훑어봤어. [기술스택], [주요기능] 이런 구성이네.
   이걸 바탕으로 몇 가지 물어볼게."
    ↓
질문 진행 (스캔 결과가 선택지에 반영됨)
```

### STEP 2와의 차이

| | 코드 사전 스캔 (STEP 1) | 코드 해부 (STEP 2) |
|---|---|---|
| **목적** | 질문 선택지 재료 | 의도 필터로 기능 분류 |
| **깊이** | 구조 + 스택만 | 기능별 상세 분석 |
| **의도 필터** | 아직 없음 | seed.md 기반 적용 |
| **산출물** | 없음 (질문에 반영) | code-analysis.md |

## 질문 (변수 수집)

질문은 순서대로 진행하되, 대화하듯 자연스럽게 묻는다.
각 질문에는 **인사이트 질문**(저자가 놓친 부분을 짚어주는 추가 질문)을 함께 던진다.
**코드 사전 스캔 결과**를 활용하여 선택지를 구체적으로 만든다.

### 선택지 제안 규칙

모든 질문은 `AskUserQuestion` **선택형 UI**로 **한 번에 하나씩** 묻는다.

**UI 규칙**:
- **선택지 최대 4개** — UI 제한. "Other(직접 입력)"은 자동 제공됨
- **한 번에 한 질문만** — 여러 질문을 한꺼번에 나열하지 않는다
- **AI 추천 표시** — AI가 추천하는 선택지는 첫 번째에 놓고 label 끝에 `(Recommended)` 표시
- 인사이트 질문은 선택지 설명(description)에 녹이거나, 답변 받은 후 텍스트로 따로 던진다

**선택지 생성 원칙**:
- 프로젝트명, 주제, 기존 답변에서 맥락을 추론하여 현실적인 선택지를 만든다
- 선택지는 서로 다른 방향을 커버한다 (비슷한 것 나열 금지)
- 선택지마다 한 줄 설명(description)을 붙여서 왜 이걸 골랐는지 알 수 있게 한다
- 이전 질문의 답변이 있으면 그 맥락을 반영하여 다음 질문의 선택지를 조정한다
- 4개 안에 핵심 방향을 압축한다. "AI가 범용적으로 작성" 같은 옵션이 있으면 반드시 포함

**저자 응답 방식**:
- 선택지 중 하나 선택
- "Other"로 직접 작성 (자유 텍스트, 조합, 수정 모두 가능)

### 질문 목록

| # | 질문 | 인사이트 질문 | 선택지 생성 힌트 |
|---|------|-------------|----------------|
| 1 | "뭐에 대한 책이야?" | — (단순 질문으로 충분) | **코드 스캔 결과** 기술 스택으로 선택지 구성 |
| 2 | "왜 이 책을 쓰고 싶어?" | "비슷한 책이 있는데도 쓰려는 거라면, 그 책의 뭐가 아쉬웠어?" | 기존 자료의 흔한 불만 방향 + **"AI가 범용적으로 작성"** 포함 |
| 3 | "이 분야 경험이 어때?" | "실패담이 있으면 최고의 챕터 소재야" | 전형적 삽질 유형 + **"AI가 범용적으로 작성"** 포함 |
| 4 | "이 책에서 제일 하고 싶은 말?" | "독자가 동료에게 한 마디로 설명한다면?" | 주제를 비유로 압축한 한 문장 제안 |
| 5 | "독자가 뭘 이해하게 하고 싶어?" | "'이해'와 '할 수 있다'는 다르거든" | **코드 스캔 결과** 주요 기능 기반 이해/구축 범위 |
| 6 | "그러면 이건 안 다루는 거지?" | "의도 밖이면 코드 분석에서부터 빠져" | **코드 스캔 결과** 기능/의존성에서 빼기 후보. multiSelect 사용 |

## 흐름

```
코드 입력 (code/ 폴더 또는 GitHub URL)
    ↓
[코드 사전 스캔] → 기술 스택 + 구조 + 주요 기능 파악
    ↓
스캔 결과 요약 → 저자에게 공유
    ↓
1~6번 질문 진행 (스캔 결과 반영된 선택지 + 인사이트 질문)
    ↓
[인사이트 검토] → 놓친 부분 추가 질문
    → questions/pending/에 저장
    ↓
C2.요약기 → 답변 정리 → 저자 확인
    ↓
[감수 검토] → 의도 명확성 검증
  - 기술 감수자: 핵심 메시지가 기술적으로 정확한가?
  - 이야기 편집장: 차별점이 독자에게 와닿는 이야기가 되나?
    ↓
저자 확인 → 수정 → planning/seed.md 생성
```

## 진행 규칙

1. **코드 먼저**: 질문 전에 반드시 코드 사전 스캔을 실행한다. 코드가 없으면 넣어달라고 안내.
2. **선택형 UI, 하나씩**: `AskUserQuestion` 선택형 UI로 한 번에 한 질문만 묻는다. 선택지 최대 4개 (Other 자동 제공). 서술형 나열 금지.
3. 질문 1~3을 먼저 묻고, 답변을 받은 뒤 4~6을 진행한다.
4. 각 답변은 `answers.md`에 즉시 저장한다.
5. 기획서가 있으면 분석 후 빠진 부분만 질문한다.
6. 주제가 너무 넓으면 좁혀줄 것을, 너무 좁으면 넓혀줄 것을 제안한다.
7. 완료 후 "STEP 1 완료! 의도가 확립됐습니다. `코드 분석`으로 STEP 2를 시작할까요?" 안내.

## 산출물 템플릿: planning/seed.md

```markdown
# 씨앗

## 주제
(한 줄)

## 집필 동기
(왜 이 책이 필요한가. 저자가 직접 작성하거나, AI가 범용적으로 작성)
> 저자 미지정 시: AI가 "이런 책이 필요한 이유"를 독자 관점에서 범용적으로 작성한다.

## 저자 경험
(이 분야에서의 경험, 실패담 포함)
> 저자 미지정 시: AI가 "이 기술을 처음 접하는 사람이 겪는 전형적 경험"으로 범용 작성한다.
> 서문/프롤로그 생성 시 "저자 = 만든 사람"을 전제하지 않는다.

## 핵심 메시지
(독자가 동료에게 한 마디로 설명할 문장)

## 책의 의도
(뭘 이해시킬 것인가 — 모든 결정의 필터)

## 의도 밖 범위
(완성 코드에 있어도 다루지 않는 것들)

## 차별점
(기존 책 대비 이 책만의 포인트)
```
